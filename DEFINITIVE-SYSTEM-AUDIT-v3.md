## 1.1 [CRITICAL] Access Letter Missing Credentials — The Entire Purpose of the Letter

**What VistA Terminal Does vs What We Do:**
The VA Kernel 8.0 Systems Management Guide (August 2025) documents a "Reprint Access Agreement Letter" option (Figure 19) and a 4-page "Security Forms" system (Figures 20-23) that includes:
- Page 1-2: User account details (name, title, division, keys)
- Page 3: Computer Account Access Policy Form (user signs acknowledging rules)
- Page 4: User Account Notification Form (contains the ACCESS CODE and VERIFY CODE)

Our access letter is a single page that:
- Shows name, role, department, division ✓
- Shows human-readable capabilities (improved from raw keys) ✓
- Shows login URL ✓
- Does NOT include the actual ACCESS CODE or VERIFY CODE ✗
- Does NOT include an acknowledgment/signature section ✗
- Says "contact IT for password reset" — but this is a NEW account, not a reset ✗

The whole point of the terminal's "User Account Notification Form" is to hand the user their login credentials on paper. Our letter fails at its primary purpose.

**How to Fix:**
1. After successful user creation, pass {accessCode, verifyCode} from the wizard form state to the success screen component via React state (NOT sessionStorage — keep in memory only).
2. Success screen shows a "Print Welcome Letter" button that generates the letter WITH credentials.
3. Letter format (based on VA Security Forms):
   - Header: Organization name, division, date
   - Section 1: Account details (name, role, department)
   - Section 2: LOGIN CREDENTIALS (Access Code, Temporary Verify Code) — clearly boxed
   - Section 3: First-login instructions (1. Go to URL, 2. Enter credentials, 3. Change password, 4. Set e-signature)
   - Section 4: System capabilities (human-readable)
   - Section 5: Acknowledgment line (user signs to confirm receipt)
   - Footer: "CONFIDENTIAL — Destroy after first login. Generated by [admin name] on [date]"
4. The detail panel version of "Print Access Letter" (accessed later, not from creation) should NOT include credentials — it becomes an account information letter.
5. Clear credentials from React state when admin navigates away from success screen.

**Verify:** Create user → success screen → Print Welcome Letter → letter shows actual username and temp password → user signs acknowledgment → letter is useful

## 1.2 [HIGH] Navigation Structure Doesn't Match VistA's Administrative Model

**What VistA Terminal Does vs What We Do:**
VistA terminal's EVE → XUSER menu tree (documented in Kernel Technical Manual, Table 6-9) has this structure:

```
EVE (Systems Manager Menu)
├── User Management [XUSER]
│   ├── Add a New User [XUSERNEW]
│   ├── Edit an Existing User [XUSEREDIT]  (5-page ScreenMan form)
│   ├── Deactivate a User [XUSERDEACT]
│   ├── Reactivate a User [XUREACT]
│   ├── Grant Access by Profile [XUSERBLK] (clone user)
│   ├── List Users [XUSERLIST]
│   ├── User Inquiry [XUSERINQ]
│   └── Reprint Access Agreement Letter
├── Menu Management [XUMAINT]
│   ├── Key Management
│   │   ├── Allocation of Security Keys
│   │   ├── De-allocation of Security Keys
│   │   ├── Show Keys of a User
│   │   └── Show Holders of a Key
│   ├── Edit Options
│   ├── Display Menus
│   └── Delegate Options
├── Device Management
│   ├── Set Up a Device
│   ├── Delete a Device
│   └── Terminal Type Setup
├── TaskMan Management
│   ├── Schedule/Unschedule Options
│   ├── Requeue Tasks
│   └── Delete Tasks
└── Site Parameters
```

Our navigation has 8 groups with 20 items, organized differently from the terminal but not necessarily better. The specific problems:

1. **"Overview" group has only 1 item** (Dashboard) — wasteful. Dashboard should be the default landing page, not a nav item.
2. **"People" and "Access Control" are split** but in terminal they're one menu (User Management includes key allocation). An admin creating a user and assigning keys does it in one workflow, not two separate pages.
3. **"Locations" is too sparse** — only Clinics and Wards. Terminal's Hospital Locations include Operating Rooms, Modules, and Other types. We filter these out now (good) but we could show "Other Locations" for completeness.
4. **"Organization" is confusing** — "Divisions" is correct now but "Departments & Services" maps to File 49 SERVICE/SECTION which is a clinical concept, not an organizational one.
5. **"System Settings" has 4 items including Devices** — Devices are infrastructure, not settings. In terminal, Device Management is its own menu branch.
6. **"Mail Administration" has only 1 item** (Mail Groups) — this should be merged with "Monitoring → Messages & Alerts" since they're both MailMan-related.
7. **"Monitoring" has 4 items** — System Health, Audit Trail, Messages & Alerts, Reports. This is a good grouping but "Messages & Alerts" is really MailMan which is communication, not monitoring.
8. **"Add Staff Member" as a separate nav item** — it's a button action, not a page. It should be a button on the Staff Directory page, not a nav entry.

**How to Fix:**
Restructure to match VistA's mental model while being modern:

```
People & Access
  ├── Staff Directory (includes "Add" button, replaces separate nav item)
  ├── Roles & Permissions
  └── Permission Catalog

Facility Setup
  ├── Clinics (File 44 TYPE=C)
  ├── Wards (File 44 TYPE=W)
  ├── Divisions (File 40.8)
  ├── Departments (File 49)
  └── Devices & Printers (File 3.5)

Security & Policy
  ├── Authentication Settings
  ├── Audit Trail
  └── System Configuration

Communication
  ├── MailMan Messages
  └── Mail Groups

System
  ├── System Health
  ├── Module Settings
  └── Reports

Dashboard (default landing — not a nav group, just the /admin default)
```

This reduces 8 groups to 5, removes the single-item groups, and matches how a sysadmin thinks: "I need to manage people," "I need to set up the facility," "I need to configure security," "I need to communicate," "I need to monitor the system." 

**Verify:** Navigation restructured → each group has 2+ items → no single-item groups → mental model matches VistA admin workflow

## 1.4 [MEDIUM] Division Selector: What Happens When You Select a Different Division?

**What VistA Terminal Does vs What We Do:**
The upper-right dropdown shows Medical Center Divisions from File 40.8. When you select a different division, the Staff Directory filters by that division (via FacilityContext).
But what about OTHER pages? When you switch divisions:
- Does the Clinics page filter to clinics in that division?
- Does the Wards page filter to wards in that division?
- Does the Patient search filter to patients in that division?
- Does the Dashboard show counts for that division?
If the answer is "no" to any of these, the division selector is misleading — it appears to be a global filter but only affects Staff Directory.

**How to Fix:**
Either: make the division selector truly global (all pages respect it) or clearly label it as "Staff Directory filter" and move it to the Staff Directory page instead of the system bar.

**Verify:** Switch division → ALL pages show data for selected division, or selector is page-specific

# SECTION 15: VistA INPUT TRANSFORM COMPLIANCE

VistA validates every field via input transforms stored in the data dictionary.
Our frontend validation must match or exceed VistA's rules. If our frontend accepts
a value that VistA rejects, the user gets a confusing error after submission.

### Name (.01) (File 200)
**VistA Rule:** UPPERCASE LAST,FIRST MI. Only A-Z, comma, space, hyphen. 3-35 chars. No apostrophes, no numbers, no special chars.
**Verify:** Enter lowercase → auto-uppercase. Enter O'BRIEN → warning about apostrophe.

### SSN (9) (File 200)
**VistA Rule:** 9 digits. Format: 000000000 or 000-00-0000. Not all zeros. Not all same digit.
**Verify:** We don't write SSN — but if we add full SSN field later, validate these rules.

### DOB (5) (File 200)
**VistA Rule:** FileMan internal format: YYYMMDD where YYY = year - 1700. Valid date. Not future.
**Verify:** Server converts ISO to FM: 1990-05-15 → 2900515. Verify conversion is correct.

### Sex (4) (File 200)
**VistA Rule:** SET OF CODES: M or F only.
**Verify:** Dropdown only offers M and F. No free text.

### Title (8) (File 200)
**VistA Rule:** POINTER to File 3.1. Must match existing entry.
**Verify:** Free text fails. Must be dropdown from File 3.1 entries.

### Service/Section (29) (File 200)
**VistA Rule:** POINTER to File 49. Must match existing entry.
**Verify:** Dropdown from File 49 entries. Type-ahead search.

### NPI (41.99) (File 200)
**VistA Rule:** 10 digits. Luhn check digit with 80840 prefix.
**Verify:** Our Luhn validation matches VistA's. Verify.

### DEA (53.2) (File 200)
**VistA Rule:** 2 letters + 7 digits. Check digit algorithm.
**Verify:** Our format check passes. DEA check digit not validated — add it.

### Provider Class (53.5) (File 200)
**VistA Rule:** POINTER to File 7. Must match existing entry.
**Verify:** Verify File 7 entries exist in our Docker. Use dropdown.

### Primary Menu (201) (File 200)
**VistA Rule:** POINTER to File 19. Must be valid option.
**Verify:** Currently no UI selector. Must validate against File 19.

### Phone (.132) (File 200)
**VistA Rule:** Numeric with optional formatting. VistA stores raw.
**Verify:** Accept formatted input, strip to digits for VistA, format for display.

### Email (.151) (File 200)
**VistA Rule:** Free text. VistA has minimal validation.
**Verify:** Our regex is sufficient. VistA accepts almost anything.

### Patient Name (.01) (File 2)
**VistA Rule:** Same as File 200: UPPERCASE LAST,FIRST. A-Z/comma/space/hyphen.
**Verify:** Same validation as staff names.

### Patient SSN (.09) (File 2)
**VistA Rule:** 9 digits required for VA patients. Pseudo-SSN with reason for non-VA.
**Verify:** Full 9 digits required. Validate format.

### Marital Status (.05) (File 2)
**VistA Rule:** POINTER to File 11. Must match.
**Verify:** Dropdown from File 11 entries.

### Race (.06) (File 2)
**VistA Rule:** POINTER to File 10. Multiple allowed.
**Verify:** Multi-select from File 10 entries.

### Religion (.08) (File 2)
**VistA Rule:** POINTER to File 13. Must match.
**Verify:** Dropdown from File 13 entries.

### SC Percentage (.302) (File 2)
**VistA Rule:** Numeric 0-100 in increments of 10.
**Verify:** Validate: 55 → error. 50 → accepted.

### Clinic Name (.01) (File 44)
**VistA Rule:** Free text but no special chars beyond A-Z, 0-9, space.
**Verify:** Validate against VistA naming rules.

# SECTION 23: WORKFLOW EDGE CASES — What Happens When Things Go Wrong

### Two admins edit the same user simultaneously
Last write wins with no warning. Fix: add ETag/version check. If user was modified since load, show 'This user was modified by another admin. Refresh before saving.'

### Admin creates a user, then loses network before success screen
User exists in VistA but admin doesn't know the DUZ. Auto-save won't help because creation already succeeded. Fix: on reconnect, search for the user by name and show 'User may have been created. Please verify.'

### Admin assigns a role where 3 of 6 keys don't exist in VistA
3 keys assigned, 3 silently fail. User has partial capabilities. Fix: show per-key results on success: 'Assigned: PROVIDER, ORES, OR CPRS GUI CHART. Failed: ORCL-SIGN-NOTES (not found), ...'

### Admin deactivates a user who is currently logged in
User continues working until session expires naturally. Fix: server should check DISUSER on every authenticated API request, not just login. Deactivated → immediate 401 on next API call.

### VistA Docker restarts while admin is mid-wizard
Broker connection drops. Current state: auto-save exists but 'VistA unavailable' banner may not appear until submit. Fix: periodic health check during wizard (every 30s). Show warning immediately.

### Admin changes their OWN role to non-admin
Could lock themselves out. Fix: prevent removing your own XUMGR key. Show: 'Cannot remove your own administrative access.'

### Clone user → source user is deactivated between opening dialog and confirming
Clone may create a new user with 0 keys (source has no active profile). Fix: re-check source user status before executing clone.

### Patient registered with SSN that already exists
VistA may allow duplicate SSNs (with different DFNs) or reject depending on configuration. Fix: run duplicate check (ZVE PATIENT DUPL) before registration and show matches.

### Admission to a ward with 0 available beds
VistA may allow over-capacity admissions. Our UI should warn: 'Ward 3A has 0 available beds. Proceed anyway?'

### Discharge patient who has active medication orders
VistA may block or require discontinuation. Our UI should surface this: '5 active medication orders must be discontinued before discharge.'

### Transfer patient to a ward in a different division
May require inter-facility transfer process. Our UI doesn't distinguish same-facility vs cross-facility transfers.

### Admin searches for a patient who was merged/deleted
Merged patients should redirect to the surviving record. Deleted patients should show clear 'Record not found' with explanation.

### Browser crashes mid-form with unsaved changes (patient demographics)
Patient pages may not have auto-save like StaffForm does. Add sessionStorage auto-save for patient demographic edits.

### API returns 503 (VistA unreachable) on page that was working moments ago
Page should show current data with a banner: 'Connection to VistA lost. Data may be stale. [Retry]' — not clear the page.

### Admin tries to create a clinic with a name that already exists in File 44
VistA may allow duplicate names. Our UI should check and warn: 'A clinic named CARDIOLOGY already exists. Create anyway?'
