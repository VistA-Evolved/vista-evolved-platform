# 2. Product Architecture Summary
One tenant = one VistA instance.
VistA remains the clinical and operational system of record.
The product is a modern commercial client and integration layer over VistA.
Terminal and legacy GUIs (CPRS, VS GUI, BCMA) remain reference clients.
Product language on the front end is modern and tenant-friendly.
Semantic correctness on the back end maps exactly to VistA file structures and RPCs.
The product is organized as an enterprise platform with multiple VistA-aligned workspaces, each backed by specific VistA package families. The shell uses an app-launcher model where users open the workspace relevant to their role. All workspaces share one tenant context, one patient truth, one staff truth, and one location truth.

# 6. Precise Instructions for the UX Design Team
## 6.1 What You Receive
You receive the following artifacts, in this priority order:
Priority 1 (start immediately): Page Contracts Vol 1 (Admin User/Provider) — 11 pages, every field/button/dropdown/state specified with backend file/field mapping.
Priority 2 (start after Priority 1 underway): Page Contracts Vol 2 (Scheduling) — 20 pages, every field/button/dropdown/state specified with exact RPC names.
Priority 3 (receive when ready): Page Contracts Vol 3 (Patients) and Vol 4 (Clinical) — will contain 25-35 more pages.
Context documents: Module-to-Package Mapping Matrix (overall system understanding) and Terminal RPC Ledger (dependency and gap understanding).
## 6.2 How to Read a Page Contract
Each page contract in the volumes specifies:
Page ID and Name — e.g. USR-01: Staff Directory, SCH-02: Appointment Book
Purpose — what business job this page solves
Primary Role — who uses this page (System Admin, Scheduler, Clinician, etc.)
Layout description — header, filters, table, drawers, tabs, action bars
Every field — with its product label, backend file.field reference, data type, required/optional flag
Every dropdown — with where its options come from (which VistA file or RPC)
Every button — with its label, visibility rules (who sees it and when), the action it performs, and the exact backend RPC it calls
Every status/state — default, loading, empty, no-results, error, permission-denied, conflict, and any domain-specific states
Every warning — with the exact condition that triggers it and the severity level
Every confirmation — what must the user confirm before a high-risk action executes
## 6.3 Your Design Process (Step by Step)
Step 1: Read the page contract completely before opening your design tool.
Step 2: Count the fields, dropdowns, buttons, and states. Make sure your design accounts for ALL of them, not just the happy path.
Step 3: Design the default/loaded state first.
Step 4: Design the empty state and no-results state.
Step 5: Design every error and warning state mentioned in the contract.
Step 6: Design every confirmation dialog mentioned in the contract.
Step 7: Annotate your design with the Page ID, field labels matching the contract, and button labels matching the contract.
Step 8: Hand off to engineering with: (a) your design files, (b) the original page contract, (c) a mapping note confirming which contract fields map to which design elements.
## 6.4 Rules You Must Follow
Rule 1: Never add a field that is not in the page contract without requesting a contract amendment first.
Rule 2: Never remove a warning or confirmation that is in the page contract.
Rule 3: Never use raw VistA terminology (file numbers, IEN, DUZ, key names) on tenant-facing screens. Use the product vocabulary from the contracts.
Rule 4: Never design a Delete button for users, patients, or appointments. Use Deactivate, Cancel, or Mark Inactive per the contract.
Rule 5: Always design for all states listed in the contract, not just the happy path.
Rule 6: Action buttons must be state-dependent as specified. Example: the Check-In button only appears when status = Scheduled. The No-Show button only appears when the appointment time has passed and the patient was not checked in.
Rule 7: Design for desktop first, then tablet responsiveness. This is an enterprise operational system, not a consumer app.
Rule 8: The product should feel serious, enterprise-grade, high-density-capable, trustworthy, and efficient under operational pressure. Avoid playful consumer aesthetics.
## 6.5 What To Do When Something Is Unclear
If a page contract does not answer a question about a field, dropdown option source, button behavior, or state, the answer is: ASK before designing. Do not invent. The product must stay grounded in VistA truth. Filing a question against the contract is always safer than guessing.

# 7. Precise Instructions for the AI Coding / Development Team
## 7.1 What You Receive
You receive the following artifacts, in this priority order:
Priority 1: Terminal RPC Ledger — tells you exactly which RPCs exist, which are missing, and what custom wrappers need to be built.
Priority 2: Page Contracts — tell you exactly what each page needs: every backend call, every file/field touched, every input/output contract, every security requirement.
Priority 3: Module-to-Package Mapping Matrix — tells you the big picture of which packages power each workspace.
Reference: CPRS Delphi source code at github.com/WorldVistA/VistA/tree/master/Packages/Order Entry Results Reporting/CPRS/CPRS-Chart — the r*.pas files contain every RPC call CPRS makes. This is the single best reference for how a client talks to VistA.
## 7.2 How to Read a Page Contract (Engineering Perspective)
For each page contract, the engineering team should extract:
(a) BACKEND CALLS: Every RPC name listed in the contract. Existing RPCs can be called directly. Wrappers marked as needed must be built first.
(b) CONTEXT OPTIONS: Every RPC runs inside a B-type context option. For scheduling, this is SDECRPC or SDESRPC. For CPRS RPCs, this is OR CPRS GUI CHART. The user must have these options in their menu tree.
(c) FILE/FIELD REFERENCES: Every field maps to a VistA file number and field number. This tells you exactly what data to read/write.
(d) SECURITY RULES: Which keys the user must hold, what happens if they lack them, what must be enforced server-side vs client-side.
(e) STATE TRANSITIONS: What status changes are allowed, what side effects each action produces, what audit events should be recorded.
## 7.3 Your Build Process (Step by Step)
Step 1: Build the RPC Broker adapter first. This is the transport layer between your web frontend and VistA. It must handle: TCP connection to VistA listener, Access/Verify authentication, context creation (CreateContext for B-type options), RPC invocation, structured result parsing, session timeout, error propagation.
Step 2: Build the Scheduling workspace first (Phase 2). It uses only EXISTING RPCs. Zero custom M code required. This proves your broker adapter works, proves your page rendering works, proves your authentication works, before you need to write any M.
Step 3: Build the Admin User/Provider workspace (Phase 1). This requires CUSTOM WRAPPER RPCs in M. Use the page contracts to know exactly what each wrapper must do: what files to read/write, what fields, what security checks, what return structure.
Step 4: For each custom wrapper RPC, follow this exact pattern:
(a) Create an M routine in the ZVE namespace (e.g. ZVEUSR for user wrappers, ZVESIG for signature wrappers)
(b) Register the RPC in REMOTE PROCEDURE #8994 with NAME, TAG, ROUTINE, RETURN VALUE TYPE
(c) Create or use a B-type context option in OPTION #19 and add the RPC to its RPC multiple
(d) The wrapper must: validate inputs, enforce file-level security, call the appropriate FileMan APIs (DIE for edits, FILE^DICN for creates, etc.), handle errors, return structured results
(e) Test by calling the wrapper through RPC Broker and verifying the result matches what terminal would produce
Step 5: Build Patient Search using existing ORWPT RPCs (ORWPT LIST ALL, ORWPT LAST5, ORWPT ID INFO). Build registration using custom wrappers.
Step 6: Build Clinical Workspace using existing OR namespace RPCs. Study the CPRS Delphi r*.pas files to understand exact call patterns:
rCore.pas — core RPCs: ORWU USERINFO, ORWU NEWPERS, ORWU CLINLOC, ORWU DT
rOrders.pas — order RPCs: order actions, signatures, detail retrieval
fNotes.pas + TIU RPCs — note creation, retrieval, signing
fProbs.pas + ORQQPL RPCs — problem list CRUD
fMeds.pas + ORWPS RPCs — medication display
fLabs.pas + ORWLRR RPCs — lab result display
fConsults.pas + GMRC RPCs — consult workflows
## 7.4 Rules You Must Follow
Rule 1: Every RPC must be called inside a valid context option. Never call RPCs without CreateContext.
Rule 2: Never invent a data model that does not map to a VistA file. If the page contract says the field comes from #200 .01, your code reads from #200 .01. Not from a parallel database.
Rule 3: Never implement DELETE for users (#200), patients (#2), or appointments (#409.84). Implement deactivation, cancellation, or status change per the contracts.
Rule 4: Access codes, verify codes, and electronic signature codes must NEVER be stored or transmitted in plain text. They must be hashed using Kernel-compatible hashing.
Rule 5: Every custom wrapper RPC must be verified against terminal equivalence: the same action performed via terminal and via your wrapper must produce identical VistA database state.
Rule 6: Every page must handle the backend-unavailable state. If the broker connection fails, show an error and disable destructive actions. Never silently fail.
Rule 7: When you encounter a field or action not covered in the page contracts, STOP and request a contract amendment. Do not guess backend behavior.
Rule 8: Use the CPRS Delphi source code as your RPC discovery tool. The r*.pas files contain every RPC name CPRS calls. When building a clinical page, find the corresponding CPRS Delphi file first and study its RPC calls before writing any code.

# 11. Definition of Done for Each Workspace
A workspace is not considered complete until ALL of the following are true:
(1) Page contracts exist for every page in the workspace.
(2) High-fidelity designs exist for every page, covering all states.
(3) All required RPCs (existing or custom wrapper) are verified working.
(4) Every page is implemented and wired to its specified backend calls.
(5) Every field reads from / writes to the correct VistA file and field.
(6) Every dropdown loads options from the specified source.
(7) Every button is visible only to the correct roles in the correct states.
(8) Every warning and confirmation appears as specified.
(9) Every error state is handled (backend unavailable, permission denied, validation failed, conflict).
(10) Terminal equivalence is verified: actions in the modern client produce the same VistA database state as the equivalent terminal action.
