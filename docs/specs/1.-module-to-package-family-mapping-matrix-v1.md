# 1. Purpose of This Document
This is the first implementation-grade truth-layer artifact for the VistA Evolved project. It exists because the previous planning materials (domain packets, workspace architecture, visual diagrams) were directionally correct but still too abstract to safely hand to a design team or AI-coding-heavy engineering team.
This document answers, for each product workspace:
(a) Which exact VistA package families power it?
(b) What legacy client surfaces already exist (CPRS, VS GUI, BCMA, etc.)?
(c) What is mostly terminal-shaped today with no existing GUI?
(d) What is configuration/admin work vs. daily operational work?
(e) What RPC coverage is known to exist, and where are the gaps?
(f) Which workspaces are realistic in MVP Wave 1, 2, or later?
(g) What tenant business profiles (full hospital, standalone pharmacy, standalone lab, etc.) need which workspaces?
## 1.1 How to Use This Document
Design team: Use this to understand which workspaces to design first, what real package truth sits behind each one, and where GUI precedents already exist that you can study for semantic reference (not visual copying).
Engineering / AI coding team: Use this to understand which packages you will need to call, where existing RPCs likely exist, where custom wrapper RPCs will be needed, and which terminal-only configuration surfaces must be investigated before screen contracts can be written.
Product team: Use this to prioritize MVP waves and understand which tenant profiles need which workspace subsets.

# 2. Architecture Foundation
One tenant = one VistA instance. VistA remains the clinical and operational system of record. The new product is a modern client and integration layer. Terminal and legacy GUIs remain reference clients, not visual templates. Product language on the front end is modern and tenant-friendly; semantic correctness on the back end maps exactly to VistA structures.
VistA is an integrated EHR system whose applications share a common data store built on the M language. The Kernel component acts as middleware between the OS and applications. VistA integrates its applications using APIs, RPCs, and HL7 messaging. (Source: VA VistA Monograph, July 2023)
CPRS is only one clinical client. It was written or modified to communicate with VistA packages for Pharmacy, Lab, Radiology, Allergy Tracking, Consults, Dietetics, Progress Notes, Problem List, Scheduling, MAS/ADT, Kernel, FileMan, Vitals, PCMM, PCE, TIU, ASU, and Clinical Lexicon. OE/RR was completely rewritten for CPRS. (Source: VistApedia CPRS Overview)
The RPC Broker is the client/server bridge. Client applications connect to a VistA Broker Listener, authenticate via Access/Verify codes, then execute RPCs registered in the REMOTE PROCEDURE file (#8994). RPCs are secured by B-type context options in the OPTION file (#19). (Source: RPC Broker 1.1 Systems Management Guide, May 2020)

# 4. Workspace Deep-Dives
Each workspace below is expanded with: package truth, terminal option families, known RPC families, known gaps, config-vs-usage split, and implications for design and engineering.
## 4.1 Clinical Workspace
### Package Truth
Primary package: Order Entry/Results Reporting (OR). Namespace: OR, OCX. This is the largest RPC-backed package in VistA. CPRS was built on top of it. The OR namespace contains hundreds of RPCs covering orders, notes, results, patient lookup, user info, notifications, consults, allergies, problems, vitals, and more.
Supporting packages: TIU (Text Integration Utilities) for clinical documentation/notes. GMRC (Consult/Request Tracking) for consult workflows. Allergy Tracking for adverse reactions. Problem List for longitudinal problems. Vitals/Measurements for vital signs. PCE (Patient Care Encounter) for encounter capture. Clinical Lexicon for standardized terminology.
### Legacy GUI Precedent
CPRS GUI is the strongest precedent in all of VistA. Its Delphi source code is publicly available in the WorldVistA repository under Packages/Order Entry Results Reporting/CPRS/. The source code reveals every RPC call, every screen, every workflow. This is the single best reference for understanding how a modern clinical client should communicate with VistA.
### Known RPC Families (OR Namespace)
ORWU (user utilities): ORWU NEWPERS, ORWU USERINFO, ORWU CLINLOC, ORWU DT, etc.
ORWPT (patient utilities): ORWPT LIST ALL, ORWPT LAST5, ORWPT ID INFO, etc.
ORWPS (medications/pharmacy display): cover sheet meds, active meds.
ORWOR (orders): order actions, order details, signatures.
ORQQAL (allergies): allergy lists, allergy details.
ORQQPL (problem list): problem lists, problem details.
ORWLRR (lab results): lab result retrieval.
ORWRA (radiology): radiology reports.
TIU (notes): TIU CREATE RECORD, TIU GET RECORDS, TIU SIGN RECORD, etc.
### Config vs Usage Split
DAILY USE (covered by CPRS RPCs): Patient lookup, chart review, note writing, order entry, result viewing, allergy management, problem management, consult management.
CONFIGURATION (mostly terminal): Order dialog setup, order set creation, notification parameters, orderable item maintenance, display group configuration, team/list management, GUI parameter settings, print/device configuration. The CPRS Setup Guide (July 2025) documents extensive terminal-based configuration including CPRS Configuration menus, Order Check Expert System, ORMTIME parameters, Consult service setup, Notification management, and GUI parameter tools.
### Gaps and Custom Wrapper Needs
Clinical daily-use workflows: Strong RPC coverage. Most can be modernized by studying CPRS source code and reusing existing RPCs.
Clinical configuration: Weak RPC coverage. Order dialog maintenance, notification parameters, team/list management, and many setup tasks are terminal-only and will require custom wrapper RPCs or new admin surfaces.

## 4.2 Patients and Registration
### Package Truth
Primary package: Registration/ADT/PIMS. The ADT module is the focal collection point of patient information, encompassing demographic, employment, insurance, and medical history data. Many other modules (Laboratory, Pharmacy, Radiology, Nursing, Dietetics) utilize information gathered through ADT options. (Source: VistA Monograph)
Supporting packages: MPI/PD (Master Patient Index / Patient Demographics) for identity correlation and ICN handling. Duplicate Record Merge for governed duplicate resolution. Patient Record Flags for clinical warnings and cautions.
Core file: PATIENT file (#2). This is the foundational patient identity record in VistA.
### Legacy GUI Precedent
No strong standalone GUI for registration/ADT. CPRS has patient selection and some demographic display, but the full registration workflow is terminal-based. Some patient list and selection patterns from CPRS are useful references.
### Terminal Option Families
Register a Patient: Used when patient is applying for care.
Load/Edit Patient Data: Used when patient is not applying for care but record needs to be entered or maintained. These are explicitly different workflows per the ADT Registration manual.
ADT options: Admission, Discharge, Transfer movements.
Means Test and eligibility determination.
Patient Record Flag management.
Duplicate Record Merge utilities (Manager Menu, Merge Menu, Site Parameter Menu).
### Known RPC Families
DG namespace: Some registration-related RPCs exist (DG REGISTER PATIENT, some demographic RPCs).
ORWPT namespace: Patient lookup RPCs used by CPRS (ORWPT LIST ALL, ORWPT LAST5, etc.).
VAFCTFU namespace: Some treating facility / MPI-related calls.
### Gaps and Custom Wrapper Needs
MAJOR GAP: Full registration workflow has limited RPC coverage. Terminal-based Register a Patient and Load/Edit Patient Data flows are complex multi-step processes that will likely need custom wrapper RPCs for safe modernization.
MAJOR GAP: Duplicate Record Merge is a governed, non-reversible process with its own security model. It cannot be reduced to a casual edit button. Will need very careful wrapper design if exposed in modern UI.
MAJOR GAP: Patient Record Flag management is terminal-based and will need wrappers.

## 4.3 Scheduling
### Package Truth
Primary packages: Scheduling/PIMS, VistA Scheduling Enhancements (VSE) / VS GUI. Namespace: SD, SDES, SDES2, SDEC.
Core files: HOSPITAL LOCATION (#44), SDEC APPOINTMENT (#409.84), SDEC APPT REQUEST (#409.85), MEDICAL CENTER DIVISION (#40.8), INSTITUTION (#4).
### Legacy GUI Precedent
VS GUI / VSE is the STRONGEST modern GUI precedent in all of VistA outside CPRS. The VSE Technical Manual documents extensive RPC-backed scheduling operations including graphical patient, clinic, provider, and resource scheduling, multiple clinic schedule views, check-in workflows, cut/paste rescheduling, and appointment request management.
### Known RPC Families (SDES/SDES2 Namespace)
Appointment operations: SDES2 CREATE APPOINTMENT, SDES2 CREATE WALKIN APPT, SDES GET APPTS BY CLINIC, SDES2 GET APPTS BY PATIENT DFN3.
Appointment request operations: SDES2 CREATE APPT REQ, SDES2 EDIT APPT REQ, SDES2 DISPOSITION APPT REQ.
Clinic operations: SDES2 CREATE CLINIC, SDES2 EDIT CLINIC, clinic inactivation/reactivation RPCs.
Clinic availability: Cancel clinic availability, restore clinic availability.
Check-in: SDES2 CHECKIN, SDES SET APPT CHECK-IN STEP.
Search: SDES GET CLINIC INFO, SDES GET DIVISION LIST, person/clinic search RPCs.
### Config vs Usage Split
DAILY USE (strong RPC coverage): Appointment booking, rescheduling, cancellation, check-in, appointment requests, clinic schedule viewing. These are well-covered by the SDES/SDES2 RPC surface.
CONFIGURATION (partly terminal): Clinic creation/editing has some RPC coverage via SDES2. Schedule template definition, stop code assignment, availability pattern setup, and some clinic parameters may still require terminal or wrapper work.
### Gaps and Custom Wrapper Needs
RELATIVELY SMALL GAP compared to other domains. VSE has done much of the modernization groundwork. The main gaps are in complex schedule-template configuration and some site-parameter-level scheduling settings.
This is the recommended FIRST vertical slice for proving the modern client architecture.

## 4.4 Pharmacy
### Package Truth
This is NOT one package. It is at least three distinct package families:
(a) Outpatient Pharmacy (PSO): Prescription processing, fills, renewals, ePrescribing, suspense, refills, controlled substances. Namespace: PSO.
(b) Inpatient Medications (PSJ): Inpatient drug order management, pharmacist verification, nurse administration workflows. Namespace: PSJ. The PSJ manuals are targeted to pharmacists, nurses, and supervisors and explicitly interact with CPRS and BCMA.
(c) Bar Code Medication Administration (BCMA/PSB): Bedside barcode verification and medication administration. Namespace: PSB, ALPB. BCMA is its own GUI application family with its own manager workflows.
Supporting packages: Pharmacy Data Management (PDM/PSS) for drug file, formulary, orderable item maintenance. National Drug File (NDF) for national drug data. Order Checks for medication safety validation. Pharmacy Product System - National (PPS-N) for enterprise pharmacy data.
### Legacy GUI Precedent
BCMA GUI: Dedicated barcode medication administration client. This is a real, separate GUI.
CPRS ordering: CPRS provides the clinician-facing medication ordering interface, but this is upstream only. The actual pharmacy fulfillment, verification, and dispensing work happens in terminal-based pharmacy tools.
Some outpatient pharmacy tools exist but are limited.
### Terminal Option Families (VERY LARGE)
Outpatient: Patient prescription profile, process a prescription, suspense management, refill processing, controlled substance management, medication action profiles, label printing, CMOP (Consolidated Mail Outpatient Pharmacy) management, ePrescribing workflows.
Inpatient: Unit dose dispensing, IV admixture, ward-level medication management, pharmacist order verification, medication order entry backdoor.
Configuration: Drug file maintenance (file #50), formulary management, orderable item setup, dispense drug setup, site parameter configuration (file #59), division setup, ward parameters (file #59.6), administration schedule editing, medication instruction file (#51), standard administration times, Clozapine registration, auto-refill configuration, bingo board setup. (Source: Pharmacy Setup Checklist from WorldVistA FOIA)
### Known RPC Families
PSO namespace: Some outpatient pharmacy RPCs exist but the surface is much smaller than the OR namespace.
PSB namespace: BCMA-related RPCs for medication administration verification.
CPRS-side: OR namespace RPCs handle clinician-facing ordering, but pharmacy fulfillment RPCs are limited.
### Gaps and Custom Wrapper Needs
VERY LARGE GAP: Pharmacy fulfillment workflows (dispensing, verification, release, queue management, inventory-adjacent logic) have very limited RPC coverage. These are heavily terminal-driven and will require significant custom wrapper RPC development.
VERY LARGE GAP: Drug file/formulary/orderable item maintenance is entirely terminal and is critical configuration work that affects everything CPRS displays for medication ordering.
MAJOR DESIGN CONSIDERATION: The Pharmacy workspace MUST distinguish between outpatient and inpatient medication worlds. They are separate packages with different workflows, different user roles, and different operational realities. A single generic pharmacy module will fail.

## 4.5 Laboratory
### Package Truth
Primary package: Laboratory (LR). Namespace: LR. Core files: LABORATORY TEST (#60), LAB DATA (#63), AUTO INSTRUMENT (#62.4), ACCESSION (#68), and many others.
The Laboratory User Manual organizes work into major menu families: Phlebotomy, Accessioning, Process Data in Lab, and Quality Control. (Source: VA VDL Laboratory documentation)
### Legacy GUI Precedent
No strong standalone lab GUI for the fulfillment side. CPRS displays lab results to clinicians (via OR/ORWLRR RPCs), but this is downstream viewing only. The actual laboratory work (accessioning, specimen processing, instrument interfacing, result entry, verification, QC) is heavily terminal-based.
### Terminal Option Families (VERY LARGE)
Phlebotomy: Collection lists, label printing, specimen management.
Accessioning: Log-in specimens, assign accession numbers, route to sections.
Process Data in Lab: Enter results manually, receive from auto-instruments, verify results, release results.
Quality Control: QC data entry, Levey-Jennings review, proficiency testing.
Configuration: Test definition and setup (#60), instrument interface configuration (#62.4), accession area setup (#68), collection sample definitions, lab site parameters, verify-by and comment rules.
### Known RPC Families
VERY LIMITED for lab fulfillment. The OR namespace has ORWLRR (lab result retrieval for CPRS), but this is read-only downstream viewing.
Some LR-namespace RPCs may exist for specific integrations, but the core lab operational workflow (accessioning, result entry, verification) has minimal RPC coverage.
### Gaps and Custom Wrapper Needs
VERY LARGE GAP: Almost the entire laboratory fulfillment workflow (accessioning, result entry, instrument interface management, verification, QC) will require custom wrapper RPCs. This is one of the most terminal-intensive domains in VistA.
INSTRUMENT INTEGRATION: Auto-instrument interfaces are a critical part of lab operations. The AUTO INSTRUMENT file (#62.4) and related instrument communication routines will need careful study. HL7 interfaces for lab instruments add another integration layer.
RECOMMENDATION: Start with result VIEWING in the modern client (reusing OR namespace RPCs), defer lab fulfillment to Wave 3+ after wrapper RPC strategy is proven in simpler domains.

## 4.6 Radiology and Imaging
### Package Truth
Two distinct but related package families:
(a) Radiology/Nuclear Medicine (RA): Order fulfillment, procedure workflow, report authoring/verification. Namespace: RA. Core files include RAD/NUC MED PROCEDURES (#71), RAD/NUC MED PATIENT (#70), RAD/NUC MED ORDERS (#75.1), and others.
(b) VistA Imaging (MAG): Image capture, clinical display, storage, DICOM gateway, PACS integration, TeleReader, VistARad. The Imaging Technical Manual (Rev 47, March 2025) documents extensive infrastructure including Clinical Display workstation, Background Processor, DICOM gateways, and exchange services.
### Legacy GUI Precedent
VistA Imaging has strong GUI precedents: Clinical Display workstation for image viewing, VistARad for radiologist workflow, TeleReader for remote reading, Clinical Capture for image acquisition. These are real, deployed Windows applications.
Radiology procedure/reporting workflow is more terminal-based.
CPRS displays radiology results and can launch imaging viewers, but the fulfillment side is separate.
### Known RPC Families
RA namespace: Very small RPC surface for radiology procedure workflow. Vivian/DOX shows radiology has a very limited remote procedure set compared to OR.
MAG namespace: Imaging has its own RPC families for image retrieval, display, and clinical context, but these are tied to the Imaging workstation architecture.
OR namespace: ORWRA RPCs provide radiology report viewing for CPRS.
### Gaps and Custom Wrapper Needs
LARGE GAP: Radiology procedure workflow (exam registration, status tracking, report authoring/verification) has very limited RPC coverage and will need custom wrappers.
IMAGING COMPLEXITY: VistA Imaging is not just a viewer. It includes DICOM gateways, HL7 interfaces, Background Processor for routing, and commercial PACS integration pathways. Modernizing imaging requires understanding the gateway/exchange architecture, not just building web pages.
RECOMMENDATION: Start with image/report viewing integration in the modern client. Defer deep radiology procedure workflow to Wave 3+.

## 4.7 Billing and Revenue
Primary packages: Integrated Billing (IB), PCE/Encounter families, Accounts Receivable, Fee Basis.
Terminal intensity: VERY HIGH. Billing is almost entirely terminal-driven with minimal RPC coverage.
CRITICAL NON-VA CONCERN: VA Integrated Billing is built around VA-specific eligibility, means testing, and benefit categories. For international/commercial use, significant overlay or replacement logic is needed for: local payer/HMO/insurer adapters, country-specific billing codes (ICD, CPT equivalents), claims processing workflows, revenue cycle management. This is one of the areas where VistA truth may not fully own the commercial product truth.
MVP Wave: Wave 3+ at minimum. Basic encounter/charge capture visibility may come earlier through PCE integration, but full billing modernization is a major separate effort.
## 4.8 Supply and Inventory
Primary package: IFCAP (PRC). Covers procurement, inventory, control points, delivery orders, warehouse management.
Terminal intensity: VERY HIGH. Entirely terminal-driven. IFCAP has its own user guides organized by role (requestor, control point, warehouse).
RPC coverage: Minimal. Custom wrappers needed for everything.
MVP Wave: Wave 4+. Essential for full hospital operations but not the first modernization target.
## 4.9 Admin and Security
Primary packages: Kernel (XU), FileMan (DI), MailMan, TaskMan, Menu Manager.
Terminal intensity: VERY HIGH. Core system administration (users, keys, options, parameters, devices, site config) is almost entirely terminal.
Known precedent: WebVRAM provides a limited web-based user administration precedent that can inform design.
MVP Wave: Wave 1 for user/provider/key management subset (essential for making other workspaces work). Wave 2+ for broader system admin.
CRITICAL NOTE: This workspace powers everything else. Users, security keys, menu options, division assignments, electronic signature setup, provider designation, and site parameters are ALL configured through Kernel/FileMan terminal tools. Without modernizing at least the user/provider subset, no other workspace can function properly for a new tenant.
## 4.10 Interfaces and Devices
Primary packages: HL7 (HL), VistA Imaging HL7/DICOM pathways, instrument interface families.
Terminal intensity: VERY HIGH. Interface configuration, message monitoring, error handling, and gateway management are all terminal or specialized-tool operations.
Device families to consider: Lab instruments (AUTO INSTRUMENT #62.4), vitals devices, imaging modalities (DICOM), barcode scanners (BCMA), pharmacy dispensing hardware, printers/label printers.
MVP Wave: Wave 3+ for a monitoring/management dashboard. Device-specific integration happens within each department workspace.
## 4.11 Analytics and Oversight
No single VistA package owns analytics. Reports are scattered across packages. This workspace would likely be an overlay reporting/analytics layer using data extracted from VistA, not direct VistA terminal operations.
MVP Wave: Wave 2+ for basic operational dashboards, Wave 4+ for advanced analytics.

# 6. Recommended MVP Wave Sequence
## Wave 1 - Operational Foundation
Goal: Prove the modern client architecture end-to-end with the safest, best-RPC-covered domains.
Workspaces: Admin/Security (user/provider subset), Locations, Scheduling, Patients/Registration (search and chart shell).
Why: These have the strongest RPC coverage (especially Scheduling via VSE), the lowest clinical-write risk, and are required before any other domain can function.
## Wave 2 - Clinical Foundation and Viewing
Goal: Deliver a usable clinical workspace with read-heavy and limited-write capabilities.
Workspaces: Clinical Workspace (chart shell, notes viewing, result viewing, allergy/problem viewing, limited note creation), Lab result viewing, Imaging/report viewing, basic operational dashboards.
Why: Uses the large existing OR RPC surface. Read-heavy work is safer. Note creation is a well-understood TIU workflow.
## Wave 3 - Clinical Writes and Department Basics
Goal: Enable meaningful clinical work and begin department fulfillment surfaces.
Workspaces: Clinical Workspace (order entry, signature workflows), Pharmacy (outpatient basics), Lab (accessioning basics), Billing (encounter/charge capture basics).
Why: This is where custom wrapper RPCs become heavily needed. Must be preceded by proven wrapper RPC governance.
## Wave 4+ - Full Department Operations
Goal: Complete the product for full hospital operations.
Workspaces: Pharmacy (inpatient, BCMA), Lab (full fulfillment, instruments, QC), Radiology (procedure workflow), Billing (full revenue cycle), Supply/Inventory (IFCAP), Interfaces (full monitoring/management), Analytics (advanced).

# 7. What Must Happen Before Design Can Safely Proceed
This document is the first truth-layer artifact. It is necessary but NOT sufficient. Before a design team can safely produce page-level wireframes, the following additional artifacts must be created:
(1) Terminal Option Inventory: Map option name to user job to package to downstream effect for each domain being modernized.
(2) CPRS Dependency Map: For each CPRS tab/workflow, identify the package-side configuration and fulfillment work behind it.
(3) RPC Coverage Ledger: Per workflow action, document whether an RPC exists, is adequate, or a wrapper is needed.
(4) Missing Wrapper Ledger: For every identified gap, specify the routine/API to wrap, the context option needed, and the return contract.
(5) Page Inventory Matrix: List every page needed per workspace with purpose, role, objects, and actions.
(6) Screen Contracts: Per page, define every field, button, dropdown, tab, state, warning, and backend touchpoint.
(7) Page-to-RPC/Wrapper Maps: Per page, map every UI control to its exact backend call.
Without these, the design team will still invent, and the AI coder will still guess.

# 8. Critical Warnings for All Teams
WARNING 1: Do not assume CPRS covers VistA.
CPRS is one clinical client over a much larger ecosystem. Pharmacy fulfillment, lab accessioning, radiology procedures, billing, inventory, system administration, and interface management are NOT covered by CPRS.
WARNING 2: Do not assume all workflows have RPCs.
Many workflows are terminal-only. The RPC surface varies enormously by package: OR has hundreds of RPCs; Radiology has very few; Lab fulfillment has almost none; IFCAP has essentially none.
WARNING 3: Do not treat Pharmacy as one generic module.
Outpatient Pharmacy, Inpatient Medications, and BCMA are three separate VistA package families with different workflows, users, and operational realities.
WARNING 4: Do not skip the configuration/admin side.
The reason CPRS shows what it shows is because extensive terminal-based configuration was done: order dialogs, orderable items, formulary, notification parameters, clinic setup, user/provider setup, security keys, division assignments, site parameters. Without modernizing the configuration side, a modern clinical UI will be hollow.
WARNING 5: Do not treat deletion as normal.
VistA generally does not delete records. The real lifecycle includes: inactivation, cancellation, merge (non-reversible), addendum, and audit preservation. Design for this reality.
WARNING 6: Do not build international billing from VA billing.
VA Integrated Billing is built around VA-specific eligibility and benefit structures. International/commercial billing requires overlay services, local payer adapters, and country-specific coding systems.
