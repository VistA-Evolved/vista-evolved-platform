# 1. Foundation: IFCAP — The Most Terminal-Dependent Workspace
IFCAP (Integrated Funds Distribution, Control Point Activity, Accounting and Procurement) is a VistA subsystem that manages the complete supply chain and financial procurement lifecycle. It integrates three historically separate areas: Funds Distribution (budget allocation), Control Point Activity (spending authorization), and Accounting & Procurement (purchasing and receiving).
IFCAP has ZERO known RPCs. It is the most terminal-dependent workspace in the entire VistA system. Every page, every action, every data read requires custom wrapper RPCs.
IFCAP V5.1 is documented across 6+ separate user guides corresponding to distinct user roles, each with their own menu tree and operational workflow. The Technical Manual file list spans Tables 5.2 through 5.9 covering files #410-#447. The system interfaces with the Financial Management System (FMS) in Austin for all financial document processing.
Source: IFCAP TM: 'Distribution, Control Point Activity, Accounting and Procurement (IFCAP) software, which is an application in the Veterans Health Information Systems & Technology Architecture (VistA).' GIP Manual: 'The IFCAP General Inventory System is used to manage the receipt, distribution, and maintenance of stock items received for the supply warehouse from outside vendors and distributed to primary inventory points.'

## 1.4 Segregation of Duties
IFCAP enforces segregation of duties, particularly for 1358 obligations (Miscellaneous Obligation). The Online Certification System (OLCS) verifies that the person certifying an invoice for a 1358 is NOT the same person who requested, approved, or obligated it. IFCAP uses electronic signature codes (legally binding per AT Guide: 'each IFCAP user has a signature code that functions legally as their signature') for obligation processing.
The modern UI must enforce the same segregation of duties — a user who creates a request should not be able to approve that same request; a user who obligates should not certify the same invoice.

# 5. Rules for Design and Engineering
Rule 1: IFCAP has ZERO RPCs. Every page needs custom wrappers. This workspace is Wave 4+ and must not be attempted until the wrapper governance model is proven.
Rule 2: FMS document transmission is NON-NEGOTIABLE. Every financial transaction (obligation, adjustment, payment, cancellation, vendor change) must trigger the corresponding FMS document. Wrappers that bypass FMS will cause financial system desynchronization.
Rule 3: Electronic signatures are legally binding for obligation processing. The modern UI must prompt for e-sig code and hash it (same Encrypt() pattern as Volumes 4/5) before obligation can proceed.
Rule 4: Segregation of duties must be enforced. A user who creates a request cannot approve it. A user who obligates cannot certify the invoice. Check user DUZ against prior actors in the transaction chain.
Rule 5: Control Point balance tracking must be real-time. When approving a request, show the CP balance impact BEFORE the user confirms (IFCAP terminal behavior: 'IFCAP will show the effect of the transaction on the estimated balance').
Rule 6: Purchase order numbering uses a series-based system. The numbering series (2-3 character code) classifies the PO by appropriation, payment method, and procurement source. The modern UI must support this categorized numbering.
Rule 7: Vendor changes in IFCAP trigger VRQ documents to FMS for vendor file synchronization. Never edit a vendor in #440 without triggering the VRQ.
Rule 8: Receiving requires verification against the PO. The warehouse clerk must compare delivered goods to PO line items. Discrepancies (short, over, wrong, damaged) must be recorded and routed to the Purchasing Agent.
Rule 9: GIP (Generic Inventory Package) perpetual inventory tracking is optional per inventory point. Some inventory points track perpetual inventory, others don't. The modern UI must handle both modes.
Rule 10: Fiscal year boundaries matter. Unobligated funds may expire at fiscal year end. The UI should alert CP Officials to approaching deadlines.
Rule 11: Purchase card orders follow a simplified workflow compared to formal POs but still require CP funding and IFCAP tracking.
Rule 12: IFCAP purging (removing old records per the 3-year retention requirement) is a maintenance operation using PurgeMaster (#443.1). The modern UI should provide purge status visibility but purge execution should remain a scheduled maintenance task.
Rule 13: The 826 (IFCAP) Report is the comprehensive financial report for CP status. The modern UI must be able to generate this report or its equivalent from the CP Dashboard.
Rule 14: Source code classification (mandatory government sources vs open market) affects procurement rules. Open market purchases require a Local Procurement Reason Code justification per P.L. 100-322.
