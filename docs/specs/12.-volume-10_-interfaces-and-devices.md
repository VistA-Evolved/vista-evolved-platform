# 1. Foundation: VistA's Messaging Backbone
This workspace provides operational visibility and management for the interface infrastructure that connects VistA to external systems. VistA is not an isolated system. It communicates constantly with dozens of external systems: laboratory analyzers, imaging PACS, voice recognition platforms, pharmacy dispensing cabinets, the Master Patient Index (MPI), the Financial Management System (FMS), Oracle Health/Cerner, ePrescribing networks (Surescripts), insurance payer systems, and other VA facilities. Every one of these connections relies on VistA's HL7 messaging infrastructure.
The HL7 V1.6 Site Manager & Developer Manual describes VistA HL7 as 'an M-based software product that assists M-based VISTA applications by providing a means for those applications to send and receive HL7 messages. VISTA HL7 provides a repository of supported HL7 transactions, connectivity between systems, and guaranteed delivery of messages using supported lower layer protocols.' It supports point-to-point interfaces, publish-and-subscribe event distribution, and content-based dynamic routing.
VistA has TWO coexisting HL7 engines:
(A) HL7 V1.6 Engine (original): Uses PROTOCOL #101 for event driver/subscriber model. Port 5000 (production), 5025 (test). Messages stored in #772 (text) and #773 (admin/tracking). Links in #870. Applications in #771.
(B) HLO Engine (HL7 Optimized, patch HL*1.6*126): Uses HLO APPLICATION REGISTRY #779.2 instead of PROTOCOL #101. Separate port. Namespace HLO*. 'There is virtually no shared code between the HL 1.6 and HLO engines' per the HLO supplement. HL Logical Links (#870) are still used but with a different port field (#400.08).
Every VistA clinical package that communicates with external systems uses one or both engines. Lab sends results via HL7 to instruments and POC devices. Radiology sends ORM/ORU messages to PACS and voice recognition systems (TalkStation, PowerScribe). Pharmacy sends dispensing messages (RDS). ADT sends admission/discharge/transfer notifications (A01-A08). IB sends insurance verification queries via FSC. IFCAP sends financial documents to FMS.
OPERATIONAL CRITICALITY: If the HL7 infrastructure fails, clinical data stops flowing. Lab results from instruments do not file into LAB DATA #63. Radiology reports from voice recognition do not arrive in REPORTS #74. ADT messages to MPI do not transmit. Insurance verification queries stop. This workspace exists so operations teams can detect problems within seconds and resolve them before patient care is affected.

Filers are the background processes that actually move messages through the system. They are managed via the HL7 Filer and Link Management Options menu. HL7 SM Chapter 3.1: Managing Filers.

The publish/subscribe model means: adding a new external system that needs radiology reports requires ONLY adding a new subscriber protocol linked to a new logical link. The radiology package generating the RA RPT event does not need any modification. This is the foundation of VistA's interface extensibility.

## 3.3 INT-03: HL7 Error Investigation
### Purpose
Focused error investigation page. Shows ONLY errored messages with full diagnostic detail. Provides the tools and context needed to diagnose WHY a message failed, identify whether it's an isolated issue or a pattern, and determine WHAT corrective action to take. Based on HL7 SM Chapters 4 (Troubleshooting: Solving Transmission Problems) and 5 (VISTA HL7 Troubleshooting Options).
### Error Categories and Diagnostic Guidance
Error Category
Common Causes
Diagnosis Method
Resolution Steps
Severity
Connection Refused
Destination system is down or unreachable. Wrong IP address or port in #870 configuration. Firewall blocking the connection. Destination HL7 listener not running.
(1) Ping test from INT-01/INT-04. (2) Verify #870 IP/port matches destination's actual listener address. (3) Check network connectivity (traceroute). (4) Contact destination system admin.
Fix destination availability. Correct #870 configuration if wrong. Update firewall rules if blocked. Start destination listener.
HIGH — all messages to this destination are failing
Timeout
Destination system is responding too slowly. Network latency exceeds timeout threshold. Destination overloaded.
(1) View Transmission Log timing data. (2) Check network response times. (3) Check destination system load/performance.
Often resolves on retry. If persistent: investigate destination system performance. Consider increasing timeout threshold in link config.
MEDIUM — intermittent, may self-resolve
Application Reject (NAK)
Destination application rejected the message content. Invalid data, missing required fields, patient not found at destination, diagnostic code validation failure, unsupported message version.
(1) View NAK message text in ACK detail section. The MSA segment contains rejection reason. (2) Examine the rejected message content in #772 for data issues. (3) Check if patient exists at destination system.
Fix data issue in source message. Common fixes: register patient at destination, correct diagnostic codes, update message version. May require coordination between sending and receiving application teams.
MEDIUM-HIGH — content-level issue requiring investigation
Filer Stopped
The incoming or outgoing filer background process has stopped running. This is a SYSTEM-WIDE condition affecting ALL links.
(1) Check filer status in INT-01 system banner. (2) Check TaskManager for filer process. (3) Review system error logs for filer crash cause.
Restart filer. Investigate crash cause (disk space, memory, error trap). HL7 SM Section 4.1: Filer Problems.
CRITICAL — affects ALL interfaces
Queue Corruption
Stub records or corrupted pointers in the link's In or Out queue. HL7 SM Section 4.5: 'Stub Records and Corrupted Pointers in Link Queues.'
(1) Run HL7 integrity check utilities. (2) Inspect queue global structure (^HLCS(870,IEN,...)). (3) Look for IEN mismatches or orphaned queue entries.
Follow HL7 SM Section 4.5 cleanup procedures. May require careful manual global editing. Always back up before queue cleanup.
HIGH — can block all message delivery for affected link
Authentication / Facility Mismatch
Sending or receiving facility in the message header does not match the expected facility for this link/application. HL*1.6*57 security enhancements: 'Receiving Facility Required?' and 'Sending Facility Required?' protocol validations.
(1) Check MSH-4 (Sending Facility) and MSH-6 (Receiving Facility) in the rejected message. (2) Compare to link and application configuration in #870/#771.
Correct facility identification in #870 institution field or #771 application configuration. Ensure MSH facility fields match what the destination expects.
MEDIUM — configuration issue
Version Mismatch
HL7 version in message header (MSH-12) doesn't match version expected by destination application.
Check MSH-12 (Version ID) against destination requirements. HL7 SM supports versions 2.1, 2.2, 2.3, 2.3.1.
Update interface protocol configuration to generate correct HL7 version. Or update destination to accept the version being sent.
LOW-MEDIUM — configuration issue
Message Format Error
Malformed HL7 segments, incorrect delimiters, missing required segments (e.g. MSH without proper encoding characters), truncated messages.
Examine raw message text in #772 for structural problems: missing segment terminators, wrong field separators, truncation, encoding issues.
Fix source application's message generation routine. This is an application development issue, not an infrastructure issue.
MEDIUM — requires source app fix

# 5. Rules for Design and Engineering
Rule 1: Starting/stopping HL7 links is the most operationally sensitive action in this workspace. It directly and immediately affects clinical data flow to/from external systems. Require admin mode activation, double-confirmation with link name typing, reason text, and full audit logging for all start/stop operations.
Rule 2: Build READ-ONLY monitoring first (INT-01 dashboard, INT-02 queue browser, INT-03 error investigation, INT-05 instrument monitor, INT-06 imaging monitor) in Wave 3. Add WRITE operations (link start/stop, message retry/clear/reprocess, configuration editing) only in Wave 4+ after the governance model is established and tested.
Rule 3: Auto-refresh is essential for operational monitoring. The dashboard (INT-01) should refresh at least every 60 seconds. Message queue views (INT-02) should refresh every 30 seconds when actively investigating. Auto-refresh should be visible (countdown timer) so operators know when data was last fetched.
Rule 4: Staleness detection thresholds MUST be configurable per link because different interface types have drastically different normal message frequencies. A lab chemistry analyzer is 'stale' after 1 hour of silence. An insurance verification link is 'stale' only after 24 hours. Default thresholds should be set per interface category but overridable per individual link.
Rule 5: Any interface with errors > 0 in the last 24 hours must be visually flagged on the dashboard. The error count badge is clickable and navigates directly to INT-03 filtered to that link's errors. Zero errors shows as a green check, not the number zero.
Rule 6: The dashboard must cover ALL interface categories across VistA: lab instruments, imaging PACS, voice recognition, pharmacy automation, cross-site VA HL7, financial systems, insurance verification, ADT/registration, and any other connected systems. Grouping by category provides operational context (lab team cares about lab instruments, imaging team cares about PACS).
Rule 7: HL7 message content may contain Protected Health Information (PHI) in PID segments (SSN, name, DOB, address). The message detail view MUST mask full SSN (show last 4 only) consistent with the Social Security Reduction Act compliance (IB*2*746 applied this across billing; the same principle applies to interface message viewing). Patient name and DOB may be displayed for diagnostic purposes.
Rule 8: The HL7 V1.6 engine and HLO engine coexist on the same VistA system. Links may be configured for either engine. The dashboard must monitor BOTH engines' links and message flows. HLO links use port field #400.08 instead of the standard port field.
Rule 9: Files #772 (HL7 MESSAGE TEXT) and #773 (HL7 MESSAGE ADMINISTRATION) grow continuously with every message and require regular purging per HL7 SM Section 3.8. The configuration page (INT-07) should display: current entry count, oldest entry date, purge configuration (retention days), last purge date, and a warning if purging is overdue or files are growing excessively.
Rule 10: Queue depth monitoring is a critical early-warning indicator. A growing Out queue for a link indicates either a stuck outgoing filer or an unresponsive destination system. A growing In queue indicates a stuck incoming filer or application processing bottleneck. Both conditions require immediate investigation. Threshold: > 100 pending messages triggers amber alert; > 1000 triggers red alert.
Rule 11: The ^HLMA('AC','O',#870-IEN,#773-IEN) cross-reference is the most efficient way to check outbound queue status per link. The dashboard wrapper MUST use this cross-reference rather than scanning the entire #773 file. This is an O(1) lookup per link versus O(n) scan.
Rule 12: The Interface Topology Map (INT-08) is a NEW capability not available in the terminal system. Building the topology requires traversing: #870 (which links exist and where they connect), #771 (which applications are defined), and #101 (which event driver protocols generate which messages, and which subscriber protocols route them to which links). This graph shows the full picture of VistA's interface connectivity.
Rule 13: Lab instruments have the shortest staleness thresholds and should have the most prominent dashboard placement because instrument downtime most directly and immediately affects patient care (lab results not available for clinical decisions).
Rule 14: Filer status (incoming and outgoing filers running or stopped) is a SYSTEM-WIDE concern that affects ALL links, not just one. If a filer stops, the dashboard must show a system-level alert above all interface cards. This is the most severe operational state because it means NO messages are being processed or delivered anywhere in the system.
