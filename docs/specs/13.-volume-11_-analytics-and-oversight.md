# 1. Foundation: The Cross-Package Overlay Layer
Analytics is the only VistA Evolved workspace that does not map to a single VistA package. Instead, it aggregates data from ALL ten operational workspaces into executive, managerial, and operational dashboards. Its purpose is to answer: How is this facility performing? Where are the bottlenecks? Which departments need attention? What trends are emerging?
This creates a unique technical challenge: analytics data comes from packages with very different RPC coverage. Scheduling has the richest existing RPC surface (150+ SDES RPCs). Clinical has 55+ CPRS RPCs. Lab and Radiology have viewing RPCs. But Pharmacy fulfillment, Billing, Supply, and Interfaces have ZERO RPCs — so analytics for those areas depends on the custom wrappers defined in Volumes 5, 8, 9, and 10 being built first.
WAVE PLANNING: Basic analytics (scheduling metrics, clinical activity summary) can be built in Wave 2 because the RPCs already exist. Advanced analytics (pharmacy TAT, billing revenue, supply spending, interface health) cannot be built until the custom wrappers from Volumes 5-10 are deployed in Waves 3-4. Analytics pages should be designed to gracefully degrade — show whatever data is available from existing RPCs, and add panels as custom wrappers come online.

## 1.2 Real-Time vs Periodic Analytics
Not all analytics should be real-time. The refresh strategy must balance data freshness against VistA system load:
Refresh Tier
Interval
Which KPIs
Rationale
Tier 1: Real-Time
Every 5 minutes
Patients checked in today, appointment status counts, pending pharmacy orders, pending lab verifications, interface up/down, inpatient census
These KPIs drive immediate operational decisions (staffing, workflow prioritization).
Tier 2: Near-Real-Time
Every 15-30 minutes
Unsigned notes/orders per provider, no-show rate today, STAT order counts, critical lab values pending verification
Actionable within a shift but not minute-by-minute.
Tier 3: Hourly
Every 1-4 hours
Revenue metrics (billing volume, denial counts), pharmacy fill TAT, lab specimen TAT, imaging report TAT, interface error rates
Trend-oriented metrics that change gradually during the day.
Tier 4: Daily
Once per day (overnight batch)
Provider productivity summaries, departmental volume comparisons, revenue trend charts, supply spending reports, staff login summaries, quality metrics
Trend analysis that doesn't benefit from intra-day updates. Heavy aggregation queries best run off-peak.
Tier 5: Periodic
Weekly/Monthly
Executive summary, quality outcome measures, compliance tracking, long-term trending, cohort analysis
Strategic metrics. May benefit from extract-based data warehouse rather than real-time RPC queries.
DATA WAREHOUSE CONSIDERATION: For Tier 4 and Tier 5 analytics, a separate data extract pipeline or data warehouse may be more appropriate than real-time RPC queries. RPC calls against VistA are synchronous and add load to the production system. Heavy aggregation queries (e.g. 'all billing claims for the past 6 months') should be run against extracted data, not live VistA globals.

### Dashboard Design Principles
(a) Progressive loading: Each KPI card loads independently via its own wrapper call. Do NOT block the entire dashboard waiting for a slow query. Show cards as data arrives. Cards with unavailable wrappers show 'Data source not yet configured' gracefully.
(b) Auto-refresh at per-card intervals defined by the Refresh Tier. Tier 1 cards refresh every 5 minutes; Tier 3 cards refresh hourly. Visual indicator shows last refresh time per card.
(c) Exception-driven display: Normal KPIs display with green indicators and minimal visual weight. ABNORMAL KPIs display with amber/red indicators, larger font, and optional sound alert for red conditions. The eye should be drawn to problems, not normalcy.
(d) Drill-down: Every KPI card is clickable. Clicking navigates to either the department analytics page (ANL-02 through ANL-09) for deeper analysis or directly to the operational workspace (SCH-01, RX-01, LAB-05, etc.) for immediate action. The drill-down target should be configurable per user preference.
(e) Tenant-configurable: Smaller facilities may not operate all departments. A clinic without imaging should be able to hide the imaging KPI card. A facility without inpatient beds hides the census card. Configuration at the tenant profile level.
(f) Role-based defaults: Operations managers see all KPIs. Department managers see their department KPIs prominently, others minimized. Executives see ANL-10 (summary) by default with link to ANL-01 for detail.
(g) Trend context: Every KPI number should have a comparison point. 'Patients Seen: 142' means nothing without context. 'Patients Seen: 142 (vs 158 same day last week, down 10%)' enables decision-making. Trend arrows (up/down/steady) at minimum; sparkline charts for daily trend when space permits.

ALL ANALYTICS WRAPPERS ARE READ-ONLY. They never write data. They aggregate existing data from VistA files. For complex analytics (Tier 4/5), consider an extract-based data warehouse rather than real-time RPC aggregation queries against live VistA globals.

# 5. Rules for Design and Engineering
Rule 1: Analytics is an OVERLAY layer. It reads data from other workspaces but never writes to VistA. All analytics wrappers are READ-ONLY.
Rule 2: Every number on a dashboard must be traceable to its source. Clicking any KPI drills down to either the department analytics page or the operational workspace showing the underlying records.
Rule 3: Progressive data availability: the Operations Dashboard should show whatever data is available from deployed wrappers and gracefully degrade for wrappers not yet deployed. A card for unavailable data shows 'Not yet configured' rather than breaking the dashboard.
Rule 4: Scheduling analytics (ANL-02) and clinical activity (ANL-03) can be built in Wave 2 using existing SDES and CPRS RPCs. These are the first analytics pages to deliver.
Rule 5: Pharmacy, lab, imaging, and interface analytics (ANL-05/06/07) require Wave 3 wrappers. Revenue analytics (ANL-04) requires Wave 3+ billing wrappers. Supply analytics requires Wave 4 IFCAP wrappers. Quality metrics (ANL-08) are Wave 4+ due to complex cross-package derivation.
Rule 6: Refresh rates must be tiered. Real-time operational KPIs (appointments, pharmacy queue) refresh every 5 minutes. Trend analytics (revenue, TAT) refresh hourly or daily. Each KPI card should display its last refresh timestamp.
Rule 7: Threshold colors must be configurable per tenant. A 200-bed hospital and a 20-chair outpatient clinic have very different thresholds for 'normal' queue depths. Default thresholds provided; tenant admin can override.
Rule 8: Trend context is MANDATORY. Every KPI must show comparison to a prior period (same day last week, same month last year, configurable). A number without context is meaningless.
Rule 9: Role-based dashboard defaults. Operations managers see ANL-01 full view. Department managers default to their department page (ANL-05 for pharmacy director, ANL-06 for lab director). Executives default to ANL-10 summary.
Rule 10: Heavy aggregation queries (Tier 4/5: multi-month trends, cohort analysis, quality outcomes) should consider extract-based data warehouse rather than live RPC queries. Live queries add load to the VistA production system.
Rule 11: All analytics displays should support CSV/PDF export for offline reporting and compliance documentation.
Rule 12: Tenant-configurable card visibility. Hide KPIs for departments the facility doesn't operate (imaging, inpatient, supply). Configuration at tenant profile level, not per-user.
Rule 13: The Executive Summary (ANL-10) should be a single-screen view that a facility director can scan in 30 seconds. Maximum 8-10 KPI cards, largest font sizes, strongest color coding, one-click drill-down to any department.
Rule 14: Exception alerts from analytics should integrate with the notification system. If a KPI goes RED (e.g. pharmacy queue >50, interface down, denial rate >15%), an alert should be available to subscribed users — not just visible when they happen to look at the dashboard.
