# PART 5: Billing Adapter/Overlay Architecture
## 5.1 The Problem
VistA Integrated Billing (IB) is deeply wired to VA-specific concepts. The M routines (IBMT*, IBCN*, IBXS*, IBCE*) have VA eligibility logic embedded throughout — not in one configurable file but scattered across hundreds of routines. Key VA-specific assumptions baked into IB source code include:
(a) Patient eligibility is determined by Service-Connected percentage, Means Test category (A/B/C/G), and special categories (CHAMPVA, Medal of Honor, Purple Heart, AI/AN attestation, Former OTH, Urgent Care MISSION Act, Cleland-Dole Act, COMPACT Act).
(b) Copayment amounts are set by national VA copay rate schedules, not by local payer contracts.
(c) Claims are transmitted through the Financial Services Center (FSC) in Austin, TX, not directly to payers.
(d) Remittance advice is received from FSC, not from individual payers.
(e) The Automated Biller creates bills based on VA-specific event triggers (check-in, admission, means test change).
(f) Billing clocks track VA-specific copay accumulation rules.
## 5.2 Recommended Architecture: Adapter/Overlay
Principle: Keep VistA IB's data structures but add a billing overlay service that handles local/international rules.
VistA's BILL/CLAIMS #399, INSURANCE COMPANY #36, GROUP INSURANCE PLAN #355.3, and patient INSURANCE TYPE sub-file are GOOD data structures for storing billing data regardless of country. The file structures themselves are not VA-specific — they store bills, charges, diagnoses, procedures, insurance companies, and policies. What IS VA-specific is the LOGIC that populates and processes these files.

## 5.3 Source Code Modification Assessment
Modifying VistA M source code for billing internationalization is POSSIBLE (VistA is public domain) but RISKY and EXPENSIVE:
FEASIBLE modifications (lower risk):
(a) Add configurable country/payer tables that the existing IB routines can check before applying VA-specific logic.
(b) Add hooks in IB event drivers (check-in billing, admission billing) that allow the overlay to intercept and apply local rules.
(c) Modify the Automated Biller to support configurable rule sets per tenant profile.
RISKY modifications (higher risk):
(a) Replacing VA eligibility logic throughout IBMT* routines — eligibility checks are scattered across dozens of routines.
(b) Modifying EDI generation routines (IBXS*, IBXSC*) for non-837 claim formats — deeply wired to ANSI X12 837P/837I structure.
(c) Modifying copay calculation routines — VA copay rates are referenced from multiple entry points.
RECOMMENDATION: Start with the overlay adapter approach. Use VistA IB for data storage and encounter-based charge capture. Build the eligibility, pricing, claim generation, and transmission logic in the overlay service. Only modify M source code for specific hook points that the overlay needs to integrate with. Document every M source modification with exact routine name, line number, and purpose for future VA patch merge reviews.
