# 1. Foundation: The Surgical Workflow System
Surgery V3.0 is a complete perioperative management system that tracks a surgical case from initial request through waiting list, scheduling, intraoperative documentation, anesthesia management, post-operative care, operative report dictation, CPT/ICD coding, and risk assessment reporting. It is used by surgeons, anesthesiologists, surgical nurses, coding specialists, and OR managers.
The Surgery package is distinct from other clinical modules because it manages a MULTI-PHASE process with strict temporal sequencing:
Source: VistApedia Surgery Workflow: '1) Before the surgery: Waiting list, Request operation and Scheduling. 2) During the Surgery: Operation, Anesthesia. 3) After the Surgery: Reports — Operation Report, Nurse Intraoperative Report, Anesthesia Report.' Surgery TM: 'Each entry in the SURGERY file (#130) contains information regarding a surgery case, consisting of an operative procedure, or multiple operative procedures, for a patient.'

Surgery Reports menus

## 1.4 Nightly Background Process
Surgery has its own nightly background process (SRTASK-NIGHT) that performs automated maintenance. Surgery TM: 'This option initiates a number of background tasks to clean up the surgery database and update all appropriate information. Among the tasks performed are: calculation of average procedure times, cleanup of outstanding requests, update expected start and end times for operating rooms, locking cases, transmit risk assessment and workload information, and send notification messages.'
The SRTASK-NIGHT process is essential for OR scheduling accuracy — it recalculates average procedure times which feed estimated case duration for scheduling. It also locks completed cases to prevent unauthorized modification.

## 3.1 SRG-01: Surgery Schedule / OR Board
### Purpose
The operating room schedule board — the OR manager's primary planning and operational visibility screen. Shows all operating rooms with their scheduled cases, available time slots, and service blockouts for the day/week. ## 3.3 SRG-11: CPT/ICD Coding
### Purpose
Coding personnel review the operative report and enter final procedure (CPT) and diagnosis (ICD-10-CM) codes for the surgical case. The surgeon supplies initial codes during the case; the coder verifies and may update based on the dictated operative report. Final codes are sent to PCE for billing charge capture. Surgeon must verify the case before coding is finalized.
Source: Surgery UM: 'CPT/ICD Update/Verify Menu locked with SR CODER key. Coders have ready access to the Operation Report. Procedure and diagnosis codes are supplied as defaults to the Coder when entering the final codes that will be sent to PCE.' 'Surgeons use this option to verify that the stated procedure(s), diagnosis, and occurrences are correct. Surgeon can update the Operation Name, Planned CPT Code, Diagnosis, and Intraoperative Occurrences before verifying.'

The intraoperative documentation wrapper (ZVE SRG INTRAOP) is the most complex because it must handle multiple procedures per case, multiple team members, time tracking, wound classification, specimens, implants, counts, and occurrences all within a single case record (#130). # 5. Rules for Design and Engineering
Rule 1: SURGERY #130 is the central case file. One record per surgical case, with 200+ fields spanning all perioperative phases. Changes to #130 affect the Nurse Intraoperative Report, Operation Report, Anesthesia Report, CPT/ICD coding, risk assessment, and all surgery reports. EXTREME CAUTION on writes.
Rule 2: Operative reports, nurse intraoperative reports, and anesthesia reports are stored as TIU DOCUMENTS and must be viewable in CPRS. Surgery UM: 'Make Reports Viewable in CPRS option.' The modern UI must preserve this TIU integration so reports appear in the patient's chart.
Rule 3: A single surgical case can have MULTIPLE operative procedures, each with its own CPT code. The data model must support multiple procedures per case, not just one. The primary procedure is distinguished from secondary/concurrent procedures.
Rule 4: Anesthesia can have multiple techniques with different agents in the same case. Surgery UM: 'same anesthesia technique but different anesthesia agents. By entering the element number to repeat, avoid re-entering top-level data.' The data model must support this multi-level anesthesia structure.
Rule 5: CPT/ICD coding follows a two-step verification: Coder enters codes from the operative report, then Surgeon verifies. Surgery UM: 'Surgeon can update Operation Name, Planned CPT Code, Diagnosis, and Intraoperative Occurrences before verifying the case.' Codes are NOT sent to PCE until both steps are complete.
Rule 6: Wound classification uses the CDC standard: 1. Clean, 2. Clean-Contaminated, 3. Contaminated, 4. Dirty/Infected. Surgery UM confirmed the update from 'Infected' to 'Dirty/Infected.' This classification drives surgical site infection tracking and quality metrics.
Rule 7: The SRTASK-NIGHT background process is essential for OR scheduling accuracy. It recalculates average procedure times (used for case duration estimation), cleans up outstanding requests, updates expected OR start/end times, and locks completed cases. The modern UI must not interfere with this process.
Rule 8: Operating rooms must be defined in BOTH HOSPITAL LOCATION #44 AND OPERATING ROOM #131.7 before they can be used. Non-OR procedure locations (e.g. endoscopy suite, cardiac cath lab) need only be in #44. This dual-file setup must be maintained.
Rule 9: Anesthesia agents are identified by flagging drugs in DRUG #50 (shared pharmacy file) via the 'Flag Drugs for Use as Anesthesia Agents' option. Only flagged drugs appear as selectable anesthesia agents. This connection to the pharmacy drug file (Volume 5E) must be maintained.
Rule 10: Surgery HL7 interface (#133.2) controls data exchange with external surgical systems (anesthesia information systems, surgical scheduling systems). OBR/OBX text identifiers control field-level data flow. Surgery TM: 'Set flag to R for receive, N for not receive, RR for second layer of OBX identifiers.' The modern UI should provide HL7 interface configuration visibility.
Rule 11: Perioperative occurrences (complications) are tracked separately from the main case documentation. Returns to OR within 30 days are a specific quality metric. The occurrence tracking system must support intraoperative, postoperative, and non-operative occurrence types.
Rule 12: Surgery Risk Assessment (VASQIP) data is transmitted to national databases for quality benchmarking. SRA UM: 'Surgery Risk Assessment, Case Workload Totals, and Case Work Incomplete Assessments data downloaded to text files.' Monthly reports generated automatically on the 27th. The modern UI should provide SRA status monitoring.
Rule 13: PATIENT SAFETY — UNIVERSAL PROTOCOL: The intraoperative documentation screen must support surgical time-out verification: correct patient, correct procedure, correct site/side. Laterality (Left/Right/Bilateral) must be prominently displayed and confirmed. Wrong-site surgery is a 'never event' with severe regulatory and patient safety consequences.
Rule 14: INTERNATIONAL ADAPTATION: The Surgery package workflow (waiting list -> schedule -> pre-op -> intraop -> anesthesia -> post-op -> reporting -> coding) is universal. What changes internationally: CPT procedure codes (US-specific, AMA copyright) replaced by local procedure codes (OPCS-4 UK, MBS Australia, OPS Germany), ICD-10-CM replaced by local ICD-10 variant, ASA classification is internationally recognized (no change needed), wound classification CDC standard is internationally recognized, VASQIP risk assessment is VA-specific (local quality programs substitute).
