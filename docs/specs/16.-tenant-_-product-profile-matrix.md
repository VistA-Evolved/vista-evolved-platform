# VistA Evolved
Tenant / Product Profile Matrix
8 Deployment Profiles with Workspace Visibility Mapping
March 2026 | This document defines the 8 tenant profiles the product must support: Full Hospital Network, Multi-Specialty Ambulatory Group, Single-Specialty Clinic, Standalone Outpatient Pharmacy, Hospital Inpatient Pharmacy/BCMA, Standalone Laboratory, Standalone Radiology/Imaging, Cross-Tenant Patient Portal/Identity Overlay. Each profile maps which of the 11+ workspaces are PRIMARY (default launcher, always visible), OPTIONAL (available if enabled, may be hidden by default), or HIDDEN (not relevant to this tenant type). Grounded in: real VistA deployment precedents (VA medical centers, CBOCs, WorldVistA/OpenVistA international deployments, Midland Memorial Hospital, Kingdom of Jordan, American Samoa LBJ Tropical Medical Center, UC Davis, Texas Tech). VistA supports a large variety of clinical settings from small clinics providing solely outpatient care to large medical centers with significant inpatient populations.

# 1. Why Tenant Profiles Matter
The product cannot assume every customer is a full hospital. VistA itself supports a large variety of clinical settings — the VA Monograph states that VistA is deployed across more than 1,500 sites of care from small CBOCs (Community Based Outpatient Clinics) to large medical centers. WorldVistA and OpenVistA have extended VistA to private hospitals, clinics, and international healthcare systems of varying sizes.
A standalone outpatient pharmacy does not need an OR schedule. A standalone laboratory does not need a pharmacy formulary. A single-specialty dermatology clinic does not need inpatient medications or BCMA. If the product presents all 11+ workspaces to every tenant regardless of type, the result is:
(a) Overwhelming complexity for small tenants who only need 3-5 workspaces
(b) Licensing/pricing mismatch if the product charges per workspace
(c) Configuration burden — every workspace requires setup even if unused
(d) Training overhead — staff must learn workspaces they never use
(e) Security risk — unused workspaces with default configurations may have unreviewed access
The profile matrix solves this by defining WHICH workspaces are visible, which are the HOME screen, and which roles are needed per tenant type.

## 2.1 Profile A: Full Hospital / Multi-Hospital Network
### Description
A hospital or health-system tenant using the broadest set of VistA capabilities. Includes inpatient care (wards, ICU, OR, PARR), outpatient clinics, emergency department, all ancillary departments (pharmacy, lab, radiology, surgery), and all administrative/financial functions. This is the most complex profile and should NOT be assumed as the default for every tenant.
Real-world examples: VA Medical Centers, Midland Memorial Hospital (OpenVistA), Kingdom of Jordan national system, university medical centers (UC Davis, Texas Tech).

# 5. Design Implications for the Modern UI
## 5.1 App Launcher Shell
The app launcher must be PROFILE-AWARE. When a user logs in, the launcher shows only the workspace tiles that are PRIMARY or OPTIONAL-and-enabled for their tenant profile. Hidden workspaces are not shown at all — not even as disabled/grayed tiles.
Design requirements:
(a) The launcher adapts its tile grid based on the number of visible workspaces. A standalone pharmacy (4-6 tiles) looks clean and focused, not like a hospital launcher with 6 tiles grayed out.
(b) Each tile shows the workspace icon, name, and a brief status indicator (e.g. Pharmacy shows '12 prescriptions pending', Lab shows '8 specimens in queue', Surgery shows '5 cases scheduled today').
(c) The Odoo-style inspiration applies HERE: Odoo's app launcher shows only installed apps. VistA Evolved's launcher shows only activated workspaces per tenant profile.
## 5.2 Home Screen Per Role
Different roles within the same tenant may have different home screens. When a pharmacist logs in, they should land directly on their pharmacy work queue, not on the clinical chart view. When a scheduler logs in, they should land on the appointment schedule. When a surgeon logs in, they should see the OR board.

## 5.3 Navigation & Cross-Workspace Links
Even when a workspace is OPTIONAL or HIDDEN at the launcher level, cross-workspace links within the clinical workflow must still function. For example:
(a) A clinician in Profile B (Multi-Specialty Clinic) orders a lab test. The Lab workspace may be OPTIONAL. If it's not enabled, the order still goes to VistA's Lab package — results will appear in the clinical workspace when available from an external lab.
(b) A Profile D (Standalone Pharmacy) pharmacist views a prescription. The patient's allergy information comes from PATIENT ALLERGIES #120.8 — this data exists even without the full Clinical Workspace being visible.
(c) A Profile E (Inpatient Pharmacy/BCMA) pharmacist verifies an order. The order was placed in CPRS (Clinical Workspace) — the pharmacist sees the order in their INP-01 work queue without needing to navigate to the Clinical Workspace.
Rule: VistA's data integration operates BELOW the workspace visibility layer. Workspace visibility controls the UI tiles; it does NOT control VistA's internal package cross-references, order flows, or data dependencies. Hiding a workspace does NOT disable the VistA package behind it — it only hides the UI entry point.

# 6. Rules for Product and Engineering
Rule 1: The product must NOT assume every tenant is Profile A (Full Hospital). The app launcher, navigation, role templates, and provisioning must all be profile-aware. A standalone pharmacy should feel like a pharmacy system, not a hospital system with most tabs disabled.
Rule 2: Workspace visibility (Primary/Optional/Hidden) is a UI-layer concept. It controls what the user SEES on the launcher. It does NOT disable VistA packages underneath — VistA's internal integrations continue to function regardless of UI visibility. Hiding the Lab workspace does NOT stop lab orders from being processed by VistA's Lab package.
Rule 3: Profile selection happens at tenant provisioning time and determines the initial provisioning sequence (which VistA packages to initialize, which roles to create, which parameters to set). A tenant admin can later enable OPTIONAL workspaces without re-provisioning — this just changes launcher visibility and may require additional configuration.
Rule 4: Each profile has a recommended HOME SCREEN per user role. The home screen is where the user lands after login. It should be the most frequently used workspace for that role in that profile type. Users can override their home screen in preferences.
Rule 5: Profile H (Cross-Tenant Portal) is architecturally SEPARATE from Profiles A-G. It is an enterprise overlay, not a VistA tenant. It aggregates data from multiple underlying VistA instances. It should NOT be built until Profiles A-G are stable.
Rule 6: Provisioning templates per profile reduce setup time significantly. A standalone lab (Profile F) provisioning template skips all pharmacy, surgery, and BCMA initialization — saving weeks of configuration effort. The provisioning system should offer profile-based setup wizards.
Rule 7: The design team should create profile-specific launcher mockups, not one generic launcher. Profile A shows 16 tiles. Profile D shows 6 tiles. The visual density and organization must be appropriate for each.
Rule 8: Billing configuration is profile-dependent. Profile A uses all billing modules. Profile D uses only pharmacy claims (ECME/PBM). Profile F uses only lab test billing. Profile G uses only imaging procedure billing. The billing workspace itself adapts to show only relevant billing functions per profile.
Rule 9: Device/interface requirements vary dramatically by profile. Profile A needs all device types. Profile D needs ePrescribing and POS devices. Profile F needs lab instrument interfaces. Profile G needs DICOM/PACS. The device setup wizard should be profile-filtered.
Rule 10: User role templates per profile prevent over-provisioning of access. A standalone pharmacy tenant should NOT have pre-created Surgeon or Radiologist roles — these are irrelevant and create unnecessary security surface. Only profile-appropriate roles should be pre-created.
Rule 11: INTERNATIONAL CONSIDERATION: All 8 profiles apply internationally. A standalone pharmacy in London operates similarly to one in New York — the workspace layout is the same, only the billing codes, drug databases, and regulatory requirements change. The Tenant Profile Matrix is UNIVERSAL; the internationalization overlays (billing adapter, terminology mapping, date format) are ORTHOGONAL to the profile selection.
