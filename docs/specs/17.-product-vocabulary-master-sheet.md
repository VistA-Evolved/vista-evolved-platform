# VistA Evolved
Product Vocabulary Master Sheet
VistA Back-End Term to Modern Front-End Term + FHIR R4 + International Standards
March 2026 | 134 precisely mapped terms across 10 domains. Every mapping is EXACT and VERIFIED against VistA data dictionaries (FileMan July 2025), FHIR R4 specification (hl7.org/fhir/R4), and international healthcare terminology standards (SNOMED CT, LOINC, ICD-10, ATC, UCUM, RxNorm). NO GUESSING — each mapping states the VistA file/field number, the modern product term, the FHIR R4 resource/element, and the applicable international standard. This document is the AUTHORITATIVE reference for how the modern UI presents VistA data to users. Design teams use Column C (Modern Product Term). Engineering teams use Column B (VistA File/Field) and Column D (FHIR Resource). Internationalization teams use Column E (International Standard).

# 1. How to Use This Document
This vocabulary sheet has 5 columns per term: (A) VistA Internal Term — the name used in VistA's data dictionary, terminal menus, and M source code. (B) VistA File/Field — the exact FileMan file number and field number where this data lives. (C) Modern Product Term — what the modern web UI should display to users. This is the PRIMARY output for the design team. (D) FHIR R4 Resource — the HL7 FHIR R4 resource and element that maps to this concept, for interoperability. (E) International Standard — the applicable terminology standard (SNOMED CT, LOINC, ICD-10, ATC, UCUM, RxNorm, ISO) that governs this concept internationally.
Rules for the design team:
(a) NEVER show Column A (VistA Internal Term) to end users. VistA jargon is VA-specific and confusing to non-VA users. Always use Column C (Modern Product Term) in the UI.
(b) Column A terms MAY appear in admin/config screens with Column C as the label and Column A as a tooltip or technical reference — but only for system administrators, never for clinical users.
(c) When a user searches or filters, the system should accept BOTH Column A and Column C terms and match them. A user typing 'Facility' should find what VistA calls 'Institution'. A user typing 'Institution' should also work.
Rules for engineering:
(a) Column B (VistA File/Field) is your data source. Always reference the exact file and field number when building wrappers or reading VistA data.
(b) Column D (FHIR R4 Resource) is your interoperability target. When building FHIR endpoints, map VistA data to these resources and elements.
(c) Column E (International Standard) tells you which code system to use for coded data. Use these standards for data exchange, reporting, and international deployment.

# 12. Rules for Vocabulary Usage
Rule 1: NEVER display VistA internal terms (Column A) to clinical end users. Always use Modern Product Terms (Column C). VistA jargon like 'DFN', 'DUZ', 'NAOU', 'Green Sheet', 'Stop Code', 'Means Test' is meaningless to non-VA users.
Rule 2: System administrators MAY see VistA internal terms as secondary labels or tooltips when configuring VistA-backed features. E.g. a field labeled 'Facility' may show '(VistA: INSTITUTION #4)' in admin mode.
Rule 3: Search must be BILINGUAL — accept both VistA terms and modern product terms. A search for 'Institution' and a search for 'Facility' must return the same results.
Rule 4: FHIR R4 resources (Column D) are the interoperability target. When building APIs, data exports, or integrations with external systems, use FHIR resource structures and standard terminologies.
Rule 5: International standards (Column E) determine which CODE SYSTEMS to use for coded data. Use LOINC for lab tests, SNOMED CT for clinical terms, ICD-10 for diagnoses, ATC for drug classification, UCUM for measurement units, CVX for vaccines, CPT/local equivalents for procedures.
Rule 6: VA-SPECIFIC terms (marked in Column E) must have international equivalents or be made configurable. 'Means Test' becomes 'Financial Screening'. 'Stop Code' becomes 'Visit Type Code' or is removed. 'VASQIP' becomes 'Surgical Quality Program'. 'DMC' becomes 'Collections'. 'FMS' becomes 'Financial System'.
Rule 7: FileMan date format (YYYMMDD.HHMMSS) is STORAGE ONLY. Never display to users. Always convert to ISO 8601 internally and to locale-appropriate format for display (MM/DD/YYYY for US, DD/MM/YYYY for EU/UK, YYYY-MM-DD for ISO/East Asia).
Rule 8: Internal identifiers (DFN, DUZ, IEN) are NEVER exposed to users. The modern UI generates display-friendly identifiers: Patient MRN (from the MRN field or a generated format), Employee ID (from an HR-assigned number), etc.
Rule 9: Drug terminology must be configurable per deployment country: RxNorm for US, dm+d for UK, AMT for Australia, ATC for WHO classification. The DRUG #50 local file structure supports any drug naming convention — only the national reference database (NDF) changes.
Rule 10: This vocabulary sheet is a LIVING DOCUMENT. As new volumes, packages, and features are added, new terms must be mapped here. Any term that appears in the VistA data dictionary but is NOT in this sheet should be added during the specification process for that volume.
