# VistA Evolved
Internationalization Framework
Date/Time + Currency + Units + Language/Locale + Timezone + Regulatory Compliance
March 2026 | Architecture for deploying VistA Evolved in any country. Grounded in: VistApedia FileMan date format documentation (YYYMMDD.HHMMSS), Kingdom of Jordan VistA deployment precedent (Arabic locale, dd/mm/yyyy via Kernel DEFAULT LANGUAGE), ISO 8601 (date/time), IETF BCP 47 (locale tags), Unicode CLDR (locale data), UCUM (measurement units), ISO 4217 (currency codes), E.164 (phone numbers), ISO 3166 (country codes). Healthcare regulatory standards: HIPAA (US), GDPR (EU), Privacy Act 1988 (AU), PIPEDA (CA), POPIA (ZA), PDPA (SG/TH). This framework defines WHAT needs to be internationalized, HOW it is implemented in the modern web client, and WHERE VistA's M-side infrastructure already supports localization versus where the web presentation layer handles it.

# 1. Architecture: Two-Layer Internationalization
VistA Evolved's internationalization operates across TWO distinct layers. Understanding which layer handles what is essential because it determines whether VistA M source code needs modification (expensive, risky) or whether the modern web client's presentation layer can handle the adaptation (cheaper, safer).

## 2.2 Date Display Adapter Architecture
Implementation in the modern web client:
(a) RECEIVE: VistA RPC/wrapper returns date in FileMan format (YYYMMDD.HHMMSS) as a numeric string.
(b) PARSE: Web client date utility function parses FileMan format: extract YYY (add 1700 for actual year), MM, DD, HH, MM, SS. Handle imprecise dates (missing day = month-level, missing month = year-level).
(c) CONVERT: Create JavaScript Date object or ISO 8601 string. Apply timezone offset from user's profile or browser locale.
(d) FORMAT: Use JavaScript Intl.DateTimeFormat with user's BCP 47 locale tag to produce locale-appropriate display string. Examples: new Intl.DateTimeFormat('en-US').format(date) produces '3/29/2026'. new Intl.DateTimeFormat('en-GB').format(date) produces '29/03/2026'. new Intl.DateTimeFormat('de-DE').format(date) produces '29.03.2026'. new Intl.DateTimeFormat('ja-JP').format(date) produces '2026/3/29'.
(e) INPUT: When user enters a date, parse per their locale format, convert to FileMan format (YYYMMDD.HHMMSS), send to VistA. Date pickers should use locale-appropriate format and calendar widget.
ZERO VistA M SOURCE MODIFICATION NEEDED for date display. The FileMan storage format is already locale-independent. All formatting happens in the web client presentation layer using standard JavaScript internationalization APIs.

Most clinical measurements in VistA are ALREADY in universal units (metric/SI). The main exceptions are height (inches vs cm), weight (pounds vs kg), temperature (F vs C), and some lab values (mg/dL vs mmol/L for glucose, cholesterol, creatinine). The web client must offer a UNIT PREFERENCE setting per tenant (US Customary or Metric) and convert for display. VistA stores vitals in whatever unit is entered — the conversion adapter handles display.
PATIENT SAFETY: Unit conversion errors can be fatal. The Mars Climate Orbiter was lost due to a metric/imperial conversion error. In healthcare: a weight entered in pounds but interpreted as kilograms would result in doses approximately 2.2x too high. The unit conversion adapter must be rigorously tested and must always display the unit alongside the value.

# 10. Rules for Internationalization
Rule 1: FileMan date format (YYYMMDD.HHMMSS) is STORAGE ONLY. Never display to users. Never require users to enter dates in FileMan format. Always convert to/from the user's locale format in the presentation layer.
Rule 2: Store internally in locale-independent formats: FileMan dates, ISO 4217 currency codes, UCUM unit codes, E.164 phone numbers, ISO 3166 country codes, BCP 47 locale tags. Format for display at the presentation layer.
Rule 3: The presentation layer (web client) handles ALL display formatting. VistA M source code does NOT need modification for: date display, currency display, number formatting, measurement unit conversion, UI translations, phone/address formatting, or text direction.
Rule 4: VistA M source code MAY need modification (or overlay adapter) for: billing business logic (eligibility rules, charge calculation, claim generation), controlled substance schedule definitions, prescription validity rules, and regulatory-specific business rules. These are handled by the Billing Adapter Architecture document and configurable rule engines.
Rule 5: Clinical data (patient names, progress notes, allergy descriptions, clinical documentation) is NEVER translated. It stays in the language it was entered. UI labels and system-generated text ARE translated.
Rule 6: ALWAYS display the unit of measure alongside numeric values. Never show '72' without specifying 'kg' or 'lb'. Never show '37.0' without specifying 'C' or 'F'. Unit display prevents potentially fatal misinterpretation.
Rule 7: The tenant locale configuration (Section 8) must be set at provisioning time. Changing locale settings on a running system is possible but requires careful review of existing data display (e.g. switching from US Customary to Metric does not change stored vital sign values, only display).
Rule 8: RTL (right-to-left) support requires CSS logical properties throughout: use margin-inline-start instead of margin-left, padding-inline-end instead of padding-right, etc. All CSS must be audited for RTL compatibility. Jordan deployment proves VistA works with Arabic RTL.
Rule 9: Time zones must NEVER be hardcoded. Use IANA timezone database identifiers (e.g. America/New_York, not 'EST' or 'UTC-5'). The IANA database handles DST transitions and historical timezone changes automatically.
Rule 10: Drug names may differ by country even for the same molecule: Acetaminophen (US/Canada/Japan) = Paracetamol (UK/EU/AU/most of world). The product must support configurable drug naming per deployment country while maintaining the underlying VA GENERIC #50.6 or RxNorm/INN mapping for interoperability.
Rule 11: Lab reference ranges must be configurable per facility. Different countries, different analyzer manufacturers, and different populations have different normal ranges for the same tests. The LABORATORY TEST #60 reference range fields must support per-facility configuration, not just national defaults.
Rule 12: The i18n framework is ORTHOGONAL to the Tenant Profile Matrix. A standalone pharmacy in London (Profile D, en-GB locale) and a standalone pharmacy in New York (Profile D, en-US locale) have the same workspace layout but different date formats, currency, and regulatory requirements. Profile determines WHAT workspaces are visible; locale determines HOW data is displayed.
