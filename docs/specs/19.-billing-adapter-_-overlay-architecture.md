# VistA Evolved
Billing Adapter / Overlay Architecture
International Billing Framework for Non-VA Healthcare Organizations
March 2026 | Defines the architecture for replacing VA-specific billing logic with configurable international billing support while preserving VistA IB's data structures. Grounded in: IB V2.0 Technical Manual (Sept 2025, IB*2*668), IB User Manual (Sept 2023), EDI IB User Guide (April 2025), AR V4.5 TM (Dec 2025), ePayments UM (Dec 2025), VHA Directive 1599(1), Financial Policy Vol XII Ch 4. Key VistA IB files: BILL/CLAIMS #399, INSURANCE COMPANY #36, GROUP INSURANCE PLAN #355.3, IB SITE PARAMETERS, IB ACTION #350, RATE TYPE #399.3, BILLING RATES. IB namespaces: IB (#350-389), DGCR (#399-399.5), DIC (#36). Automated Biller event drivers: scheduling check-in, MAS movement, nightly background job.

# 1. The Problem: VA Billing Logic in VistA IB
VistA Integrated Billing (IB) V2.0 is a comprehensive billing system, but it is DEEPLY wired to VA-specific concepts. The following VA-specific assumptions are embedded throughout the IB M routines (IBMT*, IBCN*, IBXS*, IBCE*, DGCR*), not in configurable parameters:

BOTTOM LINE: VistA IB's DATA STRUCTURES (#399 BILL/CLAIMS, #36 INSURANCE COMPANY, #355.3 GROUP INSURANCE PLAN) are GOOD and mostly universal — they store bills, charges, payers, and policies. What is VA-specific is the LOGIC that determines eligibility, calculates charges, formats claims, and transmits them. The overlay adapter replaces the LOGIC while keeping the DATA STRUCTURES.

# 2. The Solution: 7-Layer Billing Overlay Adapter
The billing overlay adapter sits BETWEEN VistA IB's data structures and the external billing world. It intercepts billing events, applies LOCAL rules instead of VA rules, and generates LOCAL claim formats instead of HIPAA 837.
## 2.1 Architecture Diagram (Conceptual)
VistA Clinical Data (encounters, diagnoses, procedures, providers) -> Layer 1: Encounter Data Extraction -> Layer 2: Eligibility Engine -> Layer 3: Charge Calculation -> Layer 4: Claim Generation -> Layer 5: Claim Transmission -> Layer 6: Remittance Processing -> Layer 7: Revenue Reporting. Each layer is CONFIGURABLE per country/payer/tenant.

Claims from Layer 4. Payer routing configuration.

Transmission confirmation: accepted, rejected (with reason), pending. Claim tracking number from clearinghouse/payer.
## 6. Remittance Processing

ERA/835 data (US), or local remittance format. Payment amounts, adjustment reasons (CARCs/RARCs or local equivalents).
ERA/835 via FSC -> MailMan -> ePayments/EDI Lockbox in AR. Auto-posting logic. VA-specific adjustment handling.
CONFIGURABLE remittance engine: (US non-VA) ERA/835 from commercial clearinghouse. (UK) NHS payment reconciliation files. (AU) Medicare payment advice. (Generic) Configurable remittance format parser. Auto-matching logic: match payment to claim by claim number, patient, amounts. Post to AR (#430) payment transactions.
Payments posted to AR. Denials logged with reason codes. Underpayments flagged. Overpayments flagged for refund review.
## 7. Revenue Reporting
Generate financial reports per local accounting, tax, and regulatory requirements.
AR #430 (bills), #340 (debtors), payment transactions, charge data.
FMS (Financial Management System) integration with US Treasury. VA-specific revenue reporting (MCCR reports, DSS extracts).
CONFIGURABLE reporting engine: (US non-VA) Standard financial reports (aged receivables, collection rates, payer mix, revenue by service). (UK) NHS financial reporting. (AU) Medicare billing reconciliation reports. (Generic) Configurable report templates. Export to local accounting system (QuickBooks, Xero, SAP, local ERP).
Financial reports: revenue by period, payer mix, aged receivables, collection rate, denial rate, cost-to-collect.

# 8. Rules for Billing Overlay Architecture
Rule 1: VistA IB's DATA STRUCTURES (#399, #36, #355.3) are KEPT. The overlay replaces the LOGIC (eligibility determination, charge calculation, claim formatting, transmission) — NOT the data storage. Bills are still stored in #399 (or a parallel table that AR can read). Payments are still posted to AR #430.
Rule 2: The overlay operates as a SERVICE alongside VistA, not as M source code modification. It connects to VistA via RPCs/wrappers to read clinical data and write billing data. It has its own database (PostgreSQL or equivalent) for configuration tables, fee schedules, and claim templates. It does NOT modify VistA M routines.
Rule 3: Event drivers (check-in, admission, prescription fill, procedure) are UNIVERSAL clinical events. The overlay captures these events and applies LOCAL billing rules. The event TRIGGER stays in VistA; the billing RESPONSE comes from the overlay.
Rule 4: For US non-VA deployment, the overlay uses the SAME claim format (HIPAA 837) as VA IB. The differences are: commercial clearinghouse instead of FSC, commercial eligibility instead of VA Means Test, charge master instead of VA copay rates. This is the EASIEST international profile because the formats match.
Rule 5: For international deployment (UK, AU, EU), the overlay replaces MORE of the IB pipeline because claim formats, payer systems, and pricing models are fundamentally different. The overlay's configurable templates handle this without changing VistA.
Rule 6: AR (#430) payment posting is UNIVERSAL. Regardless of how the bill was generated or the claim was transmitted, the payment posting logic in AR is the same: create payment transaction, update bill balance, reconcile with financial system. The overlay's remittance engine parses the LOCAL remittance format and posts via standard AR routines.
Rule 7: The billing rules engine must be CONFIGURABLE by non-programmers. Billing rules change frequently (payer contracts renegotiated, fee schedules updated annually, regulatory requirements change). The rules engine should use decision tables or a visual rule editor, not hardcoded M routines.
Rule 8: Fee schedules must be IMPORTABLE from external sources: CMS fee schedule downloads (US), NHS tariff publications (UK), MBS schedule downloads (AU), payer contract data. Manual fee schedule entry is not scalable.
Rule 9: Claim transmission must support MULTIPLE channels simultaneously. A single tenant may submit: HIPAA 837 to commercial insurers via clearinghouse, Medicare claims directly to CMS, Medicaid claims to state systems, and workers compensation claims to specific payers — all using different formats and channels.
Rule 10: The overlay AUDIT LOG is legally required. Every billing decision (why this charge, why this code, why this payer) must be logged and auditable. This supplements VistA IB's own IB ACTION #350 audit trail.
Rule 11: FHIR-based billing is emerging. VistA IB TM (2025) already references '837 FHIR Transaction Processing' as a parameter. The overlay should support FHIR Claim and ExplanationOfBenefit resources as a claim format option, positioning VistA Evolved ahead of industry adoption.
Rule 12: The overlay does NOT replace VistA IB for VA deployments. If the tenant IS a VA facility (or equivalent government healthcare using VA billing rules), VistA IB's native logic runs unchanged. The overlay only activates for non-VA tenants. This is determined by the tenant's Billing Profile in the Tenant Profile Matrix.
