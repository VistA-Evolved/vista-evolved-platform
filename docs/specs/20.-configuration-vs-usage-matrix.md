# VistA Evolved
Configuration vs Usage Matrix
Parameter Files + Change Frequency + Scope + Risk Level + UI Tier
March 2026 | Maps every VistA configuration/parameter file to: (1) change frequency (setup-once, annual, monthly, daily, per-event), (2) who configures it (system admin, package coordinator, supervisor, user), (3) scope (system-wide, division, location, user), (4) risk level (CRITICAL/HIGH/MEDIUM/LOW), (5) modern UI tier (provisioning wizard, admin settings, supervisor settings, user preferences). Grounded in: CPRS Setup Guide (July 2025), Pharmacy Setup Checklist, Kernel 8.0 TM (July 2025), IB V2.0 TM (Sept 2025), Surgery V3.0 TM, Lab V5.2 TM, all package-specific TMs. Purpose: ensures the modern UI exposes configuration at the RIGHT level — not burying critical setup in user preferences, and not forcing daily-use settings into admin-only screens.

# 1. Why Configuration Classification Matters
VistA has hundreds of configuration parameters spread across dozens of parameter files. In the terminal environment, ALL configuration is done through the same data-entry interface — a system admin and a pharmacist use the same scroll-mode prompts. There is no visual distinction between a one-time site setup parameter and a daily operational preference.
In the modern UI, configuration must be TIERED:
Tier 1 (Provisioning Wizard): Set ONCE during tenant/site setup. Locked after go-live unless deliberately unlocked by system admin. E.g.: Institution #4 setup, Division #40.8, Operating Room definitions, NDF initial load.
Tier 2 (Admin Settings): Changed INFREQUENTLY by system admin or package coordinator (ADPAC). E.g.: Pharmacy system parameters (#59.7), billing rates, HL7 interface flags, scheduling templates.
Tier 3 (Supervisor Settings): Changed PERIODICALLY by department supervisors. E.g.: Ward-specific medication parameters, clinic availability patterns, report schedules, form letter templates.
Tier 4 (User Preferences): Changed FREELY by individual users. E.g.: Default printer, notification preferences, home screen, patient list sort order, display columns.
RISK: Exposing a Tier 1 parameter (e.g. Institution Station Number) at Tier 4 (user preferences) allows accidental modification that could corrupt the entire site. Exposing a Tier 4 parameter (e.g. default printer) only at Tier 1 (provisioning wizard) forces users to submit a helpdesk ticket for a trivial preference change. BOTH are bad. Classification prevents both.

# 10. Rules for Configuration Management
Rule 1: The modern UI must enforce TIER-BASED ACCESS to configuration. Tier 1 parameters (provisioning) are only accessible during initial setup or by system admins with explicit unlock. Tier 2 parameters require package coordinator or system admin role. Tier 3 requires supervisor role. Tier 4 is freely accessible to the authenticated user.
Rule 2: Every Tier 1 and Tier 2 parameter change must be AUDITED: who changed what, when, from what value, to what value. This is essential for regulatory compliance and troubleshooting. VistA's FileMan audit trail supports this at the field level.
Rule 3: CRITICAL-risk parameters should require CONFIRMATION before save: a warning dialog explaining the impact, requiring the user to type a confirmation phrase (not just click OK). E.g. changing Station Number, modifying AR categories, altering drug file entries.
Rule 4: Configuration parameters should show their CURRENT VALUE, DEFAULT VALUE, and LAST CHANGED date/user in the admin UI. This helps administrators understand what has been customized vs what is at default.
Rule 5: The provisioning wizard (Tier 1) should be a GUIDED WORKFLOW that walks through all required configuration in dependency order. E.g. create Institution -> create Division -> create Hospital Locations -> create Operating Rooms -> configure Pharmacy System -> load NDF -> create Drug File entries -> etc.
Rule 6: Configuration IMPORT/EXPORT must be supported for Tier 2+ parameters. This enables: (a) cloning configuration from a test environment to production, (b) copying configuration from one site to another in a multi-site deployment, (c) backing up configuration before changes.
Rule 7: The Tenant Profile Matrix (Phase 2, Artifact 1) determines WHICH configuration sections are relevant. A standalone pharmacy (Profile D) does NOT need Surgery configuration or Radiology configuration. The admin settings screen should only show configuration sections for activated workspaces.
Rule 8: Background job scheduling (TaskMan equivalents) should be visible in an admin dashboard: which jobs are scheduled, when they last ran, whether they succeeded or failed, next scheduled run. This replaces the terminal-based TaskMan monitoring with modern observability.
Rule 9: Drug file maintenance (DRUG #50, NDF matching, formulary updates) is the HIGHEST-VOLUME configuration activity in a running system. The modern UI must make drug file management efficient: bulk import from NDF updates, formulary committee decision tracking, automatic Master File Update to CPRS after changes.
Rule 10: Parameter DEPENDENCIES must be documented and enforced. E.g.: changing a ward's Treating Specialty (#42.4 mapping) affects bed section reporting, billing categorization, and staffing reports. The admin UI should warn about downstream effects before allowing the change.
Rule 11: INTERNATIONAL ADAPTATION: Many VA-specific parameters (Station Number format, copay rates, DMC referral criteria, FMS transmission settings) are not relevant internationally. The configuration UI should hide VA-specific parameters for non-VA tenants and show country-appropriate equivalents per the billing overlay adapter and i18n framework.
Rule 12: The configuration matrix in this document is a LIVING REFERENCE. As new volumes and packages are added to the specification, their configuration parameters must be added here with proper tier classification, change frequency, scope, and risk assessment.
