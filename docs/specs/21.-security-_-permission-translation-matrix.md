# VistA Evolved
Security / Permission Translation Matrix
VistA Security Keys -> Modern RBAC Roles -> SMART-on-FHIR Scopes
March 2026 | Maps every VistA security key (SECURITY KEY #19.1) to: (1) modern role-based access control (RBAC) role, (2) permission group, (3) SMART-on-FHIR scope for API access, (4) mutual exclusion rules, (5) minimum required electronic signature level. Grounded in: VistApedia Security Key catalog (complete key descriptions), CPRS Setup Guide (July 2025, key assignment rules including ORES/ORELSE mutual exclusion), all package Technical Manual/Security Guides. VistA uses a FLAT KEY model — each user has a set of independently assigned security keys. The modern product uses a HIERARCHICAL ROLE model — users are assigned roles, each role grants a set of permissions. This matrix bridges the two models.

FLAT: Each user in NEW PERSON #200 has zero or more security keys from SECURITY KEY #19.1 assigned independently. No hierarchy. No grouping. A user with 15 different responsibilities may have 40+ individual keys assigned.
HIERARCHICAL RBAC: Users are assigned one or more ROLES. Each role grants a set of PERMISSIONS. Permissions map to specific actions (read, write, delete) on specific resources (patient chart, pharmacy orders, lab results). Roles can inherit from parent roles.
Each VistA key maps to one or more PERMISSIONS. Permissions are grouped into ROLES. When provisioning users, assign ROLES instead of individual keys. The system translates roles into VistA key assignments for backward compatibility.

Mutual Exclusion
Some keys are mutually exclusive by convention, NOT enforced by VistA. CPRS Setup Guide: 'DO NOT give users both the ORES key and the ORELSE key.' Enforced by administrator discipline.
Mutual exclusion enforced by system. Assigning a role that conflicts with an existing role triggers a warning/block.
The modern UI enforces ORES/ORELSE, PROVIDER/non-PROVIDER, and other mutual exclusion rules that VistA relies on administrator discipline for.
Electronic Signature
Required for order signing. CPRS Setup: 'Users may have the ORES key but not have an electronic signature code, and will NOT be allowed to electronically sign orders.' Signature code stored in #200 field 20.4.

No formal VistA mechanism. Sensitive patient access controlled by DG SENSITIVITY key + patient sensitivity flag. Access logged but not explicitly 'break-the-glass.'
Formal break-the-glass: user requests access to restricted record, provides justification, access granted with enhanced audit logging, automatic supervisor notification.
Enhance VistA's sensitivity mechanism with modern BTG workflow.

# 11. Rules for Security Implementation
Rule 1: ORES and ORELSE are MUTUALLY EXCLUSIVE. The system must enforce this at role assignment time. A user cannot have both 'Physician' role (ORES) and 'Nurse' role (ORELSE) simultaneously. CPRS Setup Guide: 'DO NOT give users both the ORES key and the ORELSE key.' The modern UI must make this impossible, not just inadvisable.
Rule 2: Electronic signature is MANDATORY for order signing. Users with ORES-equivalent roles must complete electronic signature authentication before any order is released to ancillary services. This replaces VistA's stored signature code (#200 field 20.4) with modern authentication (re-enter password, biometric, PIV/smart card).
Rule 3: LRLAB key should NOT be given to anyone outside of laboratory staff. VistApedia explicitly states this. The modern role system must prevent assigning Lab Technologist or Lab Supervisor roles to users who also hold clinical roles outside the lab — or at minimum, flag this as an unusual combination requiring justification.
Rule 4: XUPROG (programmer access) should NOT EXIST in production environments. This key bypasses all security controls and allows direct global manipulation. The modern product must NOT provide M programmer access through the web UI. If M-level debugging is needed, it should be done through separate secure terminal access, not the clinical application.
Rule 5: Break-the-glass for sensitive records. VistA's DG SENSITIVITY key + patient sensitivity flag must be enhanced with a formal BTG workflow: (1) user attempts to access sensitive record, (2) system blocks and displays warning, (3) user provides justification, (4) access granted with enhanced audit trail, (5) supervisor notification sent. BTG events must be reviewable in audit reports.
Rule 6: Role templates from the Tenant Profile Matrix (Phase 2, Artifact 1) define WHICH roles are pre-created per tenant type. A standalone pharmacy (Profile D) pre-creates Pharmacist, Pharmacy Technician, CS Inspector, Billing Clerk, and System Admin roles — but NOT Physician, Nurse, Radiologist, or Surgeon roles.
Rule 7: Permission inheritance should follow a clear hierarchy: System Admin inherits all permissions. Department Supervisor inherits all staff permissions for their department. Staff roles have operational permissions only. Read-only roles see data but cannot modify. The hierarchy must be documented and reviewable.
Rule 8: SMART-on-FHIR scopes (Column E in the key tables) define what FHIR API access each role gets. When the product exposes FHIR APIs (Phase 2, Artifact 9: FHIR API Architecture), these scopes must be enforced: a Pharmacy Technician cannot access surgery data via FHIR even if they try to call the Procedure endpoint directly.
Rule 9: Temporary access elevation must be audited. If a supervisor temporarily grants additional permissions to a user (e.g. covering for an absent colleague), this must be: (a) time-limited (auto-expires), (b) logged with reason, (c) approved by supervisor. VistA has no built-in mechanism for this; the modern product must add it.
Rule 10: IB PARAMETER EDIT key protects billing configuration. IB TM: 'Starting with IB*2*568, locked with security key IB PARAMETER EDIT.' The modern product must maintain this protection: billing configuration changes require Billing Supervisor role, are audited, and may require approval workflow for critical parameters.
Rule 11: INTERNATIONAL ADAPTATION: The security key architecture is UNIVERSAL. Roles like Physician, Nurse, Pharmacist, Lab Tech exist in every healthcare system. What changes: (a) credential requirements per role vary by country (medical license number type), (b) scope of practice varies (nurse prescribing authority differs), (c) regulatory audit requirements vary (HIPAA, GDPR, etc.). The role system must support configurable scope-of-practice rules per country.
Rule 12: Every VistA key assignment and modern role assignment must be AUDITABLE. The audit trail must show: who was assigned what role, when, by whom, and what VistA keys were auto-granted as a result. Role changes are security-sensitive events and must be logged for compliance review.
