# VistA Evolved
Audit / Irreversibility Matrix
Action Reversibility + Audit Level + Undo Mechanism + Patient Safety Impact
March 2026 | Maps every clinical and administrative action across all workspaces to: (1) Reversibility — can the action be undone and how? (2) Audit Level — what is logged and where? (3) Undo Mechanism — how to reverse if needed. (4) Patient Safety Impact — what happens if this action is wrong? (5) Regulatory Requirement — which regulations mandate the audit trail? Grounded in: IB ACTION #350 (billing audit trail — 'data designed to provide an audit trail of transactions'), FileMan field-level auditing, Kernel audit logging (timestamps, IP address, user, menu options), VA OIG FISMA audit requirements, HIPAA Security Rule audit controls (45 CFR 164.312(b)), GDPR Article 30 (records of processing activities). Categories: R = Reversible (can undo freely), RA = Reversible with Approval (undo requires supervisor), I = Irreversible (cannot undo, only amend/addend), C = Compensating (create a correcting record rather than deleting).

# 1. Audit & Reversibility Framework
## 1.1 Reversibility Categories
Category
Code
Definition

Example
Reversible
R
Action can be undone by the same user or equivalent role. No approval needed. Original state fully restored.
Standard Undo button or Edit action. No confirmation beyond normal save.
Change appointment time, update patient phone number, edit unsigned progress note.
Reversible with Approval
RA
Action can be undone but requires supervisor or second-party approval. May require documented reason.
Undo/Reverse action available but triggers approval workflow: reason required, supervisor notification, approval before execution.
Cancel a signed order, reverse a bill, remove a verified lab result, retract a signed note.
Compensating
C
Action cannot be truly undone, but a CORRECTING action can be recorded that offsets or amends the original. Original record preserved with correction appended.
Amend/Addend/Correct action instead of Delete/Undo. Creates a new record referencing the original.
Amend a signed progress note (addendum), issue a corrected lab result (amendment), create a billing adjustment (decrease/cancellation transaction). IB UM: 'If charges removed, a separate cancellation-type transaction created.'
Irreversible
I
Action cannot be undone and no compensating action is possible. Once executed, the effect is permanent.
Double-confirmation required: warning dialog, reason entry, typed confirmation phrase. Prominent warning about irreversibility.
Patient death entry, controlled substance destruction witnessing, data purge, record merge.

# 10. Rules for Audit and Irreversibility
Rule 1: The medical record is a LEGAL DOCUMENT. Signed clinical notes, verified lab results, signed operative reports, medication administration records, and other finalized clinical documentation can NEVER be deleted. They can only be AMENDED with addenda that preserve the original and clearly indicate the correction. This is not a technical limitation — it is a legal requirement.
Rule 2: Financial transaction audit trails (IB ACTION #350, AR #430 transaction history) must follow the IB pattern: 'Data in this file should not be added to, edited, or deleted.' Corrections create COMPENSATING transactions (cancellations, adjustments, decreases) — they do not modify the original record. This is standard financial audit practice.
Rule 3: Every action categorized as RA (Reversible with Approval) must trigger a WORKFLOW: (a) user initiates reversal action, (b) system captures reason, (c) notification sent to supervisor, (d) supervisor reviews and approves/denies, (e) if approved: action executed and logged, (f) if denied: requester notified. The modern UI must implement this approval workflow.
Rule 4: Every action categorized as I (Irreversible) must have DOUBLE CONFIRMATION: (a) warning dialog stating the action is irreversible, (b) reason entry required, (c) typed confirmation (e.g. type 'CONFIRM DESTRUCTION' — not just click OK), (d) for CS destruction: second witness confirmation on separate device/session. The modern UI must make irreversibility unmistakably clear.
Rule 5: Patient record merge is the HIGHEST-RISK administrative action. Wrong merge combines two different patients' clinical records. The modern UI must require: (a) side-by-side comparison of both records, (b) explicit confirmation of each data element to retain, (c) supervisor approval, (d) merge preview before execution, (e) comprehensive audit trail. Consider requiring TWO independent approvals.
Rule 6: Controlled substance actions (dispensing, destruction, inventory adjustment) are subject to DEA regulations and CANNOT be reversed. The audit trail is a legal document subject to DEA inspection. The modern UI must enforce: witness requirements, perpetual inventory balance, and discrepancy reporting. See Vol 5D rules.
Rule 7: Lab result amendments must clearly show BOTH the original and corrected values. The ordering provider must be NOTIFIED of the amendment. The modern UI must: (a) display amended results with a visual indicator (e.g. 'Amended' badge), (b) show both values on hover/expand, (c) log notification delivery to ordering provider.
Rule 8: Audit log storage must be IMMUTABLE. The IB ACTION #350 pattern — data should not be added to, edited, or deleted — must apply to ALL audit logs. Modern implementation: append-only database tables with hash chaining (each entry includes hash of previous entry, making tampering detectable). Export audit logs to separate, isolated storage nightly.
Rule 9: Audit RETENTION periods vary by data type and jurisdiction. Clinical records: per state/country medical record retention law (typically 6-10 years after last patient contact, longer for minors). Financial records: per federal/state financial record retention (typically 6-10 years). Controlled substance records: per DEA (minimum 2 years). The system must enforce minimum retention and prevent premature purging.
Rule 10: Data export and access logging must capture ENOUGH DETAIL to support breach investigation. If a breach is suspected, the audit trail must answer: Who accessed what patient data? When? From where (IP/device)? What did they do with it (view, print, export, copy)? HIPAA breach investigation requires this level of detail.
Rule 11: The SRTASK-NIGHT (Surgery nightly) and PRCA NIGHTLY PROCESS (AR) perform case locking and data integrity checks. The modern UI must provide visibility into which records have been locked by nightly processes and must not allow casual editing of locked records without supervisor override.
Rule 12: INTERNATIONAL ADAPTATION: Audit requirements vary by country but the DIRECTION is always toward MORE auditing, not less. HIPAA (US), GDPR (EU), and equivalents all require access logging, data modification tracking, and breach notification. The audit framework should default to the STRICTEST level (L2/L3) for clinical and financial data, and allow relaxation only where regulatory profile explicitly permits it.
