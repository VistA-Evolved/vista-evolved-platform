# 1. Device Integration Architecture
Medical device integration is one of the most complex aspects of EHR deployment. Devices range from simple barcode scanners (USB keyboard-wedge, no special driver) to complex imaging modalities (DICOM network protocol with worklist, storage, and query/retrieve services). The modern product must support all categories while abstracting the complexity behind a consistent integration framework.

Keyboard-Wedge / HID
Device sends data as keyboard keystrokes. No special driver or protocol. Simplest integration.
Barcode scanners (1D/2D), magnetic stripe readers, some weight scales. BCMA uses keyboard-wedge scanners (Vol 5C).
Web browser receives keystrokes in focused input field. Detect scan vs manual typing by speed (scans complete in <100ms). Pattern matching on barcode format (GS1, Code 128, etc.).
Serial / RS-232
Device communicates via serial port (COM port). Legacy protocol still common in lab instruments.
Older lab analyzers (chemistry, hematology, coagulation), some vital signs monitors, some scales.
Serial-to-TCP bridge (e.g. Digi PortServer, Moxa NPort). VistA AUTO INSTRUMENT #62.4 handles serial lab communication. Modern: use middleware (Mirth Connect, Rhapsody) for serial-to-HL7/FHIR translation.
HL7 v2.x Messaging
Device sends/receives HL7 v2.x messages (ORM, ORU, ADT, SIU) over TCP/IP (MLLP — Minimum Lower Layer Protocol).
Lab analyzers (modern), imaging RIS-PACS communication, ADT feeds, surgery interfaces (#133.2). VistA uses HL7 LOGICAL LINK #870 for HL7 connections.
HL7 v2.x remains the dominant device protocol. Integration engine (Mirth Connect, Rhapsody, or custom) translates between device HL7 and VistA HL7/modern FHIR. VistA Imaging: 'Approved PACS HL7 Interfaces' list (Dec 2025).
DICOM
Medical imaging standard. Network services: Store (C-STORE), Query/Retrieve (C-FIND/C-MOVE/C-GET), Worklist (MWL), Print, Structured Reporting.
All imaging modalities: X-ray, CT, MRI, ultrasound, nuclear medicine, mammography, PET, dental, ophthalmology, endoscopy, surgical video. VistA Imaging: 'Approved DICOM Modality Interfaces' list (Dec 2025 — hundreds of approved devices across 20+ categories).
DICOM gateway (VistA Imaging DICOM gateway or modern equivalent). DICOMweb for modern HTTP-based DICOM access (STOW-RS, WADO-RS, QIDO-RS). IHE profiles for enterprise imaging workflows.
ASTM E1381 / LIS2-A2
Lab instrument communication standard. Bidirectional: host sends orders to analyzer, analyzer returns results. Character-based protocol over serial or TCP.
Lab analyzers that don't support full HL7: older chemistry analyzers, some POCT devices, some specialty analyzers.
VistA AUTO INSTRUMENT #62.4 supports ASTM protocol. Modern: ASTM-to-HL7 middleware translation. Newer analyzers support HL7 directly, making ASTM less common in new installations.
IEEE 11073 / SDC
Point-of-care device communication standard. Real-time physiological data: vital signs, waveforms, ventilator parameters, infusion pump status.
Bedside vital signs monitors, ventilators, infusion pumps, anesthesia machines, patient-controlled analgesia (PCA) pumps, telemetry systems.
IEEE 11073 for device semantics. IHE PCD (Patient Care Devices) profiles. Modern: FHIR Device + Observation resources for device data. Real-time streaming via WebSocket or MQTT.
POCT1-A
Point-of-care testing connectivity standard (CLSI/IEEE). Connects POCT devices (glucometers, blood gas analyzers, iStat, etc.) to LIS/EHR.
Bedside glucometers, i-STAT blood gas, rapid strep/flu, urinalysis dipstick readers, pregnancy tests, coagulation (INR).
POCT1-A defines device-to-data-manager protocol. Data manager aggregates POCT results and files to LIS/EHR via HL7. Modern: FHIR-based POCT integration emerging.
Proprietary API / SDK
Vendor-specific interface. Device manufacturer provides SDK, REST API, or custom protocol.
Automated dispensing cabinets (Omnicell, BD Pyxis), smart infusion pumps (Alaris, Plum), surgical robots (da Vinci), e-prescribing hubs, pharmacy POS terminals.
Vendor-specific integration using their API/SDK. Wrap in adapter pattern to standardize data flow into VistA/modern product. Each vendor integration is a separate development effort.

# 9. Rules for Device Integration
Rule 1: BARCODE SCANNING (BCMA) is the MOST CRITICAL device integration for patient safety. Every BCMA workflow depends on barcode scanning to verify the Five Rights. The product must support multiple scanner connection methods (USB HID, Bluetooth, wireless) and must handle scan failure gracefully (scanner disconnected, unreadable barcode, barcode not in system).
Rule 2: Lab instrument integration via AUTO INSTRUMENT #62.4 is VistA's PROVEN infrastructure for bidirectional analyzer communication. The modern product must preserve this infrastructure. Do NOT attempt to replace VistA's lab instrument interface with a from-scratch implementation — it has decades of analyzer-specific tuning.
Rule 3: DICOM is the ONLY acceptable protocol for medical imaging device integration. The modern product must provide: (a) Modality Worklist (MWL) so technologists see scheduled exams on the modality, (b) Storage (C-STORE SCP) to receive acquired images, (c) Query/Retrieve (C-FIND/C-MOVE) for image access. VistA Imaging's approved modality list (Dec 2025) with hundreds of validated devices is a significant asset.
Rule 4: Device integration MIDDLEWARE (e.g. Mirth Connect, Rhapsody, or custom integration engine) is ESSENTIAL for production deployment. Devices speak many protocols (HL7 v2.x, ASTM, DICOM, POCT1-A, proprietary). The middleware translates device protocols to VistA's expected format and handles: connection management, error recovery, message queuing, logging, and monitoring.
Rule 5: The Tenant Profile Matrix (Phase 2, Artifact 1) determines which device integration categories are relevant per tenant type. Device setup wizards in the admin UI should be PROFILE-FILTERED — a standalone pharmacy (Profile D) should not see options for configuring lab analyzers or DICOM gateways.
Rule 6: Every device integration must have a MONITORING DASHBOARD showing: (a) connection status (online/offline), (b) last communication timestamp, (c) message throughput (messages/hour), (d) error rate, (e) queue depth for pending messages. Device failures must trigger alerts to technical staff via the notification system.
Rule 7: Device VALIDATION is required before clinical use. VistA Imaging: 'VA facilities are required to complete a quality review for the images produced by the device before it may be placed into clinical use.' This applies to all device categories: lab results must be verified against known control samples, imaging quality must be reviewed, BCMA scanning must be tested with actual medication barcodes.
Rule 8: Smart infusion pump integration (drug library download + auto-programming from BCMA) is an EMERGING patient safety standard. The modern product should plan for this integration even if not implemented in Wave 1. The drug library (dose limits per drug per patient weight) comes from the pharmacy formulary; the infusion parameters come from the BCMA medication order.
Rule 9: POCT device integration requires PATIENT IDENTITY VERIFICATION at the device. The glucometer or iSTAT must scan the patient's wristband barcode before accepting the specimen. This prevents wrong-patient POCT results — a common lab safety event. The POCT data manager enforces this workflow.
Rule 10: INTERNATIONAL ADAPTATION: Device protocols (DICOM, HL7, ASTM, IEEE 11073) are INTERNATIONAL STANDARDS. The same GE CT scanner uses the same DICOM protocol in New York, London, and Tokyo. Device integration is one of the MOST UNIVERSAL aspects of the product — the same integration code works worldwide. What may change: (a) local device brands/models, (b) regulatory validation requirements, (c) local barcode formats (GS1 internationally, NDC-based in US pharmacy).
Rule 11: Automated Dispensing Cabinet (ADC) integration is VENDOR-SPECIFIC and complex. Each ADC vendor (BD Pyxis, Omnicell, Aesynt) has different interface requirements, different data models, and different medication matching rules. ADC integration should be treated as a SEPARATE integration project per vendor, not a generic device category.
Rule 12: Surgical video (robotic surgery, laparoscopic video) is increasingly treated as part of the medical record and stored via DICOM. The VistA Imaging Dec 2025 approved list includes Intuitive Surgical (da Vinci) Hub Media Server. The modern product should support DICOM video storage and playback for surgical documentation.
