# VistA Evolved
FHIR R4 API Architecture
Resource Mapping + Read/Write Capabilities + SMART Authorization + Regulatory Compliance
March 2026 | Defines VistA Evolved's complete FHIR R4 API layer. Grounded in: VA Lighthouse FHIR API (existing read-only precedent aggregating VistA/CDW/HDR/MPI data), HL7 FHIR R4 specification (v4.0.1), US Core Implementation Guide (v7.0.0), 21st Century Cures Act (ONC mandates FHIR R4 APIs for certified health IT), TEFCA (finalized Dec 2024 — requires FHIR support for participating networks), SMART on FHIR (v2.0 — OAuth2-based app authorization), International Patient Summary (IPS — ISO 27269), UK FHIR profiles (NHS Digital), AU FHIR profiles (HL7 AU Base). J P Systems (Lighthouse data mapping contractor): 'Key issues included resolving basic differences in workflow for medication orders, fitting specialized VA vocabularies into FHIR categories, and managing different access constraints in clinician-facing and patient-facing situations.' VistA Evolved's FHIR API goes BEYOND Lighthouse: it supports WRITE operations (not just read-only), enabling third-party apps to create orders, document encounters, and update patient data through FHIR — with full VistA validation and security.

# 2. FHIR API Architecture
## 2.1 Layered Architecture
The FHIR API is a LAYER that sits between external consumers (third-party apps, patient portals, HIEs) and VistA's data. It does NOT replace VistA's internal data model — it translates VistA data into/from FHIR format on-the-fly.
Layer
Component
Function
Technology
## 1. External Consumer
Third-party apps, patient portals, HIE nodes, CDS services, payer systems, research platforms
Sends FHIR requests (GET, POST, PUT, DELETE). Receives FHIR responses (JSON/XML resources).
HTTPS, OAuth2 bearer tokens, SMART on FHIR app launch
## 2. API Gateway
Authentication, authorization, rate limiting, request routing, API key management, audit logging
Validates OAuth2 tokens. Enforces SMART scopes (patient/*.read, patient/Observation.write, etc.). Rate limits per consumer. Logs all API access for HIPAA audit.
Kong, AWS API Gateway, or custom. OAuth2/OIDC provider (Keycloak, Auth0, or custom).
## 3. FHIR Server
FHIR R4 compliant server handling RESTful operations: read, search, create, update, delete, batch, transaction
Parses FHIR requests. Routes to appropriate VistA data adapter. Returns FHIR-formatted responses. Handles FHIR search parameters, _include, _revinclude, pagination.
HAPI FHIR (Java, open source), Microsoft FHIR Server (.NET), or custom FHIR facade. HAPI FHIR recommended for flexibility and community support.
## 4. VistA Data Adapter
Translation layer between FHIR resources and VistA FileMan data. READ: queries VistA via RPCs/wrappers, transforms to FHIR. WRITE: validates FHIR resource, translates to VistA data format, calls VistA RPCs/wrappers to persist.
Maps every FHIR resource element to specific VistA file/field (using Product Vocabulary Master Sheet mappings). Handles: date format conversion (FileMan -> ISO 8601), terminology mapping (VA local codes -> FHIR standard codes), identifier translation (DFN -> Patient.id).
Custom adapter layer. Uses ZVE* custom wrapper RPCs from Volumes 1-12 for VistA data access. Wraps VistA RPC Broker calls.
## 5. VistA Instance
YottaDB/GT.M + M routines + FileMan data dictionary
The system of record. ALL data persists in VistA globals. FHIR API is a view/interface layer, not a separate data store. VistA validation rules (order checks, drug interaction checking, eligibility verification) apply to FHIR write operations.
VistA (one instance per tenant, per product architecture).
CRITICAL ARCHITECTURAL PRINCIPLE: VistA remains 100% the system of record. The FHIR API does NOT store data in a separate FHIR database. Every FHIR read queries VistA in real-time. Every FHIR write persists through VistA's validation pipeline. This ensures data consistency: the web UI, the FHIR API, and any terminal access all see the same data.

# 4. SMART on FHIR Authorization
SMART on FHIR (Substitutable Medical Applications and Reusable Technologies) is the standard framework for authorizing third-party apps to access FHIR data. It uses OAuth2 with healthcare-specific extensions.
Component
Implementation
Notes
OAuth2 Authorization Server
Integrated with VistA Evolved's identity provider (Keycloak, Auth0, or custom). Issues access tokens with SMART scopes.
Must support: Authorization Code flow (for user-facing apps), Client Credentials flow (for backend service apps), PKCE (Proof Key for Code Exchange) for public clients.
SMART Scopes
Scopes define what data the app can access. Format: [context]/[ResourceType].[operation]. E.g. patient/Observation.read, user/MedicationRequest.write, launch/patient.
Scopes map to VistA security keys via Security/Permission Translation Matrix (Phase 2, Artifact 6). A SMART scope of patient/MedicationRequest.write requires the underlying user to have ORES-equivalent permission.
Patient Context Launch
App launches in context of a specific patient. FHIR server provides patient ID in token response. App can only access that patient's data.
Used for: EHR-embedded apps (launched from within VistA Evolved UI for a specific patient). Patient portal apps (patient launches, sees only own data).
EHR Launch
App launched from within the EHR UI. EHR provides launch context (patient, encounter, user) to the app.
VistA Evolved web UI acts as the EHR launch host. Third-party apps receive patient/encounter context automatically.
Standalone Launch
App launches independently, then connects to FHIR server. User authenticates and selects patient.
Used for: standalone clinical tools, population health apps, research tools.
Backend Service (System) Access
Server-to-server communication. No user interaction. Uses client credentials + JWT assertion.
Used for: bulk data export, population health reporting, HIE data exchange, system integration.
Token Introspection / Refresh
Access tokens are short-lived (e.g. 5-15 minutes). Refresh tokens allow renewal without re-authentication. Token introspection endpoint for validation.
Enforce token expiration. Revoke tokens on user logout or role change.

## 5.1 Write Capability: Beyond Read-Only
VA Lighthouse is READ-ONLY. VistA Evolved's FHIR API must support WRITE operations to enable: third-party order entry (CDS apps placing orders), patient portal actions (appointment requests, medication refill requests, demographic updates), and external system data filing (lab results from reference labs, immunization records from pharmacies, clinical documents from outside providers).
WRITE operations go through VistA's FULL validation pipeline. A MedicationRequest created via FHIR triggers the same drug interaction checks, allergy checks, duplicate order checks, and formulary verification as an order entered through the web UI. The FHIR API is NOT a backdoor around clinical safety checks.

# 9. Rules for FHIR API Architecture
Rule 1: VistA is 100% the system of record. The FHIR API is a TRANSLATION LAYER, not a separate data store. Every FHIR read queries VistA in real-time. Every FHIR write persists through VistA's validation pipeline. There is NO separate FHIR database that could become inconsistent with VistA.
Rule 2: FHIR WRITE operations must go through the SAME validation as web UI operations. A MedicationRequest created via FHIR must trigger the same drug interaction checks, allergy checks, and formulary verification as an order entered through the clinical workspace. The FHIR API is NOT a backdoor around clinical safety checks.
Rule 3: SMART on FHIR scopes map to VistA security keys via the Security/Permission Translation Matrix (Phase 2, Artifact 6). An app requesting patient/MedicationRequest.write must have a user context with ORES-equivalent permission. An app requesting patient/Observation.read for lab results must have a user context with appropriate lab access.
Rule 4: Patient-facing FHIR APIs are MANDATORY under the 21st Century Cures Act (US). Patients must be able to access their complete electronic health information through third-party apps of their choosing. VistA Evolved must support this for US tenants. For international tenants: patient data access requirements vary by country.
Rule 5: The VistA Data Adapter (Layer 4) uses the ZVE* custom wrapper RPCs defined in Volumes 1-12 to access VistA data. Each FHIR resource maps to one or more wrapper RPCs. The adapter does NOT directly read VistA globals — it goes through the RPC layer, which enforces VistA's security and validation.
Rule 6: Terminology mapping is critical. VA uses local vocabularies (VA Drug Class, clinic stop codes, treating specialties) that must be mapped to standard terminologies (RxNorm, SNOMED CT, LOINC, ICD-10) for FHIR compliance. The Product Vocabulary Master Sheet (Phase 2, Artifact 2) provides these mappings. The FHIR API must include both the local code and the standard code in CodeableConcept resources.
Rule 7: US Core compliance is the MINIMUM for US tenants. The US Core Implementation Guide defines: required resource profiles, mandatory search parameters, required terminology bindings, and USCDI data class coverage. VistA Evolved's FHIR API must pass US Core conformance testing (using Inferno or equivalent test suite).
Rule 8: International tenants use their country's FHIR Implementation Guide INSTEAD OF (or in addition to) US Core. UK tenants use UK Core profiles. Australian tenants use AU Base profiles. The FHIR server must support profile selection per tenant configuration in the Internationalization Framework (Phase 2, Artifact 3).
Rule 9: Bulk data export must be AUTHORIZED and AUDITED. Population-level data export is a high-risk operation (potential for mass data breach). Bulk export requires system-level authorization, is logged in the audit trail, and should be rate-limited. The Audit/Irreversibility Matrix (Phase 2, Artifact 7) classifies data export as L2 audit level.
Rule 10: FHIR Subscriptions enable the modern product to participate in event-driven healthcare networks (TEFCA, HIE notifications, care coordination). Subscription topics should be configurable per tenant — not all tenants need all subscription types.
Rule 11: The FHIR CapabilityStatement resource is the 'menu' of what the FHIR server supports. It must accurately reflect: supported resources, supported interactions (read, search, create, update), supported search parameters, supported SMART scopes, and conformance to implementation guides. External systems read the CapabilityStatement to discover the server's capabilities.
Rule 12: Performance and scalability are critical. Lab-heavy facilities may generate thousands of Observation resources per day. The FHIR API must handle: efficient search (indexed search parameters), pagination (for large result sets), caching (ETag/Last-Modified headers), and bulk operations (for batch data exchange). Response time target: < 500ms for single resource reads, < 2 seconds for searches, < 5 minutes for bulk export initiation.
