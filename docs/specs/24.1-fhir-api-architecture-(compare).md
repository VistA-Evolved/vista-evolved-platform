# VistA Evolved
FHIR API Architecture
HL7 FHIR R4 Interoperability Layer for VistA Evolved
March 2026 | Defines VistA Evolved FHIR R4 API layer: VistA file-to-FHIR resource mapping, SMART-on-FHIR authorization, US Core/USCDI compliance, read/write operations, Bulk Data Access, Subscriptions. Grounded in: VA Lighthouse FHIR API (production at developer.va.gov), WorldVistA FHIR-on-VistA (open source Spring Boot + HAPI FHIR R4), ONC 21st Century Cures Act, TEFCA (Dec 2024), USCDI v4, SMART App Launch v2.0, HL7 FHIR R4 specification.

# 1. Why FHIR Is Mandatory
FHIR R4 is not optional — it is a regulatory, market, and technical requirement for any modern EHR product.
Driver
Requirement
Impact
ONC 21st Century Cures Act
Certified health IT must support FHIR R4 APIs. Patient access APIs mandatory. Information blocking prohibited.
VistA Evolved MUST expose patient data via FHIR R4 for ONC certification.
TEFCA (Dec 2024)
Qualified Health Information Networks must support FHIR APIs. Creates nationwide HIE infrastructure.
Tenants participating in data exchange must have FHIR capability.
USCDI v4
Minimum data set exchangeable via FHIR: demographics, allergies, medications, problems, procedures, vitals, labs, notes, immunizations.
FHIR API must expose all USCDI v4 elements via US Core profiles.
CMS Interoperability Rules
Medicare/Medicaid payers must support Patient Access API (FHIR). Prior auth via FHIR.
Tenants billing CMS must integrate with payer FHIR APIs.
International Equivalents
UK: NHS Digital FHIR UK Core. Australia: My Health Record FHIR. EU: European Health Data Space moving toward FHIR. Canada: pan-Canadian FHIR.
FHIR R4 is the GLOBAL standard. Same FHIR layer with country-specific profiles.
Third-Party App Ecosystem
SMART-on-FHIR enables clinical apps to launch within EHR with patient context.
Product must support SMART App Launch for third-party app marketplace.
VA Lighthouse Precedent
VA operates production FHIR R4 APIs at developer.va.gov. WorldVistA has open-source FHIR-on-VistA. J P Systems provided VistA-to-FHIR terminology mappings.
Proven feasibility. VistA data CAN be mapped to FHIR R4.

## 2.1 Layered Architecture

Custom mapping layer translating between VistA structures and FHIR resources.

# 9. Rules
Rule 1: VistA is ALWAYS the system of record. FHIR API is a GATEWAY — reads from and writes to VistA. Every FHIR write goes through VistA business rules. If VistA rejects, FHIR returns OperationOutcome error. FHIR does NOT bypass VistA.
Rule 2: FHIR resource IDs must be STABLE and CONSISTENT. Derivable from VistA DFN/IEN. Same ID today and tomorrow for same record. Must survive FHIR server restarts.
Rule 3: US Core profiles are MINIMUM conformance for US deployment. Every resource must conform to applicable US Core profile. Validation in build pipeline.
Rule 4: SMART-on-FHIR scopes must align with Security/Permission Matrix. Nurse (ORELSE) gets user/MedicationAdministration.write but NOT user/MedicationRequest.write. Scope enforcement at authorization server.
Rule 5: Terminology mapping via Product Vocabulary Master Sheet. VistA internal codes translated to: LOINC (labs), RxNorm (meds), SNOMED CT (clinical), ICD-10 (diagnoses), CVX (vaccines), UCUM (units). Missing mappings flagged.
Rule 6: CapabilityStatement (GET /metadata) must ACCURATELY declare supported resources, search parameters, interactions, profiles, and SMART endpoints. Clients rely on it for discovery.
Rule 7: FHIR writes must respect Audit/Irreversibility Matrix. Creating signed note via FHIR POST subject to same C/L3 rules as UI creation. No backdoor to bypass audit.
Rule 8: Bulk Data export is HIGH-RISK. Backend Services auth + tenant admin approval. Every export logged: who, what, when, how much. Encrypted in transit and at rest.
Rule 9: Performance targets: single read < 200ms, search < 500ms, Bulk Data initiation < 5s (async). Caching for reference data (Organization, Location, Practitioner, Medication).
Rule 10: Subscriptions opt-in per tenant. Admin configures active topics and endpoints. Failures must not affect clinical operations.
Rule 11: INTERNATIONAL ADAPTATION: FHIR R4 is global but country profiles exist: US Core, UK Core, AU Base, Canadian FHIR, International Patient Summary. Profile selection per tenant locale. Core structure same — profiles add country-specific elements.
Rule 12: WorldVistA FHIR-on-VistA and VA Lighthouse are PRECEDENTS proving feasibility. J P Systems identified key challenges: medication order workflow differences, VA vocabulary-to-FHIR category mapping, clinician-vs-patient access constraints. Build on precedents, not from zero.
