# 1. Purpose and Scope
This document is the second truth-layer artifact for VistA Evolved. It sits directly below the Module-to-Package Family Mapping Matrix and directly above the future Page Inventory Matrix and Screen Contracts.
It answers three questions that the previous artifact did not yet answer:
(A) Terminal Option Inventory: What are the actual terminal menu options, organized by workspace, that configure and operate VistA? What user job does each option perform? Is it configuration/admin or daily operational work? What files does it touch?
(B) CPRS Dependency Map: For each major CPRS feature area, what VistA packages, terminal configuration, and setup tasks power it? What must be configured in the terminal before CPRS (and our modern client) can work correctly?
(C) RPC Coverage Ledger: For each major workflow action in the Wave 1 workspaces, does an existing RPC exist? Is it adequate? Is a custom wrapper needed? What is the context option? What files are touched?
This volume covers the 4 Wave 1 workspaces: Admin and Security, Scheduling, Clinical Workspace, and Patients and Registration.

## 2.4 Terminal Option Inventory: Site Parameters
The KERNEL SYSTEM PARAMETERS file (#8989.3) is the master site configuration file. It contains hundreds of parameters that control system behavior across all packages. Key parameter groups documented in the Kernel 8.0 Technical Manual (July 2025) include:
Signon/Security parameters: failed signon attempts limit, auto-sign-off interval, multiple signon rules, verify code change frequency, access code length requirements.
Device Handler parameters: spool directory, default terminal types, resource device limits.
TaskMan parameters: task scheduling, maximum concurrent tasks, background job limits.
Audit parameters: signon audit, option audit, file audit settings.
ePCS parameters: electronic prescribing for controlled substances configuration.
WRAPPER NEEDED: There is no RPC to read or edit site parameters. A governed admin wrapper is needed for any site parameters exposed in the modern admin UI.
## 2.5 Terminal Option Inventory: Division and Institution Setup
Division setup is critical for multi-hospital/multi-clinic tenants. The relevant files are:
INSTITUTION file (#4): Defines each institution/facility with station number, name, address.
MEDICAL CENTER DIVISION file (#40.8): Links divisions to parent institution. Each division has its own characteristics.
NEW PERSON file (#200) DIVISION multiple (#16): Each user is assigned to one or more divisions. The login context (DUZ(2)) is set from this during sign-on.
Source: Kernel 8.0 Signon/Security User Guide (August 2025): Users should be set up only for the stations they should access. DUZ(2) stores the login institution IEN.
WRAPPER NEEDED: Division/institution editing is done through FileMan Enter/Edit on files #4 and #40.8. No RPC exists for this. A governed wrapper is needed for tenant facility setup in the modern UI.

## 2.7 Electronic Signature Setup
Electronic signature in VistA is configured through the NEW PERSON file (#200). The relevant fields are:
ELECTRONIC SIGNATURE CODE (#20.4): A hashed code the user enters to sign documents and orders. This is NOT a visual signature; it is a secret code like a password.
SIGNATURE BLOCK PRINTED NAME (#20.2): The text that prints as the signer's name when documents are printed.
DATE E-SIG LAST CHANGED (#20.1): Tracks when the signature code was last changed.
Source: Kernel 8.0 Signon/Security User Guide (August 2025): Enter a new electronic signature code. This is a code (like a password) used to electronically sign documents within VistA.
CRITICAL FOR MODERN CLIENT: The modern product's visual signature feature (drawing on screen, tablet, etc.) must be treated as a supplemental enhancement, NOT a replacement for VistA's electronic signature code system. The VistA code must still be set up and valid. Visual signature images can be stored as supplemental artifacts linked to the user record.
WRAPPER NEEDED: E-sig code setup is done through terminal Edit User. A governed wrapper RPC is needed to allow admin setup of signature block name and to verify/manage e-sig readiness without exposing the actual code.

# 8. What This Means for Design and Coding Teams
## 8.1 For the Design Team
You now know which workflows have real RPC backing (scheduling daily-use, clinical chart viewing, basic patient search) and which workflows have NO backend API and must be designed knowing that custom wrappers will be built (user management, key assignment, CPRS configuration, registration, pharmacy fulfillment, lab fulfillment).
When designing pages for well-covered areas (scheduling appointments, viewing charts, searching patients), you can design with confidence that the backend calls exist.
When designing pages for poorly-covered areas (user admin, clinic config, order menu setup), you must design knowing that engineering will need to CREATE the backend calls. Your screen contracts must be extra precise about what data is needed and what actions are performed, because the AI coder cannot look up an existing RPC to understand the intent.
## 8.2 For the Engineering / AI Coding Team
You now know that:
(a) Scheduling is the safest first build. The SDES/SDES2 RPC surface is rich and well-documented.
(b) Clinical chart viewing reuses the large OR namespace. Study the CPRS Delphi source code to understand exact RPC call patterns.
(c) Admin/user/provider management has NO existing RPCs for the core tasks. Every wrapper you build must: create a B-type context option in OPTION #19, register the RPCs in REMOTE PROCEDURE #8994, validate inputs, enforce file-level security, and return structured results.
(d) CPRS configuration (order menus, notifications, order checks, parameters) is entirely terminal-based. This is the LARGEST custom wrapper effort in the entire project.
## 8.3 The Build Order
Step 1: Build the RPC Broker adapter and session management for the modern client.
Step 2: Build basic user admin wrappers (ZVE USER CREATE, ZVE USER EDIT, ZVE KEY ASSIGN).
Step 3: Build the Scheduling workspace using existing SDES/SDES2 RPCs.
Step 4: Build the Clinical Chart Shell using existing OR namespace RPCs.
Step 5: Build Patient Search using existing ORWPT RPCs, then registration wrappers.
Step 6: Build CPRS configuration admin wrappers one domain at a time (notifications, order menus, order checks).
Step 7: Begin pharmacy and lab wrappers for Wave 2-3.

(1) Page Inventory Matrix v1: For each Wave 1 workspace, list every page with purpose, role, objects, and actions.

These artifacts should be built workspace by workspace, starting with Scheduling (strongest RPC coverage), then Admin/User (most critical dependency), then Clinical Chart Shell (largest RPC surface), then Patient Registration (foundational identity).
