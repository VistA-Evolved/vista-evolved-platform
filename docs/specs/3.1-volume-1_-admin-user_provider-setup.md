# 1. What This Document Is
This is the third truth-layer artifact for VistA Evolved. It descends from the Module-to-Package Mapping Matrix (artifact 1) and the Terminal Option Inventory + RPC Coverage Ledger (artifact 2) into the actual page-level design and engineering handoff.
For each page in the Admin User/Provider Setup workspace, this document specifies: the exact purpose, the user role, every visible field with its backend file/field source, every button with what it does on the backend, every dropdown with where its options come from, every status/state the page must handle, every warning, every confirmation, and the exact RPC or wrapper that powers it.
This is not conceptual. This is implementation-grade.
Important: This volume covers only the Admin User/Provider Setup subset of the Admin & Security workspace. This is deliberately bounded. User/provider setup is the most critical dependency for every other workspace (scheduling, clinical, registration). It must be built first.

# 5. Rules for Design Team
Rule 1: Every field on every page maps to a specific VistA file and field number. If you cannot find the mapping in this document, ask engineering before designing.
Rule 2: Every dropdown's options come from a specific VistA file. Do not invent dropdown options.
Rule 3: Every button triggers a specific backend action (existing RPC or planned wrapper). If the backend does not exist yet, the button must be marked as 'needs wrapper' in your design handoff.
Rule 4: Never expose raw VistA field names, file numbers, or key names to normal tenant users. Use the product vocabulary translations in this document.
Rule 5: Never design a Delete button for users. VistA users are deactivated (DISUSER + TERMINATION DATE), not deleted.
Rule 6: Electronic signature codes are NEVER visible to admins. Admins can only see whether one exists, clear it, or edit the signature block printed name.
Rule 7: Every warning in this document is mandatory in the design. Do not remove warnings to simplify the UI.
# 6. Rules for AI Coding Team
Rule 1: Every wrapper RPC must be registered in REMOTE PROCEDURE #8994 and secured by a B-type context option in OPTION #19.
Rule 2: Every wrapper RPC must validate inputs, enforce file-level security, and return structured JSON results.
Rule 3: Access and Verify codes must be hashed using Kernel-compatible hashing. Never store or transmit them in plain text.
Rule 4: Never implement a DELETE action for #200 entries. Implement deactivation only.
Rule 5: When assigning keys, always record GIVEN BY (admin DUZ) and DATE GIVEN.
Rule 6: Test every wrapper against terminal equivalence. The same action performed via terminal and via wrapper must produce identical results in the VistA database.
