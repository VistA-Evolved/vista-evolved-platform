# VistA Evolved
Design System & Component Library
UI Pattern Language + Design Tokens + Component Specifications + Accessibility Requirements
March 2026 | The unified design system governing every screen across all 17+ volumes. Ensures visual consistency, accessibility compliance, and clinical safety across 200+ pages and 11 workspaces. Grounded in: NIST GCR 15-996 (Technical Basis for User Interface Design of Health IT), Section 508 of the Rehabilitation Act, WCAG 2.2 Level AA, CMS Design System (design.cms.gov — built on US Web Design Standards), healthcare UI/UX best practices for EHR systems. Targets: design firms (for visual implementation), engineering teams (for component development), and AI coding tools (for automated code generation from component specifications). Technology stack assumption: React + TypeScript + Tailwind CSS + shadcn/ui component library + Chart.js/D3 for data visualization.

# 1. Design Principles
Principle
Description
Enforcement
Clinical Safety First
Every design decision must prioritize patient safety. Alerts, warnings, and critical information (allergies, drug interactions, code status) must be prominently visible and non-dismissible without documented action. NIST: 'a poorly designed EHR can induce use errors that could lead to patient injury and even death.'
Safety-critical elements use red/orange semantic colors with icons. Required action fields prevent workflow progression until addressed. PHQ-9 Item 9 gate pattern (Vol 14).
Cognitive Load Minimization
Clinical users operate under time pressure and interruption. The UI must surface RELEVANT information without overwhelming. Progressive disclosure: summary first, detail on demand. Role-based views: show only what the user needs for their role.
Dashboard patterns show summary cards with drill-down. Forms use progressive disclosure (advanced fields collapsed by default). Workspace visibility filtered by user role and tenant profile.
Accessibility as Foundation
Section 508 and WCAG 2.2 Level AA compliance is MANDATORY, not an afterthought. Design for: keyboard navigation, screen readers, low vision (high contrast), color blindness (never color-only cues), motor impairment (large touch targets). Healthcare users include aging clinicians, visually impaired staff, and users of assistive technology.
Minimum 4.5:1 contrast ratio for text, 3:1 for large text and UI elements. All interactive elements keyboard-accessible. ARIA labels on all controls. Touch targets minimum 44x44px. Color always paired with icon/text/pattern.
Consistency Across Workspaces
A user moving from Pharmacy to Laboratory to Nursing should find the same interaction patterns, the same layout structures, the same color semantics. The design system is the enforcement mechanism for cross-workspace consistency.
Shared component library. Design tokens (not hard-coded values). Page layout templates. Standardized worklist, detail view, form, and dashboard patterns.
Data Density with Clarity
Healthcare screens often display MORE data than typical business applications (lab results, medication lists, vital sign trends). The design must support high data density without sacrificing readability. Use typography hierarchy, whitespace rhythm, and color coding to create visual order in dense data.
Compact mode available for power users. Data tables with configurable density. Sparkline/trend indicators in summary views. Zebra striping for long tables.
Mobile-First for Bedside Use
Nurses and physicians increasingly use tablets at the bedside. The Nurse Dashboard (NUR-01), BCMA scanning (Vol 5C), and vital signs entry must work on tablet screens. Touch-friendly interaction with large targets.
Responsive breakpoints: Desktop (1440px+), Tablet (768px-1439px), Mobile (320px-767px). Touch targets 44x44px minimum. Swipe gestures for common actions.
International Readiness
The product serves international deployments (I18n Framework, Phase 2). Design must support: right-to-left (RTL) text for Arabic/Hebrew, variable-length translated strings, locale-specific date/number/currency formatting, and culturally neutral iconography.
Logical properties (start/end vs left/right) in CSS. Flexible layouts that accommodate 30-50% text expansion. Date/number formatting via i18n library. Avoid culture-specific icons.

# 3. Core Component Specifications
## 3.1 Patient Banner (Global)
The Patient Banner is the MOST IMPORTANT component in the entire design system. It appears at the top of every screen when a patient is selected. It provides persistent patient context so clinicians always know WHICH PATIENT they are looking at — preventing wrong-patient errors, which are among the most dangerous EHR-related safety events.
Element
Specification
Safety Rationale
Patient Name
--font-size-2xl, --font-weight-bold. ALL CAPS for last name: 'SMITH, John A.' Format: LAST, First MI. Maximum width with ellipsis truncation.
Large, bold name ensures instant patient identification.
Date of Birth + Age
Format: Jan 1, 1950 (75y). --font-size-md. Age auto-calculated and displayed for quick reference.
Age is critical for dosing, screening eligibility, and clinical decision-making.
Patient ID / MRN
Displayed prominently: MRN 12345 or Last 4 SSN: 1234. --font-mono for readability.
Unique identifier for verification. Monospace font prevents digit confusion.
Gender / Sex
Displayed as icon + text: Male / Female / Other. Gender-neutral icon option.
Relevant for medication dosing, screening protocols, and clinical assessment.
Allergies
RED background chip(s) with --color-danger. 'NKA' (No Known Allergies) in green. List up to 3 allergies; '+N more' overflow link. Icon: warning triangle.
MOST SAFETY-CRITICAL banner element. Drug allergies must be visible before ANY medication order. Red ensures attention even in peripheral vision.
Code Status
If DNR/DNI or other advance directive: displayed as a colored badge. Full code = no badge (default). DNR = AMBER badge, DNR/DNI = RED badge with 'DNR/DNI' text.
Code status must be visible during emergencies. Advance Directives from SW-05 (Vol 17).
Flags / Alerts
Safety flags: Fall Risk (yellow), Isolation (blue mask icon + type), Suicide Precaution (orange), Restraints Active (red wrist icon), Behavioral Health (purple). Maximum 4 visible; '+N' overflow.
Persistent safety awareness across all screens.
Location / Status
Inpatient: Ward/Room/Bed. Outpatient: Clinic name. Status: Inpatient / Outpatient / ER. Admit date for inpatients.
Location context for orders and documentation.
Photo (optional)
Small square photo (if available in system). Placeholder silhouette if no photo. Helps with facial recognition for bedside verification.
Visual patient identification (emerging standard).
CRITICAL SAFETY RULE: The Patient Banner must be STICKY — it must remain visible as the user scrolls down the page. If the user cannot see the Patient Banner, they cannot confirm which patient they are working with. When the banner is in its compact (scrolled) state, it collapses to a single-line showing: Name + DOB/Age + Allergies + Code Status — the minimum safety-critical data.

## 3.2 Clinical Data Table
Data tables are the workhorse component — used for lab results, medication lists, order lists, worklists, patient lists, and nearly every data display. The clinical data table must support: sorting, filtering, pagination (or virtual scrolling for large datasets), row selection, inline actions, configurable column visibility, and density modes (comfortable / compact / ultra-compact).
Feature
Specification
Notes
Column Headers
--font-weight-semibold, --color-primary text on light surface-alt background. Sort indicator arrows. Click to sort ASC/DESC/none. Resize handles between columns.
Headers must be visually distinct from data rows.
Data Rows
Zebra striping: alternate --color-surface and --color-surface-alt. Hover: subtle highlight --color-primary at 5% opacity. Selected: --color-primary at 10% opacity with left border accent.
Zebra striping critical for long rows (lab results spanning many columns).
Density Modes
Comfortable: row height 48px, --font-size-base. Compact: 36px, --font-size-sm. Ultra-compact: 28px, --font-size-xs. User preference stored per table.
Nurse Dashboard needs compact mode for 15+ patients. Lab results need comfortable mode for readability.
Critical Value Highlighting
Lab values outside reference range: text in --color-danger (high) or --color-warning (low). Background tint: --color-critical-bg for critical. Always paired with arrow icon (up/down).
Patient safety: abnormal values must stand out. ICU flowsheet and lab results rely on this heavily.
Actions Column
Right-aligned column with icon buttons (edit, view detail, print, more actions dropdown). Hover reveals less-common actions. Keyboard: Tab navigates, Enter/Space activates.
Common pattern across all worklists (Pharmacy queue, Lab pending, Radiology worklist, etc.).
Empty State
Centered illustration + text: 'No results found' or 'No orders pending.' Call-to-action button if applicable (e.g. 'Enter New Order').
Prevents user confusion when table is legitimately empty vs loading error.
Loading State
Skeleton rows (animated pulse) matching table structure. Never leave table area blank during data fetch.
Indicates data is loading, not that there are no results.
Pagination / Virtual Scroll
Default: paginated (25/50/100 rows per page). For real-time data (Nurse Dashboard, ICU flowsheet): virtual scrolling with lazy loading. Total count displayed.
Virtual scrolling essential for performance on large datasets.

# 5. Accessibility Compliance
Section 508 of the Rehabilitation Act and WCAG 2.2 Level AA compliance are MANDATORY for any product used in VA healthcare facilities (federal government requirement) and strongly recommended for all healthcare products. This section defines the accessibility requirements that every component must meet.
Requirement
WCAG Criterion
Implementation
Color Contrast (text)
#### 1.4.3 Contrast (Minimum) — 4.5:1 for normal text, 3:1 for large text (18pt+)
All design token color pairs tested with contrast checker. Dark mode and high-contrast mode have separate verified palettes.
Color Not Sole Means
#### 1.4.1 Use of Color — color must not be only visual means of conveying information
Every color-coded element has icon + text label. Red lab value also has up/down arrow. Green status also has checkmark icon.
Keyboard Navigation
#### 2.1.1 Keyboard — all functionality available via keyboard
Tab order follows logical reading order. Focus indicators visible (2px --color-primary ring). Modal traps focus. Skip-to-content link on every page.
Focus Visible
#### 2.4.7 Focus Visible — focus indicator always visible on keyboard navigation
High-visibility focus ring: 2px solid --color-primary with 2px offset. Never hidden with outline:none without replacement.
Screen Reader Support
#### 4.1.2 Name, Role, Value — all UI components have accessible name and role
ARIA labels on all interactive elements. ARIA-live regions for dynamic content (toast notifications, real-time dashboard updates). Semantic HTML structure.
Text Resizing
#### 1.4.4 Resize Text — content usable at 200% zoom without loss of content
Layout uses relative units (rem, %). No fixed-width containers that break at zoom. Horizontal scrolling avoided except for explicit data tables.
Error Identification
#### 3.3.1 Error Identification — errors described in text, not just color
Error messages appear below invalid fields in --color-danger text. Error summary at top of form. Focus moves to first error on submit.

#### 3.3.2 Labels or Instructions — labels for required input
All form fields have visible labels. Required fields marked with asterisk + aria-required. Placeholder text supplements (never replaces) labels.

Loading animations are slow pulse (not flash). No auto-playing video. Respect prefers-reduced-motion OS setting.
Touch Target Size
#### 2.5.8 Target Size (Minimum) — at least 24x24px, recommended 44x44px
All clickable/tappable elements minimum 44x44px on touch devices. Desktop: minimum 24x24px with adequate spacing.

# 8. Implementation Rules for Engineering
Rule 1: NEVER hard-code color values in components. ALWAYS use design tokens (CSS custom properties). This enables theming, dark mode, high contrast, and tenant branding to work automatically.
Rule 2: EVERY component must pass automated accessibility testing (axe-core, Lighthouse accessibility audit, jest-axe). Accessibility tests are part of the CI/CD pipeline — a component that fails accessibility testing CANNOT be deployed.
Rule 3: The Patient Banner is a GLOBAL SINGLETON — there is exactly one Patient Banner instance per page, managed at the app shell level. It is NEVER re-implemented by individual workspace modules. All workspaces consume the same Patient Banner component.
Rule 4: Critical patient safety alerts (drug interactions, allergy matches, duplicate orders, PHQ-9 Item 9) must use the Patient Safety Alert Modal pattern — a blocking modal that CANNOT be dismissed without documented action. This is non-negotiable for patient safety.
Rule 5: Form validation must follow the ERROR-FIRST pattern: on submit, scroll to and focus the FIRST error field. Display error summary at top of form listing all errors. Individual field errors displayed inline below each field. Errors persist until corrected — they are NOT dismissed by interaction with other fields.
Rule 6: Data tables must support VIRTUAL SCROLLING for datasets exceeding 100 rows. The Nurse Dashboard, Lab results worklist, and CMOP queue can have hundreds or thousands of rows. DOM rendering all rows simultaneously causes unacceptable performance degradation.
Rule 7: All interactive elements must be keyboard-navigable with a visible focus indicator. Tab order must follow logical reading order (left-to-right, top-to-bottom for LTR languages). Focus trapping in modal dialogs (Tab cycles within the modal, not to background elements). Escape key closes modals and dropdowns.
Rule 8: Real-time data updates (Nurse Dashboard, ICU Flowsheet, Pharmacy queue) must use progressive update — update individual data cells, not full-page refresh. WebSocket for push updates when available, polling (30-second interval) as fallback. Visual indicator (subtle animation) when a cell value changes.
Rule 9: The component library must be VERSIONED with semantic versioning (major.minor.patch). Breaking changes (visual or behavioral) require a major version bump. All workspaces depend on the component library — an uncoordinated change can break the entire product. Component changes go through design review before release.
Rule 10: Print stylesheets are REQUIRED for all clinical documents (progress notes, lab reports, medication lists, tray tickets, prescription labels, surgical reports). The print layout must produce a clean, readable, black-and-white document suitable for the medical record. Headers and footers include patient name, MRN, date, page number, and CONFIDENTIAL marking.
Rule 11: Right-to-left (RTL) support must be built into every component from the start using CSS logical properties (margin-inline-start instead of margin-left, padding-inline-end instead of padding-right). The I18n Framework (Phase 2, Artifact 3) specifies locale configuration. RTL languages (Arabic, Hebrew) are a deployment requirement for international readiness.
Rule 12: The Design System is a LIVING DOCUMENT — it will evolve as the product is built. However, changes to design tokens, component APIs, and clinical patterns must go through a DESIGN REVIEW process with clinical stakeholders. A pattern change that makes sense for engineering may create a patient safety risk in clinical use. Clinical validation of UI changes is mandatory.
