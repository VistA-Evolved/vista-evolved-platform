# VistA Evolved
Wireframe Specifications
WF-11: Admin / Security Workspace
Document 11 of 16 | ~10 Pages | User Management, Security Keys, Role Management, Site Parameters, Division Management, Audit Logs, Alerts/MailMan, TaskMan Monitoring, Reports, Configuration
March 2026 | The Admin/Security workspace manages the VistA infrastructure: user identity and access provisioning (NEW PERSON #200), security key assignment (SECURITY KEY #19.1), menu option management (OPTION #19), electronic signature administration, site-wide parameter configuration (KERNEL SYSTEM PARAMETERS #8989.3), multi-division facility management (INSTITUTION #4, MEDICAL CENTER DIVISION #40.8), device/printer configuration (DEVICE #3.5), TaskMan background job scheduling and monitoring, alert management (ALERT #8992), and comprehensive audit log review. VistA Kernel (XU namespace, Kernel 8.0 updated July/August 2025) provides the underlying security framework. VistA Evolved adds a ROLE MANAGEMENT layer on top of VistA's native security keys and menu trees to simplify user provisioning. This workspace is restricted to: IRM Chief, System Administrators, Application Coordinators (ADPACs), Clinical Informatics, and Privacy Officers. It does NOT display the Patient Banner (facility-level operations, not patient-specific).

Unique Behaviors
INACTIVE ACCOUNT DETECTION: Kernel TM specifies 'Deactivating and Reactivating Users' as a core admin function. Users who have not signed in within a configurable period (default 90 days per VHA Directive 6500) are flagged for investigation — either deactivate (separated employee) or confirm still active (infrequent user like consulting specialist). This is a VA Inspector General audit point. PROVIDER CREDENTIALING FIELDS: for provider-type users, the NPI, DEA#, VA#, Person Class, and Provider Type fields are CRITICAL — they determine what the provider can do in CPRS/VistA Evolved (prescribe medications, sign orders, co-sign trainee orders, etc.). An incorrect Person Class assignment can allow a user to exceed their scope of practice. SECURITY KEY IMPACT PREVIEW: when viewing a user's security keys, hovering over any key shows a tooltip: 'This key grants access to: [list of VistA options and VE workspace pages gated by this key]' — helps admin understand what adding/removing a key will enable/disable. E-SIGNATURE SECURITY: per Kernel Security Tools Manual, electronic signature codes are encrypted and 'indecipherable' even to the highest-access administrators. Admin can CLEAR (reset) a code but cannot VIEW or SET a specific code. This is a fundamental VistA security design principle.

Data Source

VistA Evolved Role definitions (custom configuration).
Unique Behaviors
NEW PERSON #200 REQUIRED FIELDS: Kernel SM Guide specifies required fields for adding a new user: Name, SSN, Access Code, Verify Code (initial), Primary Menu Option, Division. VistA Evolved adds: Role assignment and workspace access. VERIFY CODE LIFECYCLE: per Kernel, verify codes have a validity period (site parameter — typically 90 days for VA per VHA Directive 6500). After expiration, user is forced to create new verify code at next sign-in. Admin cannot see existing verify or access codes (encrypted). COSIGNATURE CHAIN: for trainee providers (residents, students), the Requires Cosignature flag and Cosigner field establish the supervisory relationship. When a trainee signs an order or note, the cosigner receives an alert to review and co-sign. This is a Graduate Medical Education (GME) and patient safety requirement — trainee orders without timely co-signature generate escalation alerts. PERSON CLASS SCOPE ENFORCEMENT: the Person Class/Taxonomy code determines what VistA allows the provider to do. A Nurse Practitioner (363L00000X) may have prescribing authority depending on state law and facility credentialing. A Medical Student (390200000X) cannot independently sign orders. The system enforces these scope limitations at the order entry and note signing level — the admin workspace ensures the Person Class is correctly assigned. DIVISION SWITCHING: multi-division users see all their assigned divisions in the System Bar User Menu 'Switch Division' option (WF-01 Section 1.1). Switching division changes the data context (which patients, which settings apply). Admin must correctly assign ALL applicable divisions for multi-site providers.

Unique Behaviors
KEY DEPENDENCY MAPPING is the most valuable feature of this page. In legacy VistA, understanding 'what does this key do?' requires reading package documentation. VistA Evolved pre-maps all key dependencies: which options, which RPCs, which workspace pages each key gates. This transforms security key management from documentation-dependent guesswork into a navigable dependency graph. Example: admin wants to know 'why can't User X access the Pharmacy verification queue?' — search for pharmacy keys, find PSJ PHARM key, check if user holds it. If not, see that it's part of the 'Staff Pharmacist' role -> assign the role. ORPHANED KEY DETECTION: keys that were manually assigned to users outside of role assignments may indicate ad-hoc provisioning that should be formalized into roles. The 'orphaned keys' filter helps admins clean up and standardize access. PACKAGE NAMESPACE CONVENTION: VistA security keys follow package namespace conventions (OR* for Order Entry, PS* for Pharmacy, LR* for Lab, RA* for Radiology, DG* for Registration, SD* for Scheduling, IB* for Billing, MAG* for Imaging, SR* for Surgery, etc.). Grouping keys by namespace helps admins understand the functional area each key belongs to. KEY AUDIT: every key assignment and removal is audit-logged with: key name, user affected, action (grant/revoke), performed by, date/time, reason (if provided). This supports VHA Directive 6500 requirements for access control documentation.

Unique Behaviors
VISTA EVOLVED ADDITION: VistA natively has NO formal 'role' concept. Users are provisioned by manually assigning individual security keys and menu options — a process that Kernel SM Guide documents across 5 screens. This is error-prone and inconsistent. VistA Evolved's Role Management adds a layer that BUNDLES keys + menus + workspace visibility into named, reusable roles. This is the single biggest usability improvement for system administration. ROLE HIERARCHY: roles can be hierarchical (e.g. 'Charge Nurse' inherits all keys from 'Registered Nurse' plus additional charge-specific keys). Hierarchy defined by parent-role linkage. Modifying the parent role cascades to child roles. PERSON CLASS VALIDATION: when a role with provider-level keys (e.g. ORES for writing orders) is assigned to a user, the system checks that the user's Person Class in #200 supports that scope. If a user classified as 'Medical Student' is assigned an 'Attending Physician' role, system warns: 'User's Person Class (Medical Student) may not support the scope of practice implied by this role. Verify credentialing before proceeding.' ROLE-BASED WORKSPACE VISIBILITY: this is the mechanism by which the Tenant Profile Matrix (Phase 2, Artifact 1) is implemented. The tenant's shape (Full Hospital, Multi-Specialty Clinic, Standalone Pharmacy, etc.) determines which workspaces exist; the user's role determines which of those workspaces they can see. A nurse at a Full Hospital sees: Clinical, Pharmacy (BCMA only), Scheduling. A pharmacist sees: Pharmacy, Clinical (medication-related views). The intersection of tenant shape and user role = the user's Navigation Rail content.

Unique Behaviors
PARAMETER SCOPE: VistA parameters exist at multiple scopes: System (all users, all divisions), Division (per division), Service (per service/section), User (individual user preferences). The parameter editor shows the current SCOPE for each parameter and allows editing at the appropriate level. Some parameters can only be set system-wide; others can be overridden per division or per user. CRITICAL PARAMETER FLAGGING: parameters that affect patient safety or clinical workflow are flagged with a RED warning icon. Examples: 'Order Check Enabled' (if turned off, drug interaction checks stop), 'Auto-Verify Lab Results' (if misconfigured, erroneous results could release without review), 'BCMA Override Allowed' (if too permissive, nurses could bypass barcode verification). Changes to critical parameters require a SECOND APPROVER (a different admin user must confirm the change before it takes effect — two-person integrity). VHA DIRECTIVE 6500 PARAMETERS: several parameters map directly to VHA information security policy requirements: Session Timeout (must be <= 15 minutes), Verify Code Validity Period (must be <= 90 days for VA), Failed Sign-In Attempts Before Lockout (must be <= 5), Password Complexity (must meet VA requirements: 8+ characters, mixed case, number, special character). These parameters have ENFORCED MINIMUMS that cannot be set below the VHA-mandated threshold — the system rejects values that violate policy.

Unique Behaviors
MULTI-DIVISION CONTEXT: when a user is assigned to multiple divisions and switches divisions (via System Bar User Menu), the entire application context changes: different patients may be visible (patients registered at that division), different parameters apply (division-level parameter overrides from AD-05), different workspaces may be available (division-level workspace visibility), different reports scope to the selected division. This is critical for users who work across multiple VA sites (e.g. a specialist who provides telehealth to CBOCs from the main medical center). CBOC SETUP: Community-Based Outpatient Clinics (CBOCs) are typically simpler than the parent medical center. A CBOC may only need: Clinical, Scheduling, Patients, and Billing workspaces — Surgery, Lab (except POCT), Radiology (except basic X-ray), and Supply (handled centrally) may be hidden. The division configuration controls this scope. STATION NUMBER FORMAT: VA uses a specific station numbering system. Main medical centers have 3-digit numbers (e.g. 521). CBOCs have parent station + suffix (e.g. 521A4, 521BY). Nursing homes use 'NH' suffix. The station number is embedded in many VistA data records and must be correctly configured for inter-facility data exchange.

Data Source

Per Audit/Irreversibility Matrix (Phase 2, Artifact 7): all C/L2 (reversible with audit) and C/L3 (irreversible) actions are logged.
Unique Behaviors
PATIENT RECORD ACCESS AUDIT: per HIPAA Privacy Rule and VHA Directive 1605.01, every access to a patient's protected health information (PHI) must be trackable. The audit log captures: who accessed which patient's data, when, and what they viewed/modified. This supports: (1) Patient requests for access history ('who has looked at my record?' — patients have the right to request this per HIPAA), (2) Privacy investigations (suspected unauthorized access — e.g. employee looked up a celebrity patient's record without a care relationship), (3) OIG audits (Inspector General reviews of access patterns). SENSITIVE PATIENT ACCESS: patients flagged as sensitive (VIP, employee, public figure — per Patient Record Flag #26.13/#26.14/#26.15 from PT-11) have ADDITIONAL audit requirements. Accesses without a care relationship (user is not on the patient's care team) are flagged for Privacy Officer review. OVERRIDE AUDIT: the Audit/Irreversibility Matrix defines which actions generate override audit events. Key overrides tracked: drug interaction override (user acknowledged interaction but proceeded with order — logged with reason), BCMA barcode override (nurse administered medication without barcode scan — logged with reason), formulary override (provider ordered non-formulary drug — logged with justification), allergy override (provider ordered medication matching patient's allergy — logged with clinical rationale). Override reports are reviewed by pharmacy, nursing leadership, and safety committees for patterns indicating worrisome behavior. AUDIT RETENTION: per VA records management policy, audit logs are retained for a minimum of 3 years (6 years for some categories). The system does not allow audit log deletion or modification — audit records are IMMUTABLE (I/L3 per Audit Matrix — the audit system itself audits cannot be tampered with). Archival: older audit data archived to long-term storage but remains searchable through the same interface.

Unique Behaviors
TASKMAN CRITICALITY: TaskMan is the heartbeat of VistA background processing. If TaskMan stops, automated processes stop: HL7 messages stop flowing (lab results don't auto-file, ADT notifications don't send), scheduled reports don't generate, CMOP transmissions don't happen. TaskMan status must be monitored continuously. ERROR TRAP MONITORING: VistA M code errors are captured in the error trap file (#3.077). CRITICAL parameters (authentication rules, E-signature rules, audit configuration): TWO-PERSON INTEGRITY — saving generates a pending change request that must be approved by a SECOND admin user before taking effect. This prevents a single compromised admin account from weakening security controls. The pending change shows in the second admin's alert queue with: what is being changed, by whom, old/new values, stated reason.
Data Source
KERNEL SYSTEM PARAMETERS #8989.3 (Kernel security parameters), site-level security configuration, VistA Evolved platform configuration — via ZVE ADMIN CONFIG wrapper.
Unique Behaviors
HIGHEST PRIVILEGE PAGE: requires IRM Chief or equivalent access (XUMGR key + VistA Evolved ADMIN MASTER key). Changes on this page affect the fundamental security posture of the entire system. VHA DIRECTIVE 6500 ENFORCEMENT: the system validates that authentication parameters meet VHA-mandated minimums. If admin attempts to set session timeout to 20 minutes (exceeding the 15-minute VHA maximum), the system REJECTS the value with: 'VHA Directive 6500 requires session timeout <= 15 minutes. Value 20 is not permitted.' These enforced minimums cannot be overridden — they are hard-coded policy constraints. TWO-PERSON INTEGRITY: the most sensitive security configurations (authentication rules, audit settings) require two independent administrators to agree on a change. This is analogous to the two-person integrity requirements for nuclear launch codes — no single individual should be able to weaken security controls unilaterally. This exceeds standard commercial EHR security models and reflects VA's heightened security requirements as a federal healthcare system processing millions of veteran records.
