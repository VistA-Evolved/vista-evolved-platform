Wireframe Specifications
WF-12: Interfaces / Devices Workspace
Document 12 of 16 | ~8 Pages | Interface Dashboard, HL7 Link Management, FHIR Endpoints, Device Connectivity, Message Browser, VistA Imaging/PACS Integration, Reports, Configuration
March 2026 | The Interfaces/Devices workspace provides centralized monitoring and management of all system integrations: VistA HL7 messaging engine (HL namespace, V1.6 and HLO coexisting engines), FHIR R4 API endpoints (per FHIR API Architecture Phase 2 Artifact 9), clinical device connectivity (per Device/Instrument Integration Catalog Phase 2 Artifact 8), VistA Imaging/PACS DICOM interfaces, and external system connections. VistA HL7 is the backbone of inter-system communication: HL LOGICAL LINK #870 defines each connection, HL7 APPLICATION PARAMETER #771 defines sending/receiving applications, PROTOCOL #101 defines event-driven message routing, HL7 MESSAGE TEXT #772 stores messages, HL7 MESSAGE ADMINISTRATION #773 tracks message status. Two HL7 engines coexist: HL 1.6 (legacy, widely deployed) and HLO (Health Level Seven Optimized, newer, HLO* namespace, #779.1/#779.2). Standard ports: 5000 (HL 1.6 multi-listener), 5001 (HLO production), 5026 (HLO test). This workspace is restricted to IRM/Interface Analysts. It does NOT display the Patient Banner. References Vol 10 (Interfaces/Devices) screen contracts.

Unique Behaviors
REAL-TIME MONITORING: dashboard auto-refreshes every 60 seconds (configurable: 30s to 5m). WebSocket preferred for instant status change detection. Filer status is checked via TaskMan task status — VistA HL7 SM Manual documents: filers are background tasks managed by TaskMan. If TaskMan itself is stopped (AD-09 TaskMan Monitor), then HL7 filers cannot run — this is a cascading failure requiring TaskMan restart first. CLINICAL IMPACT ASSESSMENT: each offline link shows an estimated clinical impact statement: e.g. 'Chemistry Analyzer Link offline: ~150 results/hour not auto-filing. Lab techs must enter results manually.' This helps IRM prioritize which links to restore first based on clinical impact. INTERFACE DEPENDENCY MAP: hovering over any link shows: which VistA packages depend on this link (e.g. the Lab instrument link feeds LR package results, which feed CPRS/Clinical workspace results viewer, which feed clinical alerts for critical values — the entire chain is impacted). AUTO-RESTART: for links with auto-restart configured (IF-07 Config), the system attempts automatic restart after a configurable delay. If auto-restart fails after N attempts, it escalates to manual intervention alert.

Layout — Link Detail Panel (right 55%, on link selection)
IDENTIFICATION: Link Name, IEN in #870, Description (free text describing what this link connects to), Connected System (external system name — e.g. 'Beckman AU5800 Chemistry Analyzer Bay 3', 'BD Pyxis MedStation Ward 4A', 'Regional CMOP Dallas'). CONNECTION: Protocol Type, Direction, IP Address, Port, DNS Domain, Timeout Settings (open timeout, read timeout, close timeout — from #870 TCP parameters), Acknowledgement Mode (Original Mode=wait for ACK before next message, Enhanced Mode=track individual message ACKs per SM Manual). APPLICATION MAPPING: Sending Application (from HL7 APPLICATION PARAMETER #771), Receiving Application (#771), Event Driver Protocol (from PROTOCOL #101), Subscriber Protocols (#101 — VistA HL7's event-driven routing: when an event occurs, the event driver protocol triggers subscriber protocols which determine where the message is sent). MESSAGE STATISTICS: messages sent/received per hour (last 24h chart), error count per hour, average response time (ms), throughput (messages/minute). QUEUE STATUS: In Queue current depth + processing rate (messages/second), Out Queue current depth + transmission rate. Queue graphs: depth over last 24 hours.
Layout — Link Actions (bottom of detail panel)
'Start Link' (start the Lower Layer Protocol for this link — equivalent to VistA 'Start/Stop Links' option from Filer and Link Management menu. For TCP server links, starts the listener on the configured port. For TCP client links, initiates outbound connection), 'Stop Link' (stops the LLP — cleanly closes connection, flushes pending messages per configured behavior), 'Ping' (TCP only — equivalent to VistA HL7 'Ping (TCP Only)' option: sends TCP connection test to remote system. Success: 'PING Worked' message per SM Manual. Failure: shows error and troubleshooting suggestions), 'Purge Queues' (clear In/Out queues of old/stuck messages — requires confirmation + audit log. VistA HL7 SM Manual: 'You can view the status of the In and Out queues using the Systems Link Monitor'), 'View Messages' (navigates to IF-05 Message Browser filtered to this link), 'Edit Configuration' (opens IF-03 for this link). FILER MANAGEMENT (separate from individual link management): 'Start All Filers' (starts Incoming Filer STARTIN^HLCSIN + Outgoing Filer STARTOUT^HLCSOUT + Link Manager EN^HLCSLM as TaskMan background tasks), 'Stop All Filers' (WARNING: stops ALL HL7 message processing facility-wide — only for maintenance).
Data Source
HL LOGICAL LINK #870 (all link definitions and status), HL7 APPLICATION PARAMETER #771 (application definitions), PROTOCOL #101 (event/subscriber routing), HL7 MESSAGE TEXT #772 (message storage), HL7 MESSAGE ADMINISTRATION #773 (message tracking and queue status), HLO APPLICATION REGISTRY #779.2 (for HLO links) — all via ZVE HL7 LINK wrapper. Systems Link Monitor data equivalent.
Unique Behaviors
DUAL ENGINE AWARENESS: VistA has TWO coexisting HL7 engines per HLO SM Manual: 'HLO engine will co-exist with the existing HL 1.6 engine. There is virtually no shared code between them.' Some links use HL 1.6 (legacy), others use HLO. The link detail shows which engine handles this link. HLO links use port 5001 (production) or 5026 (test) per HLO documentation. HL 1.6 links typically use port 5000. Queue management is different: for HL 1.6 TCP links, 'the message is transmitted directly from the VistA HL7 message files; the filers never actually copy the message to the Out queue' per SM Manual. For HL 1.6 HLLP/X3.28 links, 'filers place a copy of the message in the link Out queue prior to transmission.' Understanding this difference matters for troubleshooting queue-related issues. PROTOCOL CHAIN VISUALIZATION: clicking 'Event Driver Protocol' in the detail panel shows a visual diagram of the protocol routing chain: Event occurs in VistA application -> Event Driver Protocol #101 triggered -> Subscriber Protocols identified -> Messages generated for each subscriber -> Each subscriber's message routed to the corresponding Logical Link #870 for transmission. This visualization demystifies VistA HL7's publish-subscribe routing architecture for interface analysts. LINK DEPENDENCY: the detail panel shows which VistA packages use this link. Example: the Lab instrument link (connecting a chemistry analyzer) depends on: LR (Lab) package filing auto-instrument results, which triggers OR (Order Entry) notification to ordering provider, which triggers CPRS alert display. If this one link fails, the entire results delivery chain is impacted.

(5) APPLICATION MAPPING: Sending Application (from #771), Receiving Application (#771), Event Driver Protocol (#101 — 'VISTA HL7 acts as an Enterprise Application Integration solution for VistA applications' per SM Manual), Subscriber Protocols (#101 — multiple subscribers can receive the same event).

Test results: GREEN 'Connection successful, round-trip Nms' or RED 'Connection failed: [specific error: Connection Refused / Timeout / DNS Resolution Failed / Authentication Failed]' with troubleshooting suggestions per error type.
Layout — Save & Activate
'Save' (saves configuration but does NOT start the link), 'Save & Start' (saves and immediately starts the LLP per VistA HL7 start procedure), 'Cancel' (discards changes). Data Source

Unique Behaviors
TECHNICAL CONFIGURATION PAGE: requires deep HL7 knowledge. The form provides contextual help for each field (tooltip with VistA HL7 SM Manual references). PORT ALLOCATION: the system maintains a port registry showing which ports are already in use by other links to prevent port conflicts. Port 5000 (HL 1.6 multi-listener) and 5001 (HLO production) are reserved system ports per VistA standards. HL7 VERSION NEGOTIATION: some external systems support only specific HL7 versions. The version configured here determines which message structures VistA uses when communicating with this link. Version mismatch is a common cause of interface failures. PROTOCOL MAPPING WIZARD: for creating new interfaces, a wizard guides the analyst through: (1) identify the clinical data flow (what data needs to move between systems), (2) select or create event driver protocol in #101, (3) select or create subscriber protocols in #101, (4) configure logical link in #870, (5) test end-to-end message flow. This wizard simplifies what the SM Manual documents as a multi-step process across multiple VistA files.

Unique Behaviors
FHIR API is a VistA Evolved ADDITION — native VistA does not have FHIR capabilities (VistA uses HL7 v2.x and RPC-based APIs). VistA Evolved exposes VistA data as FHIR R4 resources, enabling interoperability with modern healthcare applications, patient-facing apps (My HealtheVet, Apple Health, third-party patient portals), clinical decision support tools, and research platforms. US CORE IMPLEMENTATION GUIDE COMPLIANCE: FHIR resources must conform to US Core profiles (mandatory for ONC certification). The resource endpoint table shows US Core compliance status per resource. RATE LIMITING: configurable per client application to prevent abuse. Default: 100 requests/minute per client. Exceeded: HTTP 429 Too Many Requests. Rate limiting protects VistA from being overwhelmed by aggressive API consumers. BULK DATA EXPORT: supports FHIR Bulk Data Access (Backend Services) for population-level data extraction — used by research, quality reporting, and public health. Bulk exports are asynchronous (submitted, processed in background, download when ready). PHI safeguards: bulk exports containing patient data require IRB-level authorization (similar to AD-07 audit requirements). PATIENT CONSENT: SMART on FHIR requires patient consent before third-party apps can access their data. Consent records stored and enforced at the authorization server level. Patients can revoke consent, immediately terminating app access.

(C) ROUTING VIEW: visual diagram showing message routing path: Source Application -> Event Driver Protocol (#101) -> Subscriber Protocol(s) (#101) -> Logical Link(s) (#870) -> Destination(s). Shows where in the routing chain an error occurred (if error status).
Layout — Message Actions (bottom of detail)
'Replay Message' (re-process this message through the HL7 engine — useful for messages that failed due to transient errors like temporary network outage. Creates a new message copy, does not modify original. Re-replayed message audit-logged), 'Forward to Link' (send this message to a different logical link — useful for routing corrections), 'View Related Messages' (shows the original message + its ACK response + any related follow-up messages by conversation ID), 'Export Message' (download raw HL7 text for offline analysis or sharing with vendor), 'Mark as Investigated' (flags error messages as reviewed — removed from error counts on dashboard after investigation).
Data Source
HL7 MESSAGE TEXT #772 (full message content), HL7 MESSAGE ADMINISTRATION #773 (message metadata: status, timestamps, routing info), HL LOGICAL LINK #870 (link identification), PROTOCOL #101 (routing chain), HL7 APPLICATION PARAMETER #771 (application identification) — via ZVE HL7 MESSAGES wrapper.
Unique Behaviors
PARSED VIEW is the KEY DIFFERENTIATOR from legacy VistA HL7 message viewing. Legacy VistA displays raw pipe-delimited messages that are difficult to read. VistA Evolved parses the message into a structured, color-coded, human-readable display with field labels from the HL7 standard. This dramatically reduces troubleshooting time — an analyst can immediately see: patient name, test ordered, result value, error description, without mentally parsing pipe-delimited text. PATIENT SEARCH ACROSS MESSAGES: the ability to search for all HL7 messages related to a specific patient (by parsing PID segments) is critical for: tracing data flow problems ('this patient's lab result never appeared in CPRS — where did the message go?'), privacy investigations ('what data about this patient was transmitted externally?'), and interface debugging ('the admission message for this patient had a malformed PID segment'). ERROR DIAGNOSIS: for messages with error status, the system displays: error code (from ACK response: AE=Application Error, AR=Application Reject, CE=Commit Error, CR=Commit Reject per HL7 standard), error description from the MSA-3 text field of the ACK, and troubleshooting suggestions based on common error patterns (e.g. 'AE: Unknown Patient — verify patient MRN in PID-3 matches a valid entry in PATIENT #2').

CAUTION banner + IRM-only access. GLOBAL IMPACT: changes to HL7 Site Parameters (#869.3) affect ALL HL7 message headers generated by this VistA instance.
