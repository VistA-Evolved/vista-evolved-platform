March 2026 | This document defines the acceptance criteria for every VistA Evolved workflow. The core principle is TERMINAL-EQUIVALENCE: every action performed through the modern web UI must produce the same state in VistA as the equivalent action performed through the VistA terminal (Roll-and-Scroll) or CPRS. This is verified by querying VistA directly via FileMan (D Q^DI), global reads (ZW ^global), or terminal application options (D ^ZU) after each test. VistA is 100% the system of record. If the modern UI shows something different from the terminal, the modern UI is WRONG.

# 1. Verification Philosophy and Framework
## 1.1 Terminal-Equivalence Principle
VistA Evolved is a modern web client layered over VistA. VistA remains 100% the system of record. This means that EVERY write operation performed through the modern UI must create, modify, or update VistA data in exactly the same way that the equivalent terminal or CPRS operation would. The verification method is simple: (1) perform the action in the modern UI, (2) query VistA directly via terminal to confirm the expected state change, (3) perform the same action via terminal/CPRS, (4) confirm the results are identical. If any discrepancy exists, the modern UI wrapper RPC is DEFECTIVE and must be fixed before the feature ships.
For READ operations, terminal-equivalence means: data displayed in the modern UI matches data displayed when querying the same patient/record via terminal FileMan inquiry or CPRS. Field values, date formats, code translations, and sort orders must match.
For WRITE operations, terminal-equivalence means: the VistA file entries created by the wrapper RPC are identical to entries created by terminal options. This includes: correct file and field population, correct cross-reference firing, correct audit trail generation, correct HL7 message generation (if applicable), correct notification/alert generation (if applicable).
