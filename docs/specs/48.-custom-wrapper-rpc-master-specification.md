# 1. RPC Architecture and Registration Pattern
## 1.1 How VistA RPCs Work
Every VistA RPC follows the same architectural pattern established by the RPC Broker (XWB namespace). When a client application (CPRS, VistA Evolved, or any other) needs to execute server-side logic, it sends an RPC request over TCP/IP to the VistA Broker Listener. The Broker Listener: (1) looks up the requested RPC name in the REMOTE PROCEDURE file (#8994), (2) verifies the RPC is allowed in the current application context (the B-type option), (3) executes the M routine at the TAG^ROUTINE entry point specified in #8994, passing the client's parameters, and (4) returns the result to the client.
The RPC Broker Systems Management Guide (May 2020) states: 'The server receives a message from the client and parses out the name of the remote procedure call and its parameters. The Broker module on the server looks up the remote procedure call in the REMOTE PROCEDURE file (#8994), verifies that the RPC is allowed to run in the context of the application, and executes the RPC using the passed-in parameters.'
## 1.2 ZVE Wrapper Registration Requirements
Every ZVE wrapper RPC must be registered in VistA before it can be called. The registration has THREE components:
Component 1: REMOTE PROCEDURE File (#8994) Entry
Each wrapper needs an entry in #8994 with: .01 NAME (the RPC name, e.g., 'ZVE USER CREATE'), .02 TAG (the M label to call, e.g., 'CREATE'), .03 ROUTINE (the M routine name, e.g., 'ZVEUSER'), .04 RETURN VALUE TYPE (1=SINGLE VALUE for simple returns, 2=ARRAY for lists/tables, 3=WORD PROCESSING for large text, 4=GLOBAL ARRAY for very large datasets, 5=GLOBAL INSTANCE for direct global reference). Also: DESCRIPTION (what the RPC does), INPUT PARAMETERS multiple (defines expected parameters by position), and AVAILABILITY (R=Restricted to context option, P=Public/unrestricted).
Component 2: B-Type Context Option in OPTION File (#19)
VistA Evolved uses a set of B-type (Broker Client/Server) options to group RPCs by security domain. Each B-type option has an RPC multiple listing which RPCs are authorized under that context. When the React client connects, it calls CreateContext with the option name. If the user has the option on their menu (PRIMARY MENU #201 or SECONDARY MENU OPTIONS #203 in NEW PERSON #200), access is granted. If not, the Broker rejects all RPC calls under that context.
Component 3: User Menu Assignment
The B-type context option must be assigned to each user's menu tree. This is done during user setup (Vol 1 AD-01/AD-02) by adding the context option to the user's SECONDARY MENU OPTIONS (#203) in NEW PERSON #200. This is how system managers control which users can access which VistA Evolved workspaces.

## 1.4 Standard Wrapper M Routine Pattern
Every ZVE wrapper M routine follows the same code pattern. This pattern MUST be followed for consistency, security, and auditability:
ZVEUSER ; VistA Evolved - User Management Wrapper RPCs
;;1.0;VISTA EVOLVED;**1**;Mar 2026;Build 1
;
; ---- RPC: ZVE USER CREATE ----
CREATE(RESULT,INPUT) ;
; Input: INPUT('NAME'), INPUT('ACCESS'), INPUT('VERIFY'), INPUT('DIVISION'), INPUT('MENU')
; Output: RESULT = JSON string {"status":"ok","duz":123} or {"status":"error","message":"..."}
;
; Step 1: Validate caller has authority
I '$D(DUZ)!('$D(DUZ(2))) S RESULT="{""status"":""error"",""message"":""Not authenticated""}" Q
;
; Step 2: Validate input parameters
I $G(INPUT("NAME"))="" S RESULT="{""status"":""error"",""message"":""Name required""}" Q
;
; Step 3: Execute VistA operation (call existing APIs or FileMan)
; ... (FileMan DBIA-approved calls to create NEW PERSON #200 entry)
;
; Step 4: Audit log the action
D LOG^ZVEAUDIT(DUZ,"USER CREATE",NEWDUZ,"Created user "_$G(INPUT("NAME")))
;
; Step 5: Return structured result
S RESULT="{""status"":""ok"",""duz"":"_NEWDUZ_"}"
Q
This pattern ensures: (A) Caller authentication is verified (DUZ and DUZ(2) must exist), (B) Input validation catches errors before file writes, (C) Existing VistA APIs are called (not direct global SETs that bypass business rules), (D) Every write is audit logged via the ZVE audit utility, (E) Structured JSON is returned for the React frontend to parse.
## 1.5 Return Types and JSON Contracts
All ZVE wrappers return structured JSON. The React API layer parses these responses. Standard envelope:
SUCCESS (single record): {"status":"ok","data":{"field1":"value1","field2":"value2"}}
SUCCESS (list): {"status":"ok","data":[{"ien":1,"name":"..."},{},...],"total":42}
ERROR: {"status":"error","code":"VALIDATION","message":"Human-readable error text"}
WARNING: {"status":"warning","data":{...},"warnings":[{"code":"DUR","message":"..."}]}
Return type in REMOTE PROCEDURE #8994 is set to: SINGLE VALUE (type 1) for most wrappers (JSON fits in a single string return), ARRAY (type 2) for very large result sets that exceed single-string limits, or GLOBAL ARRAY (type 4) for extremely large data dumps (rare, used only for report/extract wrappers).

# 2. Domain-by-Domain Wrapper Catalog
The following sections catalog every ZVE wrapper RPC organized by domain. For each wrapper, the specification includes: RPC Name (as registered in #8994), M Entry Point (TAG^ROUTINE), Input Parameters, Return Contract (JSON structure), VistA Files Touched (read and/or write), Required Security Keys (beyond the context option), Wave Assignment, and Source Page Contract reference.
NAMING CONVENTION: All wrapper RPCs use the ZVE namespace prefix. The second token identifies the domain (ADMIN, SCHED, OP, INP, LAB, RAD, BIL, SUP, SRG, etc.). The third token identifies the action or sub-domain. Example: ZVE OP VERIFY = ZVE (namespace) + OP (Outpatient Pharmacy) + VERIFY (verification action). M routines are named: ZVEdomain (e.g., ZVEADMIN, ZVESCHED, ZVEOP, ZVEINP, ZVELAB, ZVERAD, ZVEBIL, ZVESUP, ZVESRG). Routines are split when exceeding ~2,000 lines: ZVEOP, ZVEOP1, ZVEOP2.
