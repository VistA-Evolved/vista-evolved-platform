# VistA Evolved
API Gateway & RPC Broker Adapter Specification
Frontend-to-Backend Integration Architecture
Node.js API Gateway | RPC Broker TCP/IP Adapter | Authentication & Session Management | Connection Pooling | REST API Routes | TypeScript Interfaces | WebSocket Real-Time Updates | FHIR R4 Server | Error Handling | Print Service | File/Image Handling
March 2026 | This document specifies the MIDDLE LAYER between the React SPA frontend and the VistA M backend. The React frontend (specified in WF-01 through WF-16 and DS-01) calls REST API endpoints on the Node.js API Gateway. The Gateway translates these REST calls into VistA RPC Broker protocol messages, manages TCP/IP connections to VistA, handles authentication/session management, and returns structured JSON to the frontend. The M-side wrapper RPCs (specified in the Wrapper RPC Master Specification) execute on VistA and return results through this pipeline. This document tells the full-stack integration developer exactly what to build. Grounded in: RPC Broker 1.1 Developer's Guide (May 2020), RPC Broker 1.1 Systems Management Guide (May 2020), RPC Broker 1.1 Technical Manual (May 2020), VistALink 1.6 Developer Guide (March 2022), @vistadataproject/vista-client npm package (Node.js RPC client reference), and all VistA Evolved specification documents.

# 1. System Architecture Overview
## 1.1 Three-Tier Architecture
VistA Evolved uses a three-tier architecture. Tier 1 (Presentation): React SPA running in the user's browser. Communicates ONLY with the API Gateway via HTTPS REST calls and WebSocket connections. Never communicates directly with VistA. Tier 2 (Application): Node.js API Gateway running as a server-side process (Docker container). Receives REST requests from the React SPA, translates them into VistA RPC Broker protocol messages, manages TCP/IP connections to VistA, handles authentication/session lifecycle, enforces rate limits and input sanitization, and returns structured JSON responses. Also hosts: FHIR R4 server, WebSocket server for real-time updates, print service, and file/image proxy. Tier 3 (Data): VistA instance (YottaDB/M) with RPC Broker Listener running on a configured TCP/IP port. Receives RPC requests from the API Gateway, executes M routines (both existing VistA RPCs and custom ZVE wrapper RPCs), and returns results. VistA is 100% the system of record. The API Gateway NEVER writes directly to VistA globals. ALL writes go through RPCs.
## 1.2 Data Flow: User Action to VistA and Back
Step 1: User clicks 'Save' on a form in the React SPA (e.g., creating a new patient). Step 2: React component calls an API service function: apiService.patients.register(patientData). Step 3: The API service function sends an HTTPS POST request to the API Gateway: POST /api/v1/patients with JSON body. Step 4: The API Gateway's route handler validates the JWT session token, validates the request body against the TypeScript schema, then calls the RPC Broker Adapter: rpcAdapter.call('ZVE PATIENT REGISTER', params). Step 5: The RPC Broker Adapter retrieves a connection from the connection pool (already authenticated for this user's DUZ), encodes the RPC call into VistA Broker wire protocol format, and sends it over the TCP/IP socket. Step 6: VistA's Broker Listener receives the message, looks up ZVE PATIENT REGISTER in REMOTE PROCEDURE #8994, verifies context authorization, and executes REG^ZVEPAT(RESULT,INPUT). Step 7: The M routine validates inputs, creates the PATIENT #2 record via FileMan, logs the audit entry, and sets RESULT to JSON: {"status":"ok","dfn":12345,"name":"TESTPAT,NEW"}. Step 8: VistA returns the result through the TCP/IP socket. Step 9: The RPC Broker Adapter parses the response, returns it to the route handler. Step 10: The route handler sends HTTP 200 with JSON body to the React SPA. Step 11: The React component receives the response, updates state, and renders the success message.
CRITICAL: The API Gateway is a THIN translation layer. It does NOT contain business logic. All business logic lives in VistA (in the M routines). The Gateway's job is: authenticate, validate inputs (shape only, not clinical rules), translate REST to RPC, translate RPC response to REST, and manage connections. If the Gateway contains business logic, it becomes a second system of record and terminal-equivalence breaks.

# 2. RPC Broker TCP/IP Adapter
## 2.1 VistA RPC Broker Wire Protocol
The RPC Broker uses a custom binary-ish protocol over TCP/IP. The protocol is documented in the RPC Broker Developer's Guide and implemented in the CPRS Delphi client (wsockc.pas), the Java VistALink adapter, and the Node.js @vistadataproject/vista-client library. The key elements are:
Connection Establishment: Client connects to VistA Broker Listener on configured port (e.g., 9210). The Listener forks a new M process (partition) for this connection. This M process persists for the duration of the session (DUZ, DUZ(2), and all user context variables are maintained).
Authentication: Client sends XUS AV CODE RPC with encoded Access Code and Verify Code. VistA authenticates and returns DUZ (user internal entry number). From this point, all RPCs execute in this user's security context.
Context Setting: Client sends XWB CREATE CONTEXT with the B-type option name (e.g., 'ZVE CLINICAL CONTEXT'). VistA verifies the user has this option on their menu. If successful, only RPCs listed in that context option are authorized. Context can be switched during a session.
RPC Execution: Client sends RPC name + parameters encoded in Broker wire format. VistA looks up RPC in #8994, verifies context, executes TAG^ROUTINE, returns result.
Disconnection: Client sends XWB IM HERE (keep-alive) periodically, or #BYE# to disconnect. Disconnection kills the M process/partition.
## 2.2 Node.js RPC Broker Adapter Class
The RPC Broker Adapter is a Node.js class that manages TCP/IP connections to VistA and provides a clean async/await API for the route handlers. It wraps either the @vistadataproject/vista-client library or a custom implementation of the RPC Broker wire protocol.
// File: src/adapters/rpc-broker-adapter.ts

interface RPCConnection {
socket: net.Socket;        // TCP socket to VistA Broker Listener
duz: number;               // Authenticated user's DUZ
duz2: number;              // Division IEN (DUZ(2))
userName: string;           // User's display name
activeContext: string;      // Current B-type context option
lastActivity: Date;         // For idle timeout management
keepAliveTimer: NodeJS.Timer; // XWB IM HERE periodic send
}

interface RPCResult {
status: 'ok' | 'error' | 'warning';
data?: any;                // Parsed JSON from M routine return
warnings?: Array<{code: string, message: string}>;
error?: {code: string, message: string};
}

class RPCBrokerAdapter {
private pool: ConnectionPool;      // Connection pool per user session
private vistaHost: string;          // VistA server hostname/IP
private vistaPort: number;          // Broker Listener port (e.g., 9210)

async authenticate(accessCode: string, verifyCode: string): Promise<RPCConnection>
async setContext(conn: RPCConnection, contextOption: string): Promise<boolean>
async call(conn: RPCConnection, rpcName: string, params: any[]): Promise<RPCResult>
async disconnect(conn: RPCConnection): Promise<void>
}
The adapter handles: TCP socket management, Broker wire protocol encoding/decoding, keep-alive timer (sends XWB IM HERE every 60 seconds to prevent VistA timeout), connection error recovery, and graceful disconnection.
## 2.3 Connection Pool Architecture
Each authenticated user session maintains ONE persistent TCP connection to VistA (one M partition). This connection persists for the duration of the user's browser session. The connection pool is keyed by session ID (JWT token). Pool management:
Aspect
Specification
Rationale
Pool Strategy
One VistA connection per authenticated user session. Connection created at login, destroyed at logout or timeout.
VistA RPC Broker is stateful: DUZ, DUZ(2), and security context are maintained per connection. Sharing connections between users would break VistA's security model.
Maximum Connections
Configurable per tenant. Default: 200 concurrent connections. Corresponds to the Broker Listener's MAXCONN parameter in BROKER SITE PARAMETERS #8994.1.
Prevents overwhelming the VistA instance. Must be coordinated with VistA sysadmin to match Broker Listener capacity.
Idle Timeout
15 minutes of no API activity (matches VHA 6500 session timeout policy). After 14 minutes idle, the API Gateway sends a warning to the React SPA. After 15 minutes, the connection is closed (sends #BYE# to VistA).
Per VHA 6500: 'auto-logoff after 15 minutes of inactivity.' The keep-alive timer (XWB IM HERE) prevents VistA from disconnecting during active sessions, but the Gateway enforces the policy timeout.
Keep-Alive
XWB IM HERE sent every 60 seconds while the session is active (user has made an API call within the last 15 minutes).
Prevents VistA's own inactivity timeout from killing the connection during periods of client-side activity (e.g., user is typing a long note).
Connection Recovery
If the TCP socket drops unexpectedly (network error, VistA restart), the adapter detects the error on the next call attempt, marks the session as expired, and returns a 401 to the React SPA. The SPA redirects to login.
VistA connections are not recoverable after a socket drop because the M partition (with DUZ context) is lost. Re-authentication is required.
Context Switching
When the user navigates between workspaces (e.g., from Clinical to Scheduling), the adapter calls XWB CREATE CONTEXT with the new context option. Only one context can be active at a time per connection.
Per RPC Broker spec: 'CreateContext returns True if the user has the option on their menu, False otherwise.' Context switching is lightweight (no re-authentication).

# 3. Authentication & Session Management
## 3.1 Login Flow
Step 1: User enters Access Code and Verify Code on the React SPA login page. Step 2: React SPA sends POST /api/v1/auth/login with {accessCode, verifyCode} over HTTPS. Step 3: API Gateway creates a new TCP connection to VistA Broker Listener. Step 4: API Gateway sends XUS AV CODE RPC with the encoded credentials. Step 5: VistA authenticates. If successful: returns DUZ, user name, division info. If failed: returns error (bad credentials, account locked, verify code expired). Step 6: On success, the API Gateway: (a) stores the VistA connection in the connection pool keyed by a new session ID, (b) retrieves user details via ORWU USERINFO and XUS GET USER INFO RPCs (provider status, keys held, division, e-sig status), (c) generates a JWT token containing: sessionId, duz, userName, division, provider status (but NOT the Access/Verify codes -- never stored), (d) returns the JWT token to the React SPA as an HTTP-only secure cookie + JSON response with user details. Step 7: React SPA stores user context in state management, renders the App Shell with authorized workspace navigation.
## 3.2 JWT Token Structure
// JWT Payload (signed with server-side secret, never exposed to client JS)
{
"sessionId": "uuid-v4",           // Maps to connection pool entry
"duz": 12345,                      // VistA user IEN (NEW PERSON #200)
"userName": "TESTDOC,ATTENDING",    // Display name
"division": {"ien": 500, "name": "MAIN HOSPITAL", "station": "500"},
"isProvider": true,                 // Has PROVIDER key
"hasOres": true,                    // Has ORES key (can release orders)
"esigReady": true,                  // Has e-sig code set
"contexts": ["ZVE CLINICAL CONTEXT","ZVE SCHEDULING CONTEXT",...],  // Authorized contexts
"iat": 1711756800,                  // Issued at
"exp": 1711760400                   // Expires (1 hour, refreshable)
}
SECURITY: Access Code and Verify Code are NEVER stored anywhere outside VistA. They are used once during login to authenticate with VistA, then discarded. The JWT token maps to a session ID which maps to a live VistA connection. If the VistA connection dies, the JWT becomes invalid (the session ID no longer maps to a live connection). The JWT secret is rotated regularly and stored in environment variables, never in code.
## 3.3 E-Signature Flow
VistA's electronic signature is a SEPARATE credential from Access/Verify codes. It is used for signing orders, notes, and other clinical documents. The e-sig code is entered by the user at the moment of signing (not stored in the browser). Flow: (1) User clicks 'Sign' on an order or note. (2) React SPA shows e-sig modal: 'Enter your electronic signature code.' (3) User types their e-sig code. (4) React SPA sends POST /api/v1/clinical/sign with {documentId, esigCode}. (5) API Gateway sends the signing RPC (e.g., TIU SIGN RECORD or ORWOR TSALL) with the e-sig code as a parameter. (6) VistA validates the e-sig code against the hash in NEW PERSON #200 field 20.4. (7) If valid: document signed, status updated. If invalid: error returned, user prompted to retry. (8) The e-sig code is NEVER logged, cached, or stored by the API Gateway.

# 6. WebSocket Real-Time Update Service
Certain clinical workflows require real-time updates without page refresh: new orders appearing in the pharmacy queue, critical lab results triggering nurse dashboard alerts, patient check-in updating the scheduling calendar, bed status changes on the census board. The API Gateway runs a WebSocket server alongside the REST API.
## 6.1 WebSocket Architecture
Aspect
Specification
Protocol
WebSocket (WSS over TLS) on the same port as the REST API (shared HTTP/WS server via Node.js http/ws modules)
Authentication
WebSocket connection established with JWT token in the initial handshake (sent as query parameter or in the first message). Token validated against the same session store as REST API.
Channels
Channel-based subscriptions. Client subscribes to channels by sending a JSON message: {type:'subscribe', channels:['pharmacy:verification_queue', 'nursing:ward_3b', 'scheduling:clinic_456']}. Server pushes events to subscribed channels.
Event Sources
Option A (Polling): API Gateway periodically calls read-only RPCs (every 10-30 seconds configurable) and compares with cached state. If changed, pushes delta to subscribers. Option B (VistA Hooks): Custom ZVE NOTIFY M routine hooked into VistA's event system (protocol-level hooks, bulletin triggers) that pushes notifications to a shared queue read by the Gateway. Option A is simpler and used initially; Option B is a Wave 2+ optimization.
Event Format

## 10.2 Document Upload (Scanned Documents)
Users can upload scanned documents (signed advance directives, insurance cards, external records) to VistA Imaging via the API Gateway. Flow: (1) User selects file in React SPA. (2) React SPA sends multipart/form-data POST to /api/v1/imaging/upload with file + metadata (patient DFN, document type, description). (3) API Gateway validates file type (PDF/TIF/JPEG only) and size (<50MB), stores file temporarily, calls ZVE imaging wrapper RPC to register the document in VistA Imaging MAG files and move the file to the VistA Imaging file server. (4) VistA Imaging entry created, linked to patient record. (5) Scanned document appears in patient's chart under Documents tab.

## 11.2 Multi-Tenant SaaS Deployment
In the SaaS model, each tenant (hospital/clinic) has their own VistA instance. The platform runs one API Gateway per tenant, each pointed at that tenant's VistA instance. Tenant isolation is ABSOLUTE: each tenant's API Gateway connects only to their VistA, and there is no data sharing between tenants. Tenant routing is handled by subdomain (hospital-a.vistaevolved.com -> gateway-a -> vista-a; hospital-b.vistaevolved.com -> gateway-b -> vista-b) or by API key in the request header.
Alternatively, a SHARED API Gateway can serve multiple tenants by routing based on a tenant identifier in the JWT token. Each user's JWT includes their tenant ID and VistA connection info. The connection pool is partitioned by tenant. This reduces infrastructure cost but requires careful isolation validation.
## 11.3 On-Premise Deployment
For hospitals that run their own VistA instance on-premise, the API Gateway and React SPA can be deployed as Docker containers on the hospital's network. The Docker Compose file includes: the gateway container, the SPA container (nginx), and optionally a reverse proxy (Traefik or nginx) for TLS termination and routing. The hospital's existing VistA instance is NOT containerized by VistA Evolved. The gateway container is configured to connect to the hospital's existing VistA Broker Listener port.

# 13. Security Architecture
Security Layer
Implementation
Verification
Transport Encryption
TLS 1.2+ (HTTPS) for all client-to-gateway traffic. No plaintext HTTP in production. VistA Broker traffic between Gateway and VistA is on a private network (Docker bridge or hospital LAN). If crossing network boundaries, SSH tunnel wraps Broker TCP.
SEC-001: Verify no HTTP endpoints accessible. Verify TLS certificate valid.
Authentication
VistA Access/Verify codes validated by VistA Kernel (XUS AV CODE). JWT token for session management. HTTP-only secure cookies prevent XSS token theft. Credentials never stored outside VistA.
SEC-002: Verify Access/Verify codes not logged. Verify JWT is HTTP-only secure cookie.
Authorization (Workspace)
B-type context options per domain (14 context options per Wrapper RPC Master Spec Section 1.3). User must have option on their VistA menu to access workspace RPCs.
SEC-003: Attempt to call RPC outside user's context. Verify 403 returned.
Authorization (Page)
Security keys required per page (per AUX-06 Security Matrix). Gateway checks JWT for key indicators before forwarding requests. VistA-side wrapper also checks keys (defense in depth).
SEC-004: Attempt pharmacy route without PSO PHARMACIST key. Verify denied.
Input Sanitization
All request bodies validated against Zod/Joi schemas before RPC call. No raw user input passed to VistA without validation. SQL injection N/A (VistA uses M globals, not SQL). XSS prevention via JSON-only responses (no HTML generation from user input).
SEC-005: Send malformed input. Verify 400 error, not a VistA crash.
E-Signature
E-sig codes transmitted only during signing actions, validated by VistA Kernel, never stored/cached/logged by Gateway. Failed e-sig attempts tracked.
SEC-006: Verify e-sig code not in any log file after signing.
Audit Logging
All write operations logged Gateway-side (timestamp, user DUZ, action, patient DFN, IP address) AND VistA-side (ZVE AUDIT utility). Dual audit trail ensures accountability even if one system's logs are compromised.
SEC-007: Perform 10 write operations. Verify all 10 appear in both Gateway log and VistA audit.
Session Management
15-minute idle timeout (VHA 6500). JWT expiration with refresh. Concurrent session limits configurable per user. Session list visible in admin workspace (AD-07).
SEC-008: Leave session idle 16 minutes. Verify forced logout.
CORS
API Gateway sets CORS headers to allow only the known SPA origin(s). No wildcard CORS in production.
SEC-009: Attempt API call from unauthorized origin. Verify blocked by CORS.
Rate Limiting
Token bucket rate limiter: 600 requests/minute per session (configurable). Prevents brute force and accidental infinite loops.
SEC-010: Send 700 requests in 60 seconds. Verify 429 returned after limit.
