# VistA Evolved
Tenant Onboarding & VistA Configuration Guide
Step-by-Step Deployment and Setup Procedures
VistA Prerequisites | KIDS Build Installation | Broker Listener Configuration | Context Options | First Admin User Setup | API Gateway Deployment | React SPA Deployment | Tenant Profile Configuration | Workspace Visibility | Go-Live Verification Checklist
March 2026 | This document provides the complete step-by-step procedure for onboarding a new tenant to VistA Evolved. A 'tenant' is a hospital, clinic, or healthcare facility with its own VistA instance. The onboarding process takes an existing VistA instance (or a newly installed one) and configures it to work with VistA Evolved's modern web client. The guide covers: VistA prerequisites verification, ZVE KIDS build installation, RPC Broker Listener setup, B-type context option creation, first admin user setup with VistA Evolved access, API Gateway deployment and connection, React SPA deployment, tenant profile configuration (per AUX-01 Tenant Profile Matrix), workspace visibility settings, and go-live verification. Target audience: VistA System Administrators, DevOps Engineers, and Implementation Managers. This guide assumes familiarity with VistA terminal operations, KIDS installations, and Linux/Docker system administration. Grounded in: KIDS User Guide (Kernel 8.0), RPC Broker 1.1 Systems Management Guide (May 2020), VistALink 1.6 Developer Guide, hardhats.org VistA installation documentation, Kernel 8.0 Systems Management Guide (August 2025), and all VistA Evolved specification documents.

# 1. Onboarding Overview
## 1.1 What Onboarding Means
Onboarding a new tenant to VistA Evolved means: taking an existing VistA instance and layering VistA Evolved's modern web client on top of it. VistA remains 100% the system of record. The onboarding process does NOT replace VistA. It adds: (A) custom ZVE wrapper RPCs to the VistA instance (installed via KIDS), (B) B-type context options to VistA's OPTION file #19 (controlling access to the wrappers), (C) an API Gateway server connecting the React SPA to VistA's RPC Broker, and (D) a React SPA web application that users access through their browser. After onboarding, users can access VistA through THREE interfaces: (1) the traditional terminal (Roll-and-Scroll via SSH), (2) CPRS (if installed), and (3) VistA Evolved (modern web client). All three read/write the same VistA data. Terminal and CPRS continue to work exactly as before.

# 3. Phase B: ZVE KIDS Build Installation
## 3.1 Obtaining the KIDS Build File
The ZVE KIDS transport file is generated from the vista-evolved-vista-distro GitHub repository. Each wave has its own KIDS build: ZVE_1_0_P1.KID (Wave 1), ZVE_1_0_P2.KID (Wave 2), ZVE_1_0_P3.KID (Wave 3), ZVE_1_0_P4.KID (Wave 4). Download the appropriate KIDS file for the wave being deployed and transfer it to the VistA server's file system.
IMPORTANT: On GT.M/YottaDB Linux systems, run dos2unix on the KIDS file before loading to remove Windows carriage returns that cause parsing errors: dos2unix ZVE_1_0_P1.KID
## 3.2 Loading and Installing the KIDS Build
The installation uses VistA's standard KIDS utility. You must be logged in as a user with the XUPROG (programmer) security key.

$ source env.vista    (or equivalent environment setup for YottaDB/GT.M)
$ mumps -dir          (enter direct/programmer mode)

# 4. Alternative: Direct M Routine Load (Development/Testing)
For development and testing environments where KIDS infrastructure may not be fully set up, ZVE routines can be loaded directly. This method is NOT recommended for production.
Step 1: Copy M routine files to VistA's routine directory
$ cp /path/to/vista-evolved-vista-distro/routines/*.m /var/db/vista/r/
Step 2: Compile routines (GT.M/YottaDB)
$ cd /var/db/vista/o
$ for r in ../r/ZVE*.m; do mumps $r; done
Step 3: Register RPCs manually (from programmer mode)
GTM> D ^XUP
(Use FileMan to create entries in REMOTE PROCEDURE #8994 and OPTION #19)
(Or run the registration script: D REGS^ZVEINST)
WARNING: Direct routine load bypasses KIDS dependency checking, version tracking, and rollback capability. Use ONLY for development. Production deployments MUST use KIDS builds.

# 6. Phase D: RPC Broker Listener Verification
The RPC Broker Listener must be running and accessible on the network for the API Gateway to connect. The Listener is a background M process that accepts TCP/IP connections from client applications.
## 6.1 Verify Listener Is Running
Step 1: Check from VistA
GTM> D ^ZU (log in as system manager)
Select OPTION NAME: XWB MENU
RPC Broker Management
Select RPC Broker Management Option: Broker Listener Status
(Shows: port, status, PID, start time)
Step 2: Check from Linux shell
$ ss -tlnp | grep 9210
LISTEN  0  128  0.0.0.0:9210  0.0.0.0:*  users:(("mumps",pid=12345,fd=3))
(If no output: Listener is NOT running. Start it.)
## 6.2 Start Listener If Not Running
GTM> D STRT^XWBTCP(9210)
(Starts a Broker Listener on port 9210)
(Or from XWB MENU: Start Broker Listener)
NOTE: To auto-start the Listener when VistA starts, add D STRT^XWBTCP(9210) to your VistA startup script (the init.d or systemd service file). This ensures the Listener is always available after server reboots.
## 6.3 Test Connectivity From API Gateway Host
$ telnet [vista_host] 9210
Trying [vista_host]...
Connected to [vista_host].
Escape character is '^]'.
(If 'Connection refused': check firewall rules, Listener status, and port number)

Or use the RPC test utility:
$ node test-rpc-connection.js --host [vista_host] --port 9210 --access [code] --verify [code]
Connected: DUZ=12345, User=VEADMIN,SYSTEM, Division=MAIN HOSPITAL
Context set: ZVE ADMIN CONTEXT = OK
RPC test: ZVE USER LIST = OK (returned 1 users)
All tests passed.

# 7. Phase E: API Gateway Deployment
## 7.1 Docker Deployment
Step 1: Pull the API Gateway image
$ docker pull vistaevolved/gateway:1.0.0
(Or build from source: cd vista-evolved-platform && docker build -t gateway .)
Step 2: Create environment file
$ cat > /etc/vistaevolved/gateway.env << EOF
VISTA_HOST=10.0.1.50          # IP of VistA server
VISTA_PORT=9210               # Broker Listener port
JWT_SECRET=$(openssl rand -hex 32)  # Generate random 256-bit secret
SESSION_TIMEOUT_MINUTES=15    # Per VHA 6500
MAX_CONNECTIONS=200           # Match Broker MAXCONN
NODE_ENV=production
LOG_LEVEL=info
TLS_CERT_PATH=/certs/tls.crt
TLS_KEY_PATH=/certs/tls.key
EOF
Step 3: Run the container
$ docker run -d \
--name vista-evolved-gateway \
--env-file /etc/vistaevolved/gateway.env \
-v /etc/vistaevolved/certs:/certs:ro \
-v /var/log/vistaevolved:/var/log/vistaevolved \
-p 443:443 \
--restart unless-stopped \
vistaevolved/gateway:1.0.0
Step 4: Verify Gateway health
$ curl -k https://localhost:443/api/v1/health
{"status":"ok","vistaConnected":true,"activeConnections":0,"version":"1.0.0"}
If vistaConnected is false: check VISTA_HOST and VISTA_PORT, check firewall rules, verify Broker Listener is running.

## 9.2 Workspace Visibility Configuration
After selecting the tenant profile, the admin configures workspace visibility through Admin > Site Parameters > Workspace Visibility. Each workspace can be set to: VISIBLE (appears in navigation for all authorized users), HIDDEN (does not appear in navigation, inaccessible), or RESTRICTED (appears only for users with specific roles/keys). The tenant profile provides defaults, but the admin can customize. The workspace visibility configuration is stored in VistA Evolved's VE configuration parameters (VE-specific, not native VistA parameters).

INITIAL CONFIGURATION REALITY: For many site-specific parameters (especially clinical, pharmacy, and lab configuration), the initial setup will be done through VistA terminal tools, not through VistA Evolved's modern UI. As VistA Evolved matures through waves, more configuration will move to the modern UI. But initial tenant setup will likely involve terminal work for the first few deployments.

# 13. Troubleshooting
Symptom
Likely Cause
Resolution
Login fails: 'Connection to server lost'
API Gateway cannot reach VistA Broker Listener
Check: VISTA_HOST and VISTA_PORT in gateway.env. Verify Listener running (ss -tlnp | grep [port]). Check firewall rules between Gateway and VistA server.
Login fails: 'Invalid access or verify code'
Wrong credentials or VistA user account issue
Verify: user exists in NEW PERSON #200 with active Access/Verify codes. Check DISUSER flag (field 4) is not set. Try logging in via terminal (D ^ZU) to isolate whether issue is VistA-side or Gateway-side.
Login succeeds but no workspaces visible
User missing ZVE context options on SECONDARY MENU OPTIONS (#203)
Assign ZVE context options to user's SECONDARY MENU OPTIONS in NEW PERSON #200 field 203. Rebuild menus if needed (D REINDEX^XQOR).
Workspace loads but 'Insufficient permissions' on actions
User missing required security key for that action
Check user's KEYS (field 51 in #200). Compare with AUX-06 Security Matrix. Assign missing key.
RPC call returns error: 'Context option not found'
The ZVE context option was not created during KIDS install, or the user doesn't have it
Verify context option exists: D Q^DI, File 19, search for ZVE [domain] CONTEXT. If missing: re-run KIDS install. If exists: check user's secondary menu options.
Slow page loads (>5 seconds)
VistA performance issue, connection pool exhaustion, or network latency
Check: RPC response times in Gateway logs. Check VistA system load (D ^%SS or equivalent). Check connection pool utilization. Check network latency between Gateway and VistA.
Data visible in terminal but not in modern UI
Wrapper RPC not returning expected data, or data mapping issue
Call wrapper RPC directly via test utility with same patient DFN. Compare output with terminal data. Check wrapper M routine for data retrieval logic.
Note/order created in modern UI not visible in CPRS
RPC created data correctly but CPRS cache not refreshed, or data written to wrong file/field
Verify data in VistA via terminal (D Q^DI on TIU #8925 for notes, ORDERS #100 for orders). If data is correct in VistA: CPRS refresh issue (close and reopen CPRS). If data is wrong: wrapper RPC bug.
BCMA barcode scan not working
Barcode scanner not sending correct data format, or scan RPC failing
Verify: scanner configured to send data followed by Enter key. Test: type barcode value manually. Check ZVE BCMA SCAN RPC return for error details.
KIDS install fails: 'required build not found'
Attempting to install Wave 2+ without prior wave installed
Install waves in order: ZVE*1.0*1 (Wave 1) must be installed before ZVE*1.0*2 (Wave 2). Check installation history via KIDS Utilities > Display Install History.
