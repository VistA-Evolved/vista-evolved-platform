# 1. Foundation
The Clinical Workspace is the modern equivalent of CPRS — the clinician's primary patient-chart interface. It is built entirely on EXISTING RPCs in the OR, TIU, GMRC, GMV, ORQQAL, ORQQPL, ORWPS, ORWLRR, and ORWRA namespaces, all registered under the OR CPRS GUI CHART B-type context option.
ZERO custom wrapper RPCs are needed for the entire core clinical workspace.
The AI coding team's primary reference for implementation is the CPRS Delphi source code, where every r*.pas file contains the exact RPC call names, parameter structures, and return value parsing.
Source: Context option: OR CPRS GUI CHART (B-type, OPTION #19). This option contains the RPC multiple listing every clinical RPC the CPRS GUI calls. Users must have this option in their menu tree. Source: CPRS Setup Guide July 2025 + VistApedia OR CPRS GUI CHART option entry listing 2000+ registered RPCs.

#8925 signature status changes
Sign Note
ELECTRONIC SIGNATURE RULE: When signing, the user enters their electronic signature code in a password-style input field. The client MUST hash this code using the Encrypt() function from XWBHash.pas BEFORE sending it to VistA via TIU SIGN RECORD. The rCore.pas source code confirms: 'function ValidESCode(const ACode: string): Boolean; ... CallVistA(ORWU VALIDSIG, [Encrypt(ACode)], x)'. The client NEVER transmits or stores the plain-text signature code.
### Create Note Backend Action
Step 1: Call TIU CREATE RECORD with DFN, TITLE IEN, VDT, VLOC. Returns new document IEN.
Step 2: Call TIU SET DOCUMENT TEXT with document IEN and text body array.
Step 3 (if signing now): Call TIU SIGN RECORD with document IEN and Encrypt(ESCode).
On success: Document saved (unsigned or signed). If unsigned, document appears in user's unsigned document list and in the notes list with Unsigned status.
On failure: Show structured error. Common failures: title not found, visit creation failed, location invalid, user not authorized for this title, text exceeds maximum length.

# 5. Custom Wrapper RPC Summary
NONE REQUIRED for the entire core clinical workspace.
All 17 pages in this volume use only existing RPCs registered in OR CPRS GUI CHART. The entire CPRS clinical workflow — cover sheet, problems, allergies, medications, orders, notes, consults, lab results, radiology reports, vitals, encounter entry — is served by existing RPCs that have been in production use by CPRS since its initial deployment.
The AI coding team's job for this workspace is to: (1) connect to VistA via RPC Broker, (2) create context using OR CPRS GUI CHART, and (3) call these existing RPCs with the correct parameters as documented in the CPRS Delphi source code.
The ONLY area where custom wrappers may eventually be needed is for CPRS configuration management (order menus, notification rules, template management, parameter editing) — which is an admin-side concern, not a clinical-side concern, and is deferred to later waves.

# 6. Rules for Design and Engineering
Rule 1: Electronic signature codes must ALWAYS be hashed via the Encrypt() function (from XWBHash.pas) before transmission. Never send or store plain-text ESCode. This applies to note signing (TIU SIGN RECORD), order signing (ORWOR TSALL), and signature validation (ORWU VALIDSIG).
Rule 2: Order checks returned by VistA during ORWDX SAVE MUST be displayed to the user. Never suppress drug interaction, allergy, or duplicate order warnings.
Rule 3: Unsigned notes and orders must be visually prominent with amber/warning indicators. Clinicians must be reminded of their unsigned items.
Rule 4: Note deletion is only possible for the author, only while the note is unsigned, and only if no addenda have been added. These rules are enforced server-side by TIU AUTHORIZATION, but the UI must also enforce them by hiding the Delete button when conditions are not met.
Rule 5: The cover sheet must use progressive loading (ORWCV START/POLL pattern). Panels load independently. Never block the entire page.
Rule 6: Lab critical values (flagged * or HH/LL) must be displayed with maximum visual prominence.
Rule 7: Allergy NKA (No Known Allergies) is different from no-assessment-on-file. Preserve this distinction from Volume 3.
Rule 8: Order dialogs are dynamically defined by ORDER DIALOG #101.41. The modern client must render them from the dialog definition RPC, not hard-code form layouts.
Rule 9: Problem list operations (add, edit, inactivate, remove) require appropriate clinical roles. Non-providers should have read-only access.
Rule 10: Consult result notes must be linked to the consult request (#123) so they appear in the consult activity trail.
Rule 11: All RPCs in this workspace run under OR CPRS GUI CHART context. The user must have this option in their menu tree or all clinical RPCs will fail.
Rule 12: The Delphi source files at github.com/WorldVistA/VistA/tree/master/Packages/Order Entry Results Reporting/CPRS/CPRS-Chart are the definitive reference for RPC call patterns. When uncertain about parameter structure or return format, READ THE SOURCE.
