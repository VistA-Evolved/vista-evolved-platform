# SECTION A: Clinical Reminders (PXRM)
## A.1 Foundation
Clinical Reminders is a decision-support system that alerts clinicians when preventive care, screenings, or health maintenance actions are due for a patient. It is used DAILY by every clinician and drives quality metrics across the organization. Reminders evaluate patient data against configurable clinical rules and present due/not-due status on the CPRS Cover Sheet and in Health Summary components.
Why this was missing from Volume 4: Volume 4 covered CPRS chart-side clinical work (Problems, Notes, Orders, Lab Results, Vitals, Consults ordering) but did NOT cover the reminder evaluation engine that powers the Cover Sheet preventive care display, the reminder dialog system that captures structured clinical data through CPRS, or the reminder configuration and management tools.

# SECTION B: Consult Fulfillment (GMRC Receiving Side)
## B.1 Foundation
Volume 4 covered ORDERING a consult from the clinician side using ORQQCN RPCs through CPRS. This supplement covers the RECEIVING and COMPLETING side — what happens after the consult is ordered. The receiving service must: see incoming consults, accept or deny them, assign a provider, schedule the consult appointment, complete the consult with a result note, and return the result to the requesting clinician.
This is NOT the same as ordering. The consult lifecycle has two halves:
(a) ORDERING HALF (covered in Volume 4): Requesting clinician selects service, enters clinical question, attaches provisional diagnosis, submits consult order. Order goes into ORDER #100 and REQUEST/CONSULTATION #123.
(b) FULFILLMENT HALF (this supplement): Receiving service sees new consult, triages it, assigns provider, schedules patient, performs consultation, writes result note (TIU), completes consult, result flows back to requestor.

# SECTION C: Patient Care Encounter (PCE)
## C.1 Foundation
Patient Care Encounter (PCE) captures the clinical data generated during each patient visit: diagnoses (ICD), procedures (CPT), providers who participated, immunizations administered, skin tests performed, education topics discussed, and health factors documented. PCE data feeds BOTH clinical systems (Clinical Reminders use PCE data as findings) and billing systems (diagnoses and procedures from PCE drive charge capture in Integrated Billing).
PCE is the DATA PIPELINE between clinical documentation and billing. Without properly captured encounter data: clinical reminders cannot evaluate procedure/diagnosis history, billing cannot generate accurate charges, quality metrics cannot be calculated, and workload reporting is incomplete.

# 5. Rules for Design and Engineering
Rule 1 (Reminders): Clinical Reminders are evaluated AUTOMATICALLY when a clinician opens a patient chart. The evaluation engine runs against the patient's data and returns DUE/NOT DUE/N/A for each applicable reminder. The modern UI must trigger this evaluation and display results without manual intervention.
Rule 2 (Reminders): Reminder DIALOGS are NOT just forms — they generate structured clinical data (health factors, diagnoses, procedures) AND progress note text simultaneously. The modern dialog renderer must support both data capture and text generation in one interaction.
Rule 3 (Reminders): National reminders are distributed via PXRM patches and must NOT be edited locally. Local copies can be created and customized. Manager's Manual: 'Nationally distributed entry names are prefixed with VA-. Local entry names cannot start with VA-.'
Rule 4 (Reminders): Reminder evaluation can be automatically disabled during index rebuilds or manually disabled via PXRM DISABLE/ENABLE EVALUATION option. The modern UI must handle disabled-evaluation state gracefully (show message, not error).
Rule 5 (Consults): The consult lifecycle has TWO HALVES: ordering (covered in Volume 4) and fulfillment (this supplement). The modern UI must support BOTH views: the requesting clinician's view of their outgoing consults AND the receiving service's queue of incoming consults.
Rule 6 (Consults): REQUEST SERVICES #123.5 is hierarchical (ALL SERVICES at top). Services should NEVER be deleted — only disabled. GMRC TM: 'Due to tight relationship between CPRS orderable items and this file, a service should NEVER be deleted.'
Rule 7 (Consults): Consult completion requires a TIU result note signed by the consulting provider. The result flows back to the requesting clinician via notification. Without the result note, the consult is not clinically useful.
Rule 8 (Consults): Auto-forwarding routes consults to the correct sub-service automatically based on DST (Decision Support Tool) logic. GMRC*3*139 added this capability. The modern UI should support both manual forwarding and auto-forwarding.
Rule 9 (Consults): Inter-Facility Consults (IFC) use HL7 messaging between VistA sites. Error handling is critical — patient-not-found errors (#201, #202, #203, #205) must be surfaced to IFC coordinators.
Rule 10 (PCE): PCE encounter data feeds BOTH clinical reminders (health factors, diagnoses, procedures are reminder findings) AND billing (diagnoses and CPT codes drive charge capture). Incomplete PCE data causes both quality metric gaps and billing revenue loss.
Rule 11 (PCE): Health factors are the primary structured data type for Clinical Reminder resolution. When a reminder dialog captures a health factor (e.g. 'TOBACCO NON-USER'), that health factor is filed in V HEALTH FACTORS #9000010.16 and satisfies the reminder's resolution logic.
Rule 12 (PCE): Immunization tracking requires: vaccine name, CVX code, date administered, lot number, manufacturer, site of administration, route, administering provider. Series tracking (e.g. 3-dose Hepatitis B) must calculate which dose in the series this is and when the next dose is due.
Rule 13 (PCE): INTERNATIONAL ADAPTATION: CPT codes are US-specific (AMA copyright). For international deployment, the procedure coding system must be configurable: OPCS-4 (UK), MBS (Australia), OPS (Germany), or local equivalents. ICD-10 diagnosis coding is more universal but has country-specific modifications (CM, AM, GM). The PCE framework (visit + diagnoses + procedures + providers) is universal — only the code sets change.
Rule 14: All three packages interact with each other: Clinical Reminders evaluate PCE data (health factors, diagnoses, procedures, immunizations) to determine DUE status. Consult completion generates PCE encounter data. Reminder dialogs can place consult orders. This three-way integration must be preserved in the modern UI.
