CP is a SEPARATE GUI application interfacing with VistA via Consults + TIU + VistA Imaging. # 1. Architecture: Two Systems, One Workflow
## 1.1 Clinical Procedures (CP) V1.0 — The Modern Layer
Clinical Procedures V1.0 (MD namespace) is the modern framework for managing non-surgical clinical procedure results. Like VBECS for Blood Bank, CP is a SEPARATE GUI application — not a traditional VistA M terminal application. CP TM: 'The patient test result or report is displayed through CPRS. Report data is stored on the Imaging RAID and in some instances, discrete data is stored in the Medicine files.'
CP has three client components: (1) CP Gateway — receives HL7 data from vendor instruments (ECG machines, PFT devices, endoscopy systems, bedside monitors). The Gateway translates vendor-specific HL7 into VistA-compatible format. CP TM: 'CP Gateway system allows third party vendor devices to send observational data to a VistA CP system.' (2) CP Console — builds and manages clinical flowsheets for ICU data display. CP TM: 'CP Console application provides tools to build flowsheets used in the ICU for patient care, recording vital statistics.' (3) CP User — the clinician application to match instrument results to procedure orders, submit results to VistA Imaging, and create TIU documents for interpretation. CP UM: 'User application to match instrument results to the procedure order. Then the clinician submits results to Imaging and creates the TIU document.'
CRITICAL ARCHITECTURAL POINT: CP procedure results flow through CONSULTS workflow. A provider orders a procedure via CPRS (Consult order). The procedure is performed and results captured by CP. The interpreting physician creates a TIU document (interpretation report) linked to the consult. The consult is completed when the TIU document is signed. This means CP integrates with: Consults (#123), TIU (#8925), VistA Imaging, and the Ordering system (#100).
## 1.2 Medicine V2.3 — The Legacy Layer
Medicine V2.3 (MC namespace) is the older VistA package that stores discrete procedure data in Medicine-specific files (690-series). CP was designed to augment and eventually replace Medicine, but many sites still have historical data in Medicine files. CP TM: 'CP Conversion File #703.9 — stores status of conversion process for each converted Medicine report.' The modern product must read data from BOTH CP (current) and Medicine (historical) sources.

Performance optimization is critical — lazy loading, virtual scrolling, efficient data retrieval. The flowsheet must support zoom (view 4 hours vs 12 hours vs 24 hours), scroll (horizontal time scroll), and print (for shift-end documentation). Mobile/tablet access should provide a simplified 'quick glance' view with the ability to drill into full detail.

# 6. Rules for Design and Engineering
Rule 1: Clinical Procedures (CP) is a SEPARATE GUI application — like VBECS for Blood Bank. VistA Evolved integrates WITH CP rather than replacing it for instrument data capture. The modern UI handles ordering (MP-01), worklist management (MP-02), interpretation (MP-04), and result viewing (MP-05 through MP-09). CP Gateway and CP User handle the instrument interface layer. Over time, the modern product may absorb CP User functionality, but CP Gateway's HL7 instrument interface should be preserved.
Rule 2: Medicine procedure results flow through the CONSULTS workflow. The lifecycle is: Provider orders procedure (Consult) -> Procedure scheduled -> Procedure performed (CP captures data) -> Interpreter creates TIU report -> Interpreter signs report -> Consult completed. This means the medicine procedure workspace must integrate tightly with the Consults system documented in Vol 4 Supplement (Clinical Reminders + Consults + PCE).
Rule 3: ECG interpretation is the HIGHEST-VOLUME medicine procedure. A large hospital may perform hundreds of ECGs daily. The ECG viewer (MP-05) must support rapid serial review: view ECG, dictate interpretation, sign, next ECG. 'Over-read' workflows where a cardiologist reviews ECGs initially interpreted by non-cardiologists must be supported. Historical ECG comparison (showing current vs prior side-by-side) is essential for detecting changes.
Rule 4: GI Endoscopy quality metrics are REGULATORY requirements (CMS, professional societies). Colonoscopy quality indicators include: adenoma detection rate (ADR — minimum 25% for screening colonoscopies), cecal intubation rate (minimum 95%), bowel prep adequacy, withdrawal time (minimum 6 minutes), and complication rates. The endoscopy report (MP-08) must capture structured data for these quality metrics.
Rule 5: Dialysis documentation has UNIQUE regulatory requirements. CMS regulates End Stage Renal Disease (ESRD) facilities. Required forms: CMS-2728 (Medical Evidence Report), CMS-2746 (Death Notification), Monthly ESRD QIP reporting (quality incentive program: Kt/V, hemoglobin, vascular access, infections). The dialysis module must generate these CMS forms from treatment data. CP TM: 'CMS/ESRD forms.'
Rule 6: The ICU flowsheet (MP-12) is the MOST DATA-DENSE screen in the product. It must handle: (a) real-time data from bedside monitors via CP Gateway (potentially every 15 seconds for waveform-derived numerics), (b) manual nursing entries at defined intervals, (c) lab results pulled at filing time, (d) medication infusion rates from pumps. Performance is critical — the flowsheet must render smoothly even with 50+ rows and 24+ hours of data columns.
Rule 7: Cardiac catheterization data has DUAL STORAGE: some data in Medicine files (MC namespace — hemodynamics, anatomy), some in Surgery files (SR namespace — because cardiac surgery and interventional cardiology overlap). The Surgery UM documents cardiac catheterization and angiographic data entry. The modern UI must unify data from both sources into a single cardiac cath report view.
Rule 8: Pulmonary function tests require GRAPHICAL display of flow-volume loops and volume-time curves alongside tabular numeric data. The PFT viewer (MP-07) must render these graphs accurately — pulmonologists use the SHAPE of the flow-volume loop to identify patterns (obstructive scalloping, restrictive truncation, upper airway obstruction flattening). Simple numeric tables are insufficient.
Rule 9: VistA Imaging is the STORAGE BACKEND for all procedure images and reports. ECG waveforms, echo cine loops, endoscopy photos/videos, cath angiographic images, and PFT graphical data are all stored on the VistA Imaging RAID. The modern UI must integrate with VistA Imaging for retrieval and display. DICOMweb (WADO-RS) is the modern standard for accessing these images.
Rule 10: INTERNATIONAL ADAPTATION: Medicine procedures are performed identically worldwide — the same ECG machine, the same endoscope, the same dialysis machine. What varies: (a) CMS/ESRD regulatory forms are US-specific (other countries have different dialysis quality reporting), (b) CPT codes are US billing codes (other countries use different procedure coding), (c) some quality metrics have country-specific targets. The procedure documentation itself is universal.
Rule 11: CP Gateway HL7 connections are SITE-SPECIFIC and require local configuration per instrument. Each ECG machine, PFT device, endoscopy system, and bedside monitor requires its own HL7 connection configuration in CP Gateway. The Device/Instrument Integration Catalog (Phase 2, Artifact 8) documents the device types and protocols. The modern product's configuration tooling should make CP Gateway setup as streamlined as possible.
Rule 12: Tenant Profile Matrix visibility: Medicine Procedures = PRIMARY for Profile A (Full Hospital — has cardiology, pulmonary, GI, nephrology, ICU). OPTIONAL for Profile B (Multi-Specialty Clinic — may have basic ECG and PFT). HIDDEN for Profiles C-G (not applicable to standalone pharmacy/lab/imaging). Dialysis-specific pages (MP-10, MP-11) should be separately toggleable — a facility may do cardiology but not dialysis, or vice versa.
