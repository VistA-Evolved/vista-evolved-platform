# 1. Foundation: BCMA — The Safety-Critical Last Line of Defense
Bar Code Medication Administration (BCMA) addresses the single most dangerous routine clinical activity: giving medications to inpatients. Medication errors (wrong patient, wrong drug, wrong dose, wrong route, wrong time) are a leading cause of preventable harm in hospitals. BCMA uses barcode scanning to electronically verify the Five Rights of medication administration at the bedside before every dose.
BCMA is fundamentally different from every other workspace covered so far in TWO ways:
(1) It is ALREADY a GUI application. BCMA V3.0 is a Delphi/Windows GUI that connects to VistA via RPC Broker, just like CPRS. It is NOT a terminal-only system. This means EXISTING RPCs (PSB namespace) already support the core workflow. The modern web client replaces the existing Windows GUI, not a terminal interface.
(2) It is SAFETY-CRITICAL. A bug in scheduling or billing causes inconvenience. A bug in BCMA can cause patient harm or death. The verification logic, scanning workflow, override handling, and variance logging must be implemented with extreme precision.
Source: BCMA TM: 'BCMA V3.0 stores detailed information about each inpatient at your VAMC, including medications administered to them and the PRN Effectiveness.' GUI UM: 'BCMA V3.0 software includes routines, files, enhancements, maintenance fixes.' Manager UM: 'Five Rights Override Area: This field lets you control the Unable to Scan verification process by allowing the user to verify the Five Rights of medication administration.'

Every medication administration in BCMA verifies five rights through barcode scanning:

What Happens If Wrong

Scan medication barcode. System matches scanned Drug IEN or NDC to the drug on the order.

# 5. Barcode Hardware Integration Architecture
BCMA depends on barcode scanning hardware. The existing BCMA GUI uses keyboard-wedge barcode scanners (the scanner emulates keyboard input, injecting barcode data as keystrokes). The modern web client has several options for barcode scanner integration:
Approach
How It Works
Pros
Cons
Recommendation
Keyboard-Wedge (same as current)
Scanner sends barcodes as keyboard input. Web browser receives as text input in a focused field.
Zero driver install. Works with any scanner. Same hardware as current BCMA deployment.
Requires focused input field. Multiple input fields on page can cause mis-scans. No feedback to scanner (beep/LED).
RECOMMENDED for initial deployment. Works with existing scanner fleet.
WebHID API
Browser directly communicates with HID-class barcode scanners via WebHID JavaScript API.
Direct scanner control. Can distinguish scan input from keyboard. Feedback control.
Limited browser support (Chrome-based only). Requires user permission grant. Not all scanners are HID-class.
CONSIDER for future enhancement.
WebUSB API
Browser communicates with USB scanner via WebUSB JavaScript API.
Full scanner control. Device-specific features available.
Very limited browser support. Security restrictions. Complex implementation.
NOT RECOMMENDED — too restrictive.
Native App Bridge
Thin native app on workstation handles scanner communication, passes data to web app via localhost websocket.
Full scanner control. Works with any browser. Supports all scanner types.
Requires native app installation on each workstation. Additional deployment burden.
CONSIDER only if keyboard-wedge is insufficient.
Camera-Based Scanning (mobile)
Use device camera to scan barcodes via JavaScript barcode scanning library (e.g. ZXing, QuaggaJS).
No external hardware for tablets/phones. Works on any device with camera.
Slower than dedicated scanner. Lighting dependent. Not suitable for high-volume bedside use.
RECOMMENDED for tablet/mobile as secondary input method. NOT for primary bedside scanning.
PATIENT SAFETY: The barcode scanning mechanism must be reliable and fast. A slow or unreliable scanner leads to nurses bypassing the scan (using Unable to Scan override), which defeats the purpose of BCMA. Keyboard-wedge scanners are the proven approach used in thousands of VA facilities. Start with keyboard-wedge for the web client.

# 6. Rules for Design and Engineering
Rule 1: BCMA IS SAFETY-CRITICAL. A medication error can harm or kill a patient. Every aspect of the Five Rights verification must be implemented with absolute precision. There is no acceptable error rate for Right Patient / Right Drug verification.
Rule 2: The existing BCMA Delphi GUI has its own RPC surface (PSB* namespace). REUSE these RPCs. Do NOT create parallel wrappers that duplicate existing functionality. Study the Delphi source code (FOIA-available) to understand exact calling patterns.
Rule 3: Patient wristband scanning MUST occur before any medication can be administered. The system must verify that the scanned patient matches the selected patient record. WRONG PATIENT = ABSOLUTE BLOCK, no override.
Rule 4: Medication barcode scanning verifies Right Drug. If the scan fails (damaged barcode, no barcode), the Unable to Scan workflow is governed by the PSB UNABLE TO SCAN key and the Five Rights Override site parameter. Not all nurses should have override ability.
Rule 5: Administration time windows are configurable per division (0-240 minutes before/after scheduled time). Administrations outside the window are VARIANCES, not errors — they are documented and reported but not blocked. The nurse must be able to administer late medications with variance logging.
Rule 6: PRN effectiveness documentation is required within a configurable window (up to 960 minutes). If not documented, a variance is logged. The VDL must prompt for PRN effectiveness proactively.
Rule 7: Controlled substance administration may require a witness (second nurse cosignature). The PSB NO WITNESS key exempts specific users. For C-II medications, dual verification is a regulatory requirement at most facilities.
Rule 8: BCMA must have a CONTINGENCY PLAN for VistA downtime. The existing BCMA Backup System (BCBU) uses a local InterSystems IRISHealth database to maintain medication administration capability during outages. The modern web client must have an equivalent offline/contingency capability.
Rule 9: The VDL must support ward-based and patient-based views. A nurse on a ward may be responsible for 4-8 patients. The nurse needs to see which patients have medications due NOW across their assignment, then drill into individual patients for administration.
Rule 10: Barcode label printing is a pharmacy function, not a nursing function. Pharmacy prints unit dose labels (via Zebra ZPL printers) and IV labels. These labels contain barcodes that BCMA scans at the bedside. The modern web client needs web-to-Zebra print capability for pharmacy label generation.
Rule 11: BCMA displays vitals alongside medications because vital signs affect administration decisions (e.g. hold BP medication if systolic < 100, hold heart rate medication if HR < 60). Vitals data comes from GMRV VITAL MEASUREMENT via existing RPCs, not from BCMA files.
Rule 12: The BCMA MEDICATION LOG is the legal record of medication administration. Every entry includes timestamp, nurse DUZ, patient DFN, drug, dose, route, scan status, and variance status. This log is subject to regulatory audit and must never be altered — only addended.
Rule 13: Limited Access BCMA mode allows chart review and non-administration documentation without bedside scanning. This is for situations where the nurse is not at the bedside (e.g. reviewing patient records at nurse station, documenting retrospective information).
Rule 14: BCMA interacts with THREE other VistA packages: PIMS (patient demographics), Inpatient Medications/PSJ (medication orders from Volume 5B), and OE/RR/CPRS (order entry from Volume 4). These interactions are via existing VistA integration, not new custom code. The modern web client must not break these integrations.
