# 1. Foundation: The Lab RPC Gap
Laboratory has the second-largest RPC gap after Supply/IFCAP. The chart-side lab result VIEWING works perfectly via existing CPRS RPCs (ORWLRR namespace) documented in Volume 4. However, every lab fulfillment workflow — accessioning, result entry, verification, QC, phlebotomy collection list management — is entirely terminal-driven through the LR namespace menus.
This means: clinician result viewing = ZERO custom wrappers needed (use CPRS RPCs from Volume 4). Lab department fulfillment = ALL custom wrappers needed.
The Lab V5.2 package has a highly structured data model centered on LAB DATA #63, which stores all patient results sub-filed by data type subscript: CH (chemistry, hematology, toxicology, RIA, serology), MI (microbiology), BB (blood bank), SP (surgical pathology), CY (cytology), EM (electron microscopy). Each subscript has different data structures and workflows.
Source: Lab TM (March 2024): 'Six files are substantially modified in V. 5.2 release: LABORATORY TEST #60, LAB DATA #63, WKLD CODE #64, LAB MONTHLY WORKLOADS #67.9, ACCESSION #68, and LAB JOURNAL #95.' Lab PIG (July 2024): LABORATORY TEST #60 field (4) LOCATION (DATA NAME) 'points to the unique subfield in the LAB DATA file (#63) which defines the type of input for results.'

### Auto-Instrument Results
Many chemistry results arrive automatically from instruments via HL7 interface (AUTO INSTRUMENT #62.4, LA7POC logical links). When auto-instrument results are downloaded:
(a) Results are auto-filed into LAB DATA #63 by the LA7 routines
(b) Results appear pre-populated in the result entry form with instrument ID noted
(c) Lab tech reviews auto-filed results rather than manually entering them
(d) If auto-verification rules are configured, some results may auto-verify without manual review
(e) Critical values and delta check failures always require manual review even if auto-verification is configured
POC RESULTS: Point-of-care (POC) results from bedside devices follow a modified workflow per Lab POC Interface Guide (April 2024). POC results are filed via LA7POC HL7 interface and 'are not verified by the traditional laboratory methods.' The modern UI should display POC results with a distinct indicator showing they were not traditionally verified.
### Result Entry Backend Action
ZVE LAB RESULT ENTRY custom wrapper must:
(a) Write results to LAB DATA #63 under the correct subscript and field as defined by LABORATORY TEST #60 LOCATION (DATA NAME)
(b) Calculate abnormal flags by comparing result to reference ranges in #60 (account for age/sex-specific ranges)
(c) Calculate delta checks by comparing to patient's previous result in #63
(d) Generate critical value alerts if result exceeds critical thresholds
(e) Link result to accession entry in #68
(f) Update workload counters in #67.9 if configured
(g) Return success with any flags/warnings, or structured error
CRITICAL: Results must be written to the exact LAB DATA #63 field location specified by LABORATORY TEST #60. The #60 LOCATION (DATA NAME) field 'points to the unique subfield in the LAB DATA file (#63) which defines the type of input for results.' Writing to the wrong field will corrupt the patient's lab record.

### Auto-Verification Rules
VistA Lab supports auto-verification for results that meet configured criteria:
(a) Result is within normal reference range
(b) Delta check does not exceed threshold
(c) QC for this test/instrument is in control
(d) No critical values
When all criteria are met, the system auto-verifies without manual review. The modern UI should show auto-verified results with a distinct indicator (e.g. 'Auto-Verified' chip instead of 'Verified') so supervisors can identify them in review.
CRITICAL VALUES NEVER AUTO-VERIFY. Any result flagged as critical (panic value) MUST have manual verification and clinician notification, regardless of auto-verification configuration.

# 6. Rules for Design and Engineering
Rule 1: Critical lab values (panic values, HH/LL) must NEVER auto-verify and must always trigger clinician notification.
Rule 2: The CH subscript covers the vast majority of routine lab testing. Build CH workflows first. MI, BB, SP, CY, EM have different data structures and workflows — defer to later phases.
Rule 3: Accession number generation must be atomic. No two simultaneous accessions can receive the same number within an accession area/date.
Rule 4: Auto-instrument results should be visually distinct from manually entered results.
Rule 5: POC (point-of-care) results follow a modified verification pathway per Lab POC Interface Guide. They should be displayed with a POC indicator.
Rule 6: STAT orders must be visually prioritized throughout the collection, accessioning, result entry, and verification workflows.
Rule 7: Delta checks are configured per test in LABORATORY TEST #60. The modern UI must calculate and display delta check results comparing current value to the patient's previous result.
Rule 8: Reference ranges may vary by patient age and sex. The verification and result display must use the correct age/sex-specific range.
Rule 9: Lab result corrections after verification must be audited in LAB JOURNAL #95. The original value and the corrected value must both be preserved.
Rule 10: The LABORATORY TEST #60 file is extremely sensitive. Editing test definitions affects how results are interpreted, reported, and flagged across the entire system. Configuration changes (LAB-10) require extreme caution and should be restricted to the Lab Information Manager role.
Rule 11: Results written to LAB DATA #63 must go to the exact field location specified by LABORATORY TEST #60 LOCATION (DATA NAME). This is non-negotiable.
Rule 12: Clinician result viewing (LAB-01/CLN-12 from Volume 4) is completely separate from lab fulfillment. Clinicians use ORWLRR RPCs under OR CPRS GUI CHART context. Lab staff use custom wrappers under a separate context option.
