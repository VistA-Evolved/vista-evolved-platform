# 1. Foundation: Two Packages, One Workspace
This workspace spans TWO distinct VistA packages that work together:
(A) Radiology/Nuclear Medicine V5.0 (RA namespace): Manages the radiology workflow — orders, exam registration, status tracking, report entry/verification, procedure definitions, diagnostic codes, workload reporting. Entirely terminal-driven for fulfillment.
(B) VistA Imaging (MAG namespace): Manages the actual images — DICOM gateway, image storage/retrieval, clinical display, VistARad (radiology-specific image viewer), Background Processor, image annotation. Has its own RPC set for image retrieval and display.
The exam lifecycle in VistA Radiology proceeds through defined EXAMINATION STATUS #72 stages. The RA V5.0 HL7 Manual documents this lifecycle: a patient is registered for an exam, the exam is performed, images are acquired (via DICOM/VistA Imaging), a radiologist interprets the images and enters a report (often via voice recognition: TalkStation or PowerScribe interfaces), the report is verified (signed), and the verified report is released to the ordering clinician via CPRS.
Source: RA TM (May 2025): EXAMINATION STATUS #72 defines exam stages. RA HL7 Manual: 'A patient is registered within the VistA Rad/Nuc Med system for an exam. The imaging exam is performed and the images for the case are given to a radiologist. The physician uses the vendor equipment to enter findings in a report and performs whatever action is necessary to trigger the vendor software to create and send to VistA an HL7 ORU message containing the report.'

### Voice Recognition Integration Note
Many radiology departments use voice recognition systems (TalkStation, PowerScribe) for report dictation. These systems send HL7 ORU messages to VistA with report text. The modern UI should support:
(a) Manual report entry (typing directly into the form above)
(b) Reception of voice-recognition-generated text via HL7 (auto-populated into report fields, radiologist reviews and verifies)
(c) Both workflows converge at verification — regardless of how report text was entered, the interpreting physician must verify/sign.

# 6. Rules for Design and Engineering
Rule 1: Users must have an assigned Imaging Location in NEW PERSON #200 field #74 to access any radiology functionality. Enforce this at login/context creation.
Rule 2: Report verification (signing) requires the RA VERIFY key. Only interpreting physicians (radiologists, nuclear medicine physicians) should hold this key.
Rule 3: Exam cancellation requires the RA CANCEL key and generates HL7 cancel messages. The modern UI must enforce the key check and trigger the HL7 message.
Rule 4: The Impression field may be mandatory depending on the Division parameter in RAD/NUC MED DIVISION #79 field .116. Check this parameter before allowing report submission without impression.
Rule 5: STAT imaging studies must be visually prioritized throughout the worklist, registration, status tracking, and report verification workflows.
Rule 6: Image availability from VistA Imaging (#2005) should be indicated on the worklist. Clinicians and radiologists should be able to launch the image viewer directly from the worklist or report authoring screen.
Rule 7: The printset mechanism (combined report for related procedures from a parent order) is a significant radiology workflow. The modern UI must support entering one report that covers multiple descendent exam cases.
Rule 8: Voice recognition report entry via HL7 ORU is an existing interface. The modern client should NOT replace this interface but should display voice-recognition-generated reports alongside manually entered reports, both converging at verification.
Rule 9: Radiology reports move through draft -> preliminary (released/unverified) -> final (verified) stages. Each stage must be clearly indicated with distinct visual treatment.
Rule 10: Radiation dose tracking (ABSORBED DOSE #70.3) is a regulatory requirement. The modern UI should display dose information when available but dose data entry is a specialized workflow deferred to later phases.
Rule 11: Clinician report viewing (IMG-01/CLN-13 from Volume 4) uses CPRS RPCs under OR CPRS GUI CHART context. Radiology department fulfillment uses custom wrappers under a separate context option. These are distinct user experiences for distinct roles.
Rule 12: VistA Imaging infrastructure (DICOM gateway, Background Processor, image storage/network paths, VIX for remote image access) is a separate deployment concern. The modern UI integrates with VistA Imaging via MAG RPCs but does not replace the imaging infrastructure.
